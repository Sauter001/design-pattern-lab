# 패턴 단위 회고 템플릿

패턴 한 단위(레벨 하나)를 끝낼 때마다 이 틀로 회고를 남긴다. 파일 위치는 자유(예: `docs/yours/level-01.md`). 복사해서 채운다.

이 회고는 정답 문서(`docs/levels/answers/`)를 열기 **전에** 쓴다. 그래서 일부러 패턴 이름 없이 쓰게 되어 있다. 이름은 본인이 지어 붙이고, 정식 명칭과의 대조는 맨 아래 칸에서 게이트를 연 뒤에만 채운다.

---

# [내가 붙인 이름] - [등장 도메인]

지금 만든 구조에 이름을 직접 지어 붙인다. "비교 방식 갈아끼우기"처럼, 하는 일이 드러나는 이름이면 충분하다.

## 문제 상황
어떤 도메인에서 무슨 변경 압력이 반복되는가.

## before의 고통
패턴 없는 버전에서 [변경 요구사항]이 들어오면 구체적으로 어디를 몇 군데 고쳐야 하는가.

## 적용한 구조와 의도
무엇을 어디로 옮겼는가. 그리고 이 구조가 노린 의도를 한 문장으로. (같은 모양도 의도가 다르면 다른 선택이다. 모양 말고 의도를 적는다.)

## after의 개선
같은 변경이 이제 어디만 닿는가.

## 고려한 다른 길
같은 압력을 다르게 풀 수도 있었는가. 카탈로그가 아니라 "이렇게도 짤 수 있었다"를 적는 칸이다: 더 단순한 분해, 다른 위치로 옮기기, 그냥 두기까지 포함해서. 그런데도 이 길을 고른 이유를 적는다.

## 트레이드오프 (비용)
이 구조가 추가로 떠안긴 비용: 클래스 증가, 간접성, 디버깅 난이도.

## [해당 시] 의도 삽입 고지
도메인이 요구해서가 아니라 학습 목적상 심은 구조면 명시.

## 관련 커밋
before -> 고통 -> after 커밋 해시를 적어 서사를 연결한다.

## [게이트를 연 뒤에만] 정답 대조
`answers/` 정답 문서를 연 다음에 채운다. 정식 이름, 그리고 카탈로그의 정석 모양과 내 버전이 갈라지는 지점 한 줄. 맨 위에 본인이 지은 이름이 정식 이름과 얼마나 가까웠는지도 남겨 두면 재미있다.
