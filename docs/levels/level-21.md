# 레벨 21: 잎 하나든 가지 통째든, 한 손에 쥐기

또 봐요. 스물한 번째예요. 드디어 왔네요, 약속한 그 구간. 레벨 0에서 "맨 끝에 둔 마지막 보상"이라고 흘렸던 거 기억나요? 미니 DB의 파싱 층, 하나의 구조 위에 여러 조각이 역할을 나눠 올라타는 그림요. 그게 오늘부터예요. 동네는 새 곳, `mini-db/frontend` 모듈(`lab.db.frontend`)이고요.

미리 한마디. 이 구간은 앞 단계들과 달라요. 오늘 만드는 이 트리 하나를, 다음 두 단계에서 그대로 다시 써요. 같은 구조 위에 단계마다 다른 걸 올리거든요. 그러니 오늘 건 잘 간수해 두세요. 버리고 새로 짜는 게 아니에요.

## 거슬러 올라가 돌아보기

레벨 11에서 우리는 절을 하나씩 얹어 `Query`를 지었어요. 레벨 19에선 옵티마이저가 그 계획을 규칙으로 다듬었고요. 그때 내내 다룬 그 WHERE 절, `age > 18 AND (vip = true OR points > 100)` 같은 거요. 줄로 쓰면 한 줄이지만, 메모리 안에선 사실 트리예요. 괄호가 중첩을 만들고, AND와 OR이 가지를 치죠. 오늘은 그 트리를 직접 짓고 다룹니다.

## 이번에 마주칠 상황

WHERE 절을 들여다보면 두 종류가 섞여 있어요.

- 잎: 더는 못 쪼개는 단일 비교. `age > 18`, `vip = true`.
- 가지: 잎이나 다른 가지를 AND/OR로 묶은 것. 그리고 가지 안에 또 가지가 들어가요.

이걸 트리로 들고, 트리를 훑어서 뭔가 해야 해요. 이를테면 이 조건을 사람이 읽게 다시 문자열로 그리거나(EXPLAIN에 찍게요), 이 조건이 건드리는 컬럼을 모으거나요.

이번 목표는 이거예요. **트리를 훑는 코드가 "이게 잎이냐 가지냐"를 매번 따지지 않게 한다. 잎이든 가지 통째든 똑같이 "조건 하나"로 다룬다.**

## 가장 정직하게 먼저

잎과 가지를 각각 다른 타입으로 두고, 트리를 훑을 때 타입을 따져 분기해 볼게요. 제일 정직하게요.

```java
// 잎: 단일 비교
class Comparison {
    String column;
    String op;
    Object value;
}

// 가지: 논리로 묶은 것
class BoolGroup {
    String op;  /* AND | OR */
    List<Object> children;
}

// 조건을 문자열로 다시 그린다
String render(Object node) {
    if (node instanceof Comparison c) {
        return c.column + " " + c.op + " " + c.value;
    } else if (node instanceof BoolGroup g) {
        List<String> parts = new ArrayList<>();
        for (Object child : g.children) parts.add(render(child));   // 가지면 자식들로 재귀
        return "(" + String.join(" " + g.op + " ", parts) + ")";
    }
    throw new IllegalStateException("이건 또 무슨 노드람");
}
```

동작은 해요. 트리를 그려 내죠. **여기까지 첫 커밋.** before예요.

## 트리로 할 일이 늘어요

파서팀이랑 플래너팀이 하나씩 들고 와요.

- "이 조건이 건드리는 컬럼 목록을 뽑아 주세요. 인덱스 고를 때 쓰게요." 그래서 `collectColumns`를 만드는데, render랑 똑같은 instanceof 분기를 또 답니다.
- "조건 트리가 너무 깊으면 거부하게, 깊이를 재 주세요." 그래서 `depth`를 만드는데, 또 같은 분기예요.
- "NOT이랑 BETWEEN 노드도 곧 들어와요." 그럼 이제 render, collectColumns, depth 그 모든 메서드의 분기 사슬에 가지를 하나씩 더 칩니다.

시키는 대로 해 보면 이렇게 돼요.

```java
String render(Object n)               { if (n instanceof Comparison) ... else if (n instanceof BoolGroup) ... else if (n instanceof NotNode) ... }
List<String> collectColumns(Object n) { if (n instanceof Comparison) ... else if (n instanceof BoolGroup) ... else if (n instanceof NotNode) ... }
int depth(Object n)                   { if (n instanceof Comparison) ... else if (n instanceof BoolGroup) ... else if (n instanceof NotNode) ... }
```

**여기까지 두 번째 커밋.** 일단 박아 두세요.

## 혹시 문제를 느꼈나요?

이쯤 왔으면 굳이 길게 안 해도 보일 거예요. 그래도 짚어요. 습관이라.

- 트리를 훑는 메서드마다 잎이냐 가지냐 분기가 통째로 반복돼요. 게다가 "가지면 자식들로 내려간다"는 똑같은 재귀가 메서드마다 복붙돼 있고요.
- 노드 타입이 하나 늘면(NOT, BETWEEN) 그 분기 사슬을 전부 찾아다니며 가지를 쳐야 해요. 산탄총이죠. 한 군데라도 빠뜨리면 그 메서드만 새 노드를 조용히 모른 척하고요.
- 쓰는 쪽이 잎과 가지를 다른 타입으로 알아야 해서, 트리를 다룰 때마다 "지금 이게 단일이냐 묶음이냐"를 의식해야 해요.

새 구간이라 헷갈릴 짝은 아직 없어요. 대신 선만 하나 그어 둘게요. 오늘은 "트리를 어떻게 들고, 어떻게 잎/가지 안 가르고 훑느냐"는 모양만 잡아요. 그 트리에 진짜 뜻을 입히는 일, 그러니까 조건을 실제 행에 대고 참인지 거짓인지 따지는 건 다음 단계 몫이고요. **오늘은 모양, 다음은 뜻.** 이 선을 기억해 두세요. 다음 단계가 거의 똑같아 보일 거라.

## 방향만 던질게요

길이 떠오를 거예요. 방향만 대 줄게요.

1. 잎과 가지를 다른 타입으로 가르지 마세요. 둘 다 같은 약속을 따르게 해 봐요. 이를테면 `String render()` 하나를 가진 공통 인터페이스 `Condition`을요. 그럼 쓰는 쪽은 손에 든 게 잎인지 가지인지 몰라도 `render()`만 부르면 돼요.
2. 가지는 자식들도 같은 `Condition`으로 들고 있다가, 자기 `render()` 안에서 자식들의 `render()`를 부르면 돼요. 내려가기가 가지 자신 안에 딱 한 번만 있고, 메서드마다 복붙되던 재귀가 사라지죠.
3. 그러면 새 노드 타입(NOT, BETWEEN)은 그 약속을 따르는 클래스 하나 더 만들면 끝이에요. 트리를 훑던 기존 코드는 한 줄도 안 바뀌고요. 그 노드는 자기 `render()`만 알면 되니까요.
4. 핵심은 이거예요. 가지 하나가 통째로도 "조건 하나"처럼 다뤄져요. `AND(...)` 묶음을 더 큰 `OR(...)`의 자식으로 그냥 끼울 수 있죠. 잎이 들어갈 자리에 가지를 통째로 넣어도 똑같이 굴러가요. 그게 이 구조의 전부예요.

도착점은 또렷해요. **잎이든 가지 통째든 똑같이 "조건 하나"로 다뤄진다. 트리를 훑는 코드는 잎과 가지를 안 가른다. 새 노드 타입이 와도 기존 순회 코드는 한 줄도 안 바뀐다.** 거기까지 가면 after예요.

검증은 짧아요. 커밋 전에 자문해요. 새 노드 타입(예: NOT)을 추가하는데, 트리를 훑던 기존 메서드(render 같은)를 여나요, 안 여나요? 그리고 AND 묶음을 OR의 자식으로 그냥 끼울 수 있나요? 기존 메서드 안 열고 끼워지면 after예요.

## 회고: 다음 두 단계가 이 트리에 올라탑니다

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도요. 공통 약속을 두니 잎도 가지도 같은 인터페이스를 따라야 해요. 그래서 가지에만 의미 있는 것(자식 추가, 제거)을 잎도 형식상 떠안거나, 반대로 인터페이스를 좁게 잡아 가지 전용 기능을 따로 빼야 하거나요. "다 똑같이 보이게(투명성)"와 "잎엔 자식 연산이 아예 없게(안전성)" 사이의 거래예요. 본인이 어느 쪽을 골랐는지 적어 두세요. 그리고 트리를 한 겹 더 추상화한 만큼, 디버깅할 때 "지금 이 노드가 구체적으로 뭐냐"가 한 꺼풀 가려지고요.

오늘 만든 이 트리, 잘 간수하랬죠. 다음 단계에선 여기에 "뜻"을 입혀서, 진짜 행을 대고 참인지 거짓인지 따지게 할 거예요. 그다음 단계에선 또 다른 방식으로 같은 트리에 일을 시키고요. 한 구조 위에 조각들이 역할을 나눠 올라타는 그림, 레벨 0에서 약속한 그게 지금부터 세 단계에 걸쳐 펼쳐져요.
