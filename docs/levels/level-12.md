# 레벨 12: 곱으로 늘어나는 걸, 합으로 바꾼다

또 봐요. 열두 번째예요. 미니 DB 안에서 한 층 더 내려가요. 이번엔 `mini-db/storage` 모듈(`lab.db.storage`)이에요. 레벨 8에서 트랜잭션 다뤘던 그 저장 층요.

지난 단계(레벨 11)에선 조각 많은 객체 하나를 깔끔하게 짓는 걸 했죠. 오늘은 만드는 얘기가 아니에요. 이미 있는 것들이 **두 갈래로 따로따로 자랄 때 클래스가 폭발하는** 걸 막는 얘기예요. 결이 다르니 머리 한 번 바꿔 끼우고 가요.

## 이번에 마주칠 상황

미니 DB가 데이터를 어딘가에 저장해야 해요. 그런데 여기서 두 가지가 서로 따로 변합니다.

- **무엇으로 저장하나(백엔드).** 파일에 쓸 수도 있고, 메모리에만 둘 수도 있어요. 나중엔 원격 스토리지도 붙고요.
- **어떤 종류의 store인가.** 그냥 읽고 쓰는 기본 store도 있고, 값이 바뀔 때마다 이전 값을 남기는 버전 store도 있고, 자주 읽는 걸 캐시해 두는 store도 있죠.

이 둘은 서로 상관없이 늘어요. 백엔드가 하나 붙기도 하고, store 종류가 하나 생기기도 하고요. 이번 목표는 이거예요. **저장 백엔드와 store 종류, 이 두 갈래가 서로를 안 건드리고 따로따로 늘어나게 한다.**

## 가장 정직하게 먼저

store를 인터페이스로 두고, 백엔드별로 구현해요. 제일 정직하게요.

```java
interface Store {
    byte[] read(Key k);
    void write(Key k, byte[] v);
}

class FileStore implements Store {       // 파일에 읽고 쓴다
    public byte[] read(Key k) { /* 파일에서 */ }
    public void write(Key k, byte[] v) { /* 파일에 */ }
}

class InMemoryStore implements Store {   // 메모리에 읽고 쓴다
    public byte[] read(Key k) { /* 맵에서 */ }
    public void write(Key k, byte[] v) { /* 맵에 */ }
}
```

여기까진 깔끔해요. 백엔드 둘. **여기까지 첫 커밋.** before예요.

## store 종류가 생기기 시작해요

서비스가 자라면서 store 종류 요구가 들어와요.

- 복구팀: "값이 바뀔 때마다 이전 값을 남기는 버전 store가 필요해요. 롤백에 쓰게요."
- 성능팀: "자주 읽는 페이지는 캐시하는 store도요."
- 인프라팀: "원격 백엔드(네트워크 스토리지)도 곧 붙어요."

시키는 대로 만들어 보세요. 버전 store가 파일에도 메모리에도 돼야 하니 `VersionedFileStore`, `VersionedInMemoryStore` 둘. 캐시 store도 `CachedFileStore`, `CachedInMemoryStore` 둘. 그리고 원격 백엔드가 오면? 기본, 버전, 캐시 전부에 원격 짝을 또 만들어야 해요.

```java
class VersionedFileStore implements Store { /* 파일 쓰기 + 버전 기록 */ }
class VersionedInMemoryStore implements Store { /* 메모리 쓰기 + 버전 기록 */ }
class CachedFileStore implements Store { /* 파일 + 캐시 */ }
class CachedInMemoryStore implements Store { /* 메모리 + 캐시 */ }
// 원격 백엔드가 오면 여기에 또 세 개. store 종류가 늘면 백엔드 수만큼 또.
```

**여기까지 두 번째 커밋.** 일단 박아 두세요.

## 혹시 문제를 느꼈나요?

이쯤 되면 보이죠. 그래도 짚어 둘게요.

- store 종류(기본/버전/캐시)와 백엔드(파일/메모리/원격)를 **곱한 만큼** 클래스가 생겨요. 셋 곱하기 둘이면 여섯, 셋 곱하기 셋이면 아홉. 두 갈래가 한 상속 트리에 엉켜서 곱으로 늘어요.
- 새 백엔드(원격) 하나 붙이는데 store 종류 수만큼 클래스를 만들어야 해요. 새 store 종류 하나 추가하는데 백엔드 수만큼 만들어야 하고요.
- "파일에 어떻게 쓰나"라는 코드가 `FileStore`, `VersionedFileStore`, `CachedFileStore`에 똑같이 복붙돼 있어요. 백엔드 로직이 store 종류마다 중복이죠.

감싸는 거(레벨 5) 닮아 보일 수 있어요. 근데 그건 한 축이었죠. 여기선 두 축이 동시에, 서로 무관하게 자라요. 그게 이번 색깔이에요.

## 방향만 던질게요

길이 떠오를 거예요. 방향만 대 줄게요.

1. 한 트리에 엉킨 두 갈래를 떼어 놓을 수 없을까요. "물리적으로 어떻게 저장하나"(파일/메모리/원격)를 한쪽 약속으로 빼고, "어떤 종류의 store인가"(기본/버전/캐시)를 다른 쪽에 두는 거예요. 그리고 store 종류 쪽이 백엔드를 하나 품게 하고요. store는 "저장해 줘"라고 백엔드한테 넘기기만 하고, 그게 파일인지 메모리인지는 모르게요.
2. 그렇게 떼어 두면, 새 백엔드(원격)는 백엔드 약속만 구현한 클래스 하나면 끝이에요. 그 하나가 기본/버전/캐시 store 전부랑 그냥 맞물려요. 새 store 종류도 마찬가지로 하나만 만들면 모든 백엔드 위에서 돌고요. 곱(종류 수 곱하기 백엔드 수)이 합(종류 수 더하기 백엔드 수)으로 바뀌는 거죠.
3. 레벨 4(어댑터)랑 헷갈리지 마세요. 거긴 이미 안 맞는 남의 모양을 사후에 맞췄어요. 여긴 달라요. 처음부터 두 축이 따로 자랄 걸 알고 미리 갈라 놓는 거예요. 안 맞는 걸 맞추는 게 아니라, 엉킨 걸 떼어 놓는 거죠.

도착점은 또렷해요. **새 백엔드를 추가해도 store 종류 코드는 안 바뀌고, 그 반대도 그렇다. 백엔드 하나, store 종류 하나를 더하면 클래스 하나씩 늘 뿐이다. 곱이 아니라 합으로 는다.** 거기까지 가면 after예요.

## 검증: 곱이냐 합이냐

다 됐다 싶으면, 커밋 전에 세어 봐요. 이게 제일 정직한 질문이에요.

> 새 백엔드(원격) 하나 붙이는데 클래스를 **몇 개** 만들어요? 그리고 새 store 종류(읽기 전용) 하나 추가하는데 **몇 개**예요?

- 둘 다 store 종류 수만큼, 백엔드 수만큼이면 -> 아직이에요. 곱에 머문 거예요.
- 둘 다 딱 하나씩이면 -> after예요. 두 축이 따로 자라는 거죠.

## 회고: 두 축이 진짜 둘 다 늘었나

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도요. 비용은 간접성이에요. store가 직접 저장하지 않고 백엔드를 한 번 거쳐요. 읽기 한 번에 다리 하나 더 건너고요.

그리고 이건 두 축이 **진짜로 둘 다** 따로 변할 때만 값을 해요. 백엔드가 영영 파일 하나뿐이면 이 분리는 과한 거예요. "내 경우엔 두 축이 정말 둘 다 늘었나"를 적어 두세요. 한 축만 늘 거였으면 그냥 두는 게 나았을 수도 있어요. 안 쓸 이유를 분간하는 것도 실력이에요.
