# 레벨 4: 모양이 제각각인 핸들러들을 한 창구로 부르기

벌써 네 번째 단계예요. 이번이 워밍업 구간(레벨 1~4)의 마지막입니다. 끝나면 우리는 "비슷해 보이지만 실은 다른" 패턴들을 구별하는 더 까다로운 구간으로 넘어가요. 그 준비운동으로 딱 맞는 주제거든요.
먼저 지난 단계부터 짚을게요.

## 지난 단계 돌아보기

레벨 3에서는 가입이라는 사건 하나가 일어났을 때, 그 사실을 알아야 할 여러 곳에 핸들러가 직접 일일이 알리던 걸 끊어 냈죠. 핸들러는 "일이 일어났다"고 알리기만 하고, 누가 반응할지는 핸들러가 모르게요.
일대다로 퍼지는 통지였어요.

이번엔 다시 디스패처로 시선을 돌립니다. 그런데 한 가지가 달라졌어요. 레벨 1에서 우리가 만든 디스패처는 핸들러를 찾아 `handle(path)`를 불렀죠. 그게 가능했던 건 핸들러를 전부 우리가 같은 모양(
`Handler` 인터페이스)으로 만들었기 때문이에요. 현실은 그렇게 깔끔하지 않습니다.

작업 공간은 그대로 `mini-spring/mvc` 모듈(`lab.spring.mvc`)이에요.

## 우리가 마주한 것

디스패처가 바라는 핸들러의 모양은 여전히 하나예요. "경로를 주면 응답 문자열을 돌려줘."

```java
interface Handler {
    String handle(String path);
}
```

그런데 같이 굴려야 할 핸들러들이 저마다 다른 모양으로 말을 걸어와요. 우리가 새로 만든 것만 있는 게 아니거든요.

- 예전에 누가 만들어 둔 레거시 컨트롤러는 메서드 이름도 인자도 달라요. `String process(Request req)`.
- 다른 팀은 어노테이션 방식으로 짜요. 평범한 객체에 매핑만 붙여 두고, 디스패처가 알아서 그 메서드를 찾아 불러 주길 원하죠.
- 외부 라이브러리에서 받아 온 핸들러는 `Response service(Map<String,String> params)` 모양이고, 그 코드는 우리가 손댈 수 없어요.

이번 목표는 이거예요. **모양이 서로 다른 핸들러들을, 디스패처가 알던 그 한 가지 방식으로 불러서 쓴다.**

## 가장 정직하게 먼저

디스패처가 핸들러 종류를 직접 알아보고, 종류마다 다르게 부르게 해 볼게요.

```java
class DispatcherServlet {

    String dispatch(Object handler, String path) {
        if (handler instanceof Handler h) {
            return h.handle(path);                       // 우리 모양: 그대로
        } else if (handler instanceof LegacyController c) {
            Request req = new Request(path);             // 인자를 만들어 주고
            return c.process(req);                       // 이름 다른 메서드를 부르고
        } else if (handler instanceof ThirdPartyHandler t) {
            Map<String, String> params = parse(path);    // Map으로 바꿔 주고
            Response res = t.service(params);
            return res.body();                           // 결과도 우리 모양으로 꺼내고
        }
        throw new IllegalStateException("못 부르는 핸들러: " + handler);
    }
}
```

동작은 합니다. 종류마다 인자를 맞춰 주고, 알맞은 메서드를 불러 주고, 결과를 우리가 원하는 모양으로 되돌리죠. **여기까지 첫 커밋.** before예요.

## 핸들러 모양이 또 늘어납니다

얼마 안 가 핸들러 종류가 더 들어와요. 뭘 들고 오냐면요.

- 동료 개발자: "예전 레거시 컨트롤러 수십 개를 그대로 살려서 같이 돌려야 해요. 그쪽은 못 고치니까 디스패처가 맞춰 줘야 해요."
- 다른 팀: "우리는 어노테이션으로 짤게요. 메서드에 매핑만 붙이면 디스패처가 찾아서 불러 주게요."
- 통합 담당: "결제는 외부 SDK 핸들러를 그대로 끼워 쓸 거예요. 인자가 Map이든 뭐든 그쪽 규격이라 우리가 못 바꿔요."

시키는 대로 `dispatch`에 넣다 보면 `else if`가 끝없이 이어집니다.

```java
// dispatch 안의 else if 사슬에 가지가 계속 붙는다
} else if (handler instanceof AnnotatedController c) {
    // 리플렉션으로 @RequestMapping 메서드를 찾아, 경로로 인자를 맞춰 호출
    return invokeAnnotated(c, path);
} else if (handler instanceof PaymentSdkHandler p) {
    // 결제 SDK 규격대로 또 다른 변환을 한 벌 더...
}
```

**여기까지 두 번째 커밋.** 일단 박아 두세요.

## 잠깐. 하나만 직접 채워 보죠

레벨 1에서 정규식 직접 넣어 봤죠? 여기서도 손으로 한 번 겪고 가요. 이번엔 아픈 숫자가 좀 달라요.

방금 그 코드 블록, 마지막 줄에서 제가 "결제 SDK 규격대로 또 다른 변환을 한 벌 더"라고 주석으로 얼버무렸죠. 그 주석, 본인이 진짜로 채우세요. **방법은 안 알려줄게요.** 외부 SDK는 이 모양이에요.

```java
// 우리가 못 고치는 외부 코드
class PaymentSdkHandler {
    Response service(Map<String, String> params) { ... }
}
```

경로를 `Map`으로 바꿔서 `service`에 넘기고, 돌아온 `Response`에서 본문을 꺼내 우리 모양(`String`)으로 돌려주면 돼요. 그 변환을 `dispatch`의 `else if` 안에 직접 써넣으세요. 지금 당장. (커밋은 신경 쓰지 마요. 곧 다 들어낼 거예요.)

...

채웠어요? 그럼 손 떼고 `dispatch`를 위에서 아래로 훑으면서 세 개만 세어 보세요. 머릿속 말고, 숫자로요.

1. `dispatch` 한 메서드가 지금 아는 **호출 규칙이 몇 가지**예요? 우리 모양, 레거시, 서드파티, 방금 SDK...
2. 그 SDK 변환 채우면서 `dispatch`가 새로 알게 된 남의 타입이 몇 개예요? `Map`, `Response`, `PaymentSdkHandler`...
3. 모양이 하나 더 오면 저 두 숫자는 어떻게 돼요?

뻔하죠. 모양이 하나 늘 때마다 `dispatch`가 아는 호출 규칙도, 끌어안는 남의 타입도 같이 하나씩 늘어요. 끝이 없고요. 정작 그 지식은 "요청을 핸들러에게 넘긴다"는 dispatch의 본업이랑 아무 상관이 없어요.

## 혹시 문제를 느꼈나요?

이쯤 되면 뭐가 문제인지 보이죠. 그래도 정리해 둘게요.

- 핸들러 모양이 하나 늘 때마다 디스패처의 핵심 메서드를 다시 열어 `else if`를 끼워야 해요. 방금 SDK 끼우면서 본인 손으로 했고요. 게다가 거기엔 그 핸들러만의 호출 규칙(인자를 어떻게 만들고, 어떤 이름의 메서드를 부르고, 결과를 어떻게 꺼내는지)이 통째로 들어가요.
- 디스패처가 세상 모든 핸들러의 속사정을 다 알게 됐어요. 레거시의 메서드 이름, 서드파티의 인자 타입까지요. 정작 디스패처의 본업인 "요청을 핸들러에게 넘긴다"와는 상관없는 지식이죠.
- 레거시 컨트롤러 하나를 따로 시험하고 싶어도, 디스패처를 통째로 세워 그 `else if` 가지를 타야만 부를 수 있어요.

여기서 잠깐. "이거 레벨 1이랑 똑같은 거 아니야?" 싶을 거예요. 거기서도 `else if`를 없앴으니까요. 그런데 결이 달라요. 레벨 1에서 갈리던 건 **우리가 직접 만든, 처음부터 갈아끼우라고 설계한 정책**이었고 우리는 그중 하나를 골랐어요. 지금 손에 든 건 **이미 만들어져 못 고치는, 모양이 제각각인 남의 물건들**이에요. 고르는 문제가 아니라 안 맞는 모양을 맞추는 문제예요. 정책을 교체하는 게 아니라, 안 맞는 콘센트에 어댑터를 끼우는 쪽에 가깝죠.

## 이번 함정을 미리 말해 둘게요

다들 여기서 한 번 같은 데로 새요. 미리 막을게요.

`dispatch`가 지저분하니까, 변환 덩어리를 메서드로 빼요. `invokeLegacy(c, path)`, `invokeSdk(p, path)` 이런 식으로요. 그럼 `dispatch`는 이렇게 깔끔해 보이죠.

```java
String dispatch(Object handler, String path) {
    if (handler instanceof Handler h) return h.handle(path);
    else if (handler instanceof LegacyController c) return invokeLegacy(c, path);
    else if (handler instanceof PaymentSdkHandler p) return invokeSdk(p, path);
    // ...
    throw new IllegalStateException("못 부르는 핸들러: " + handler);
}
```

"오, 줄었다" 싶죠. 근데 보세요. `dispatch`는 여전히 `instanceof`로 종류를 분간하고, 새 모양이 오면 여기 `else if`를 또 붙여야 하고, `invokeSdk` 안에선 여전히 `Map`이랑 `Response`를 알아요. **dispatch가 모든 모양을 안다는 사실은 그대로예요. 변환 코드를 아래로 접어 넣었을 뿐이죠.**

레벨 1에서 본 그 함정 기억나요? `else if`를 `matches` 메서드로 뺐는데 아무것도 안 옮긴 거. 똑같아요. **메서드로 빼는 건 "줄 정리"지 "분리"가 아니에요.** 새 모양이 와도 dispatch를 안 여는 게 목표지, dispatch를 짧게 만드는 게 목표가 아니거든요.

## 방향만 던질게요

길이 떠오를 거예요. 답은 본인이 부딪혀 찾는 거예요. 저는 방향만 대 주고요.

1. 디스패처가 바라는 모양은 끝까지 `handle(path)` 하나예요. 디스패처를 고치지 말고, 모양이 안 맞는 핸들러마다 그 사이에 **번역해 줄 중간 존재**를 하나씩 끼우면 어떨까요. 바깥(디스패처)에는 `handle`을 받는 척하고, 안에서는 품은 진짜 핸들러의 `process`나 `service`로 바꿔 부르는 존재요. 함정이랑 갈리는 지점은, 변환을 dispatch 안에 접어 넣느냐 dispatch 바깥의 독립된 존재로 들어내느냐예요. 그 둘은 달라요.
2. 그러면 디스패처는 그게 레거시인지 서드파티인지 영영 몰라도 돼요. 그냥 `handle`만 부르면 되고, 인자를 만들고 결과를 꺼내는 지저분한 변환은 각자의 중간 존재 안에 숨죠. 새 모양이 와도 그 모양 전용 번역기를 하나 더 만들어 끼우면 끝이고요.
3. 지난 단계들과 묶어 보세요. 레벨 1은 여럿 중 하나를 골랐고(정책 교체), 이번엔 못 고치는 것을 억지로 맞춰 끼워요(모양 변환). 다음 단계에서는 "감싼다"는 비슷한 동작이 또 나오는데, 그때 이번 것과 무엇이 같고 무엇이 다른지 보면 재미있을 거예요.

끝까지 가면 이렇게 돼요. **새 모양의 핸들러가 와도 디스패처의 `dispatch`는 단 한 줄도 바뀌지 않는다. 핸들러 종류 하나당 번역기 하나를 끼우면 그만이다.** 거기까지 가면 after예요.

## 검증: dispatch를 여나 안 여나

다 됐다 싶으면, 커밋 전에 이거 통과하세요. 레벨 1 검증이랑 질문이 똑 닮았어요. 그럴 만하죠, 쌍둥이니까.

> 또 다른 모양의 핸들러가 하나 왔어요(예: gRPC 핸들러, 또 다른 인자에 또 다른 반환). 그걸 붙이려고 **`DispatcherServlet`을 열어요, 안 열어요?**

- 연다 -> 아직이에요. dispatch가 아직 모든 모양을 분간하고 있는 거고, 함정에 머문 거예요. 위 함정 섹션 다시 봐요.
- 안 연다(그 모양 전용 번역기 하나 새로 만들어 등록만 하면 끝난다) -> 됐어요. 그게 after예요. dispatch는 그 핸들러가 생긴 줄도 몰라요.

## 회고: 레벨 1을 옆에 펴고

다 되면 세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요.

워밍업 마지막이니까, 이번엔 꼭 레벨 1을 옆에 펴 두고 한 줄 남기세요.

> 레벨 1도 `else if`를 없앴고 이번에도 없앴는데, 없앤 이유가 어떻게 달랐나? 한쪽은 갈아끼울 정책을 고르려고(정책 교체), 다른 쪽은 못 고치는 남의 모양을 맞추려고(모양 변환)였어요.

그 구분이 입에서 자연스럽게 나오면 워밍업은 끝난 거예요.

비용도 적고요. 아까 센 숫자, 여기서 써먹어요. before에선 `dispatch` 하나가 모든 호출 규칙을 떠안았지만(모양 수만큼), after에선 dispatch가 아는 건 영영 `handle` 하나예요. 대신 핸들러 종류마다 번역기 클래스가 따로 생겼고, 진짜 호출까지 한 단계 더 거쳐 가죠. 그 간접성이 얼마나 늘었는지까지 솔직하게 적으세요. 다음 단계 열면서 이어 가죠.
