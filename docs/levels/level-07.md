## 거슬러 올라가 돌아보기

레벨 1 기억나요? 첫 단계요. 디스패처가 경로를 보고 핸들러를 고를 때, "경로가 맞는지 따지는 방식"이 제각각이라 골치였죠. 정확매칭, 접두사매칭, 정규식매칭. 우리는 그 "비교하는 방식" 한 조각만 떼어내 공통 약속 뒤에 숨기고, 디스패처 바깥에서 골라 끼웠어요. 그래서 새 매칭 방식이 와도 디스패처 본체는 안 바뀌었고요.

그 모양을 오늘 또 씁니다. 거의 똑같이요. 단, 이번엔 워밍업이 아니라 쓸모가 분명한 데서요. 그리고 하나 미리 흘려둘게요. **다음 단계에서 이거랑 그림이 똑같은 걸 또 만드는데, 거기선 운전대를 쥔 손이 바뀝니다.** 무슨 말인지는 끝에서요.

## 이번에 마주칠 상황

미니 DB가 쿼리를 실행하다 보면 두 테이블을 **조인**해야 할 때가 와요. `users`랑 `orders`를 `user_id`로 붙이는 식으로요. 그런데 조인을 실제로 하는 방법이 하나가 아니에요.

- 한쪽 행마다 다른 쪽을 전부 훑는 방법(가장 단순, 작은 테이블엔 이걸로 충분)
- 한쪽을 해시 테이블로 만들어 두고 다른 쪽을 던져 맞추는 방법(큰 테이블에 빠름)
- 양쪽을 키로 정렬해 두고 지퍼처럼 한 번에 맞물리는 방법(이미 정렬돼 있으면 유리)

세 방법은 결과가 같아요. 같은 두 테이블을 조인하면 같은 행이 나와요. 그런데 **언제 어느 게 빠른지가 달라요.** 그래서 상황을 보고 골라야 하죠.

이번 목표는 이거예요. **조인 방법을 여러 개 두고, 실행기는 그대로 둔 채 바깥에서 골라 끼운다.**

## 가장 정직하게 먼저

일단 시키는 대로, 조인하는 실행기 하나에 방법들을 다 욱여넣어 볼게요. 제일 정직하게요.

```java
class JoinExecutor {

    enum Algo { NESTED_LOOP, HASH, SORT_MERGE }

    List<Row> join(Table left, Table right, String key, Algo algo) {
        List<Row> out = new ArrayList<>();
        if (algo == Algo.NESTED_LOOP) {
            for (Row l : left.rows()) {
                for (Row r : right.rows()) {          // 모든 짝을 다 본다
                    if (l.get(key).equals(r.get(key))) out.add(merge(l, r));
                }
            }
        } else if (algo == Algo.HASH) {
            Map<Object, List<Row>> bucket = new HashMap<>();
            for (Row r : right.rows()) {
                bucket.computeIfAbsent(r.get(key), k -> new ArrayList<>()).add(r);
            }
            for (Row l : left.rows()) {
                for (Row r : bucket.getOrDefault(l.get(key), List.of())) out.add(merge(l, r));
            }
        } else if (algo == Algo.SORT_MERGE) {
            // 양쪽을 key로 정렬한 뒤 한 번에 맞물려 훑는다 (생략)
        }
        return out;
    }
}
```

동작은 해요. `Algo`만 넘기면 그 방식으로 조인해요. **여기까지 첫 커밋.** before예요.

## 옵티마이저팀이 옵니다

미니 DB에도 "어느 방법이 빠를지 정하는" 옵티마이저가 붙기 시작해요. 받은 말을 그대로 옮기면 이래요.

- "테이블 크기를 보고 알아서 골라 주세요. 양쪽이 작으면 단순 반복, 한쪽이 크면 해시, 양쪽이 이미 정렬돼 있으면 정렬 병합으로요."
- "인덱스 조인이라는 새 방법을 곧 붙일 거예요. 인덱스가 있으면 그쪽이 제일 빠르거든요. 기존 거 안 건드리고 추가됐으면 좋겠어요."
- "방법 하나만 따로 떼서 벤치마크하고 싶어요. 지금은 실행기 전체를 세워야만 하나를 돌릴 수 있어서 못 해요."

시키는 대로 넣어 보세요. 고르는 기준이 늘 때마다 저 `if (algo == ...)` 앞에 또 다른 판단이 붙고, 새 방법이 늘 때마다 `else if`가 한 칸씩 자라요. 실행기 하나가 점점 모든 조인 방식의 속을 다 끌어안습니다.

```java
List<Row> join(Table left, Table right, String key, Stats stats) {
    Algo algo;
    if (stats.bothSorted(key)) algo = Algo.SORT_MERGE;       // 고르는 일이
    else if (stats.size(right) > 10000) algo = Algo.HASH;    // 실행하는 일과
    else algo = Algo.NESTED_LOOP;                            // 한 메서드에 엉킨다

    if (algo == Algo.NESTED_LOOP) {
        // ... 위의 이중 루프 그대로
    } else if (algo == Algo.HASH) {
        // ... 위의 해시 빌드 + 탐침 그대로
    } else if (algo == Algo.SORT_MERGE) {
        // ... 정렬 후 병합 그대로
    }
    // 인덱스 조인이 오면? 여기 else if 한 칸 더. 늘 그렇듯.
}
```

**여기까지 두 번째 커밋.** 고통이 드러난 상태예요. 잠깐 저 메서드를 노려보죠.

## 혹시 문제를 느꼈나요?

걸리는 게 있을 거예요. 없으면 짚어 줄게요.

- 조인 방법이 하나 늘 때마다 실행기의 핵심 메서드를 다시 열어 `else if`를 끼워야 해요. 건드리면 안 될 곳을 자꾸 건드리죠.
- "어느 방법을 쓸지 고르는 일"이랑 "그 방법으로 실제 조인하는 일", 성격이 다른 둘이 한 메서드에 엉켜 있어요.
- 해시 조인 하나만 따로 성능을 재고 싶어도, 실행기 전체와 분기 조합을 세워야 그 가지에 닿아요.

어디서 본 그림이죠? 레벨 1이에요. 그때 매칭 방식이 딱 이랬어요. 그러니 푸는 길도 그때랑 같고요. 방향만 짚어 줄게요.

1. 변하는 건 딱 하나, "어떻게 조인하느냐"예요. 이 한 조각을 공통 약속(예: `List<Row> join(Table left, Table right, String key)` 하나짜리 인터페이스) 뒤로 빼면, 실행기는 "조인해 줘"라고 부탁만 하고 속은 모르게 둘 수 있어요.
2. 그렇게 빼 두면, 어떤 방법을 쓸지를 실행기 **바깥에서 골라 끼워** 줄 수 있어요. 단순 반복에서 해시로 갈아끼워도 실행기 코드는 한 줄도 안 바뀌고요. 인덱스 조인이라는 새 방법도, 새 식구 하나 만들어 바깥에서 끼우면 끝이에요. 실행기는 그게 생긴 줄도 몰라요.
3. "고르는 일"은 옵티마이저한테 맡기세요. 누가 고를지는 바깥의 몫이에요. 조인 방법들끼리는 서로를 몰라도 되고, 자기가 언제 쓰일지도 알 필요 없어요. 시키면 조인할 뿐이고요.

끝 그림은 또렷해요. **새 조인 방법을 추가해도 실행기는 한 줄도 안 바뀐다. 어느 걸 쓸지는 바깥(옵티마이저)이 고른다. 방법 하나만 똑 떼어 따로 시험할 수 있다.** 거기까지 가면 after예요.
