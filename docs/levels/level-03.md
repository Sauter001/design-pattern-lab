# 레벨 3: 한 번 일어난 일을 여러 곳에 알리기

안녕하세요. 세 번째 단계예요. 컨테이너에 잠깐 다녀왔으니, 이번엔 다시 MVC로 올라옵니다. 시작 전에 지난 두 단계를 같이 한 줄로 묶어 볼게요. 이번 단계의 모양은 그 둘과 비교할 때 가장 또렷해지거든요.

## 지난 단계 돌아보기

레벨 1에서는 "경로를 비교하는 방식"이라는 한 조각을 빼내 디스패처 바깥에서 골라 끼웠어요. 레벨 2에서는 "시작 뼈대"는 고정해 두고 가운데 한 칸만 컴포넌트가 채우게 했죠. 표현은 달랐지만 두 단계가 답한
질문은 사실 같았어요. **"한 자리에서 무엇이 달라지는가, 그리고 누가 그 자리를 정하는가."** 레벨 1은 바깥이, 레벨 2는 위(순서를 쥔 쪽)가 정했고요.

이번엔 질문 자체가 바뀝니다. 달라지는 건 "한 자리에서 무엇을 하느냐"가 아니라, **"한 가지 일이 일어났을 때, 거기에 반응해야 하는 곳이 몇 군데인가"** 예요. 그리고 그 군데 수가 자꾸 늘어요. 자리가
아니라 개수의 문제, 그게 이번 단계의 색깔입니다.

작업 공간은 다시 `mini-spring/mvc` 모듈(`lab.spring.mvc`)이에요.

## 우리가 만들고 싶은 것

레벨 1에서 만든 디스패처가 요청을 핸들러로 잘 넘긴다고 해 봐요. 이제 핸들러가 진짜 일을 합니다. 회원가입 핸들러를 예로 들게요. 하는 일은 단순해요. 들어온 요청으로 사용자를 하나 만들고, "가입 완료"라고
답해 주면 끝이죠.

문제는 가입이 "끝"이 아니라는 데 있어요. 가입이라는 사건 하나가 일어나면, 그걸 계기로 움직여야 하는 다른 일들이 줄줄이 딸려 옵니다. 환영 메일을 보내야 하고, 감사 로그를 남겨야 하고, 가입 수 통계를 올려야
하죠.

그래서 이번 목표는 이거예요. **하나의 사건(가입)이 일어났을 때, 거기에 따라붙는 여러 부수 작업이 빠짐없이 실행되게 만든다.**

## 가장 정직하게 먼저

핸들러가 직접 다 부르게 해 볼게요. 일어난 일 다음에, 해야 할 일들을 차례로 호출하는 거예요.

```java
class SignupHandler implements Handler {

    private final EmailSender emailSender;
    private final AuditLog auditLog;
    private final SignupStats stats;

    @Override
    public String handle(String path) {
        User user = createUser(path);          // 진짜 할 일: 가입 처리

        emailSender.sendWelcome(user);         // 곁다리 1: 환영 메일
        auditLog.write("SIGNUP", user);        // 곁다리 2: 감사 로그
        stats.increment();                     // 곁다리 3: 가입 수 집계

        return "가입 완료: " + user.name();
    }
}
```

동작은 합니다. 가입 한 줄 처리하고, 뒤따르는 셋을 순서대로 불러 주죠. **여기까지 첫 커밋.** before예요.

## 여기저기서 요청이 들어옵니다

서비스가 자리를 잡으니, 여러 팀이 저마다 "가입할 때 이것도 같이 해달라"며 찾아와요. 받은 말을 그대로 옮기면 이래요.

- 마케팅팀: "가입한 사람한테 환영 쿠폰도 하나 발급해 주세요. 첫 구매로 이어지게요."
- 운영팀: "가입이 평소보다 갑자기 몰리면 슬랙으로 알려 주세요. 이벤트인지 어뷰징인지 봐야 해서요."
- 추천팀: "가입 직후에 그 사람 추천 목록을 미리 데워 두면 첫 화면이 빨라져요. 그것도 끼워 주세요."

시키는 대로 핸들러에 넣어 보세요. 십중팔구 가운데가 이렇게 부풀어 오릅니다.

```java

@Override
public String handle(String path) {
    User user = createUser(path);

    emailSender.sendWelcome(user);
    auditLog.write("SIGNUP", user);
    stats.increment();
    slackNotifier.notifyIfSpike(stats);          // 운영팀
    couponService.issueWelcomeCoupon(user);      // 마케팅팀
    recommendationCache.warm(user);              // 추천팀
    // ... 다음 분기엔 또 뭐가 붙겠죠

    return "가입 완료: " + user.name();
}
```

그리고 이 핸들러는 이제 메일, 로그, 통계, 슬랙, 쿠폰, 캐시... 여섯 군데를 의존하게 됐어요. 생성자도 그만큼 길어졌고요. 게다가 똑같은 압력이 주문 핸들러, 탈퇴 핸들러에도 그대로 옵니다.

**여기까지 두 번째 커밋.** 고통이 드러난 상태예요. 잠깐 멈춰서, 저 `handle` 한복판을 노려보세요.

## 혹시 문제를 느꼈나요?

걸리는 게 한둘이 아닐 거예요.

- 정작 이 핸들러의 본업인 "가입 처리"는 맨 윗줄 하나뿐인데, 그 아래로 남의 일이 잔뜩 쌓여 본업을 덮어 버렸어요.
- "가입하면 이것도 해주세요" 한 건이 들어올 때마다 가입 핸들러를 다시 열어야 해요. 마케팅팀 요구를 넣다가 추천팀 코드를 건드릴 수도 있고, 쿠폰 발급에서 터지면 멀쩡하던 가입까지 같이 죽죠.
- 가입만 따로 테스트하고 싶어도 메일, 슬랙, 쿠폰, 캐시를 전부 세워 줘야 해요. 본업 한 줄을 시험하려고 곁다리 여섯을 데려와야 하는 거예요.

레벨 1, 2와 견줘 보면 결이 다르다는 게 보일 거예요. 그때는 변하는 자리가 "하나"였어요. 비교 방식 한 조각, 시작 뼈대의 빈칸 하나. 지금은 변하는 게 **반응하는 곳의 개수**예요. 하나가 둘, 셋,
여섯으로 자꾸 늘고, 더 고약한 건 가입 핸들러가 그들 하나하나의 이름을 다 알고 직접 불러 주고 있다는 점이에요. 일을 시키는 쪽이 일을 받는 쪽 전부를 손에 쥐고 있는 거죠.

길이 몇 가지 떠오를 거예요. 이름은 끝까지 숨길 거고, 가는 방향만 짚어 줄게요.

1. 가입 핸들러가 해야 하는 말은 사실 딱 한마디예요. "방금 가입이 일어났습니다." 그 한마디(무슨 일이 누구에게 일어났는지 담은 작은 **사건** 한 덩어리)와, 그 일에 무엇으로 반응할지를 갈라놓을 수 있다면
   어떨까요. 핸들러는 알리기만 하고, 반응은 핸들러 바깥의 일로요.
2. 그 한마디를 들어야 할 사람들이 **스스로 손을 들어 등록**하게 하면요? 핸들러는 "가입에 관심 있는 이들"의 목록을 들고 있되 그게 누구누구인지는 모르고, 그냥 전부에게 "이 일이 일어났어요"라고 통지만
   해요. 반응을 더하고 빼는 일이 곧 등록하고 빠지는 일이 되고, 가입 핸들러는 한 줄도 안 바뀌겠죠.
3. 레벨 1과 비교해 보세요. 그때는 쓰는 쪽이 여럿 중 **하나**를 골라 끼웠어요(일대일). 지금은 반대로, 한쪽에서 일어난 일을 그쪽이 알지도 못하는 **여럿**에게 한꺼번에 흘려보내야 해요(일대다).
   화살표가 누구에게서 누구로 향하는지, 그리고 보내는 쪽이 받는 쪽을 알아야 하는지를 다시 그려 보세요.

노리는 그림은 이거예요. **가입에 대한 반응을 추가하거나 빼도 가입 핸들러는 단 한 줄도 바뀌지 않는다. 그리고 가입 핸들러는 메일/쿠폰/슬랙이 무엇인지 더는 알지 못한다.** 거기까지 가면 after예요.

다 되면 세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적되, 이번엔 레벨 1과 꼭 나란히 두고 한 줄 남겨 보세요. "레벨 1도 변하는 걸
바깥으로 뺐고 이번에도 뺐는데, 무엇이 달랐나?" 한쪽은 여럿 중 하나를 고르는 일이었고 다른 쪽은 하나가 여럿에게 알리는 일이었다는 것, 그 차이를 스스로 말로 옮길 수 있으면 이번 단계는 제대로 닿은 거예요.
비용도 잊지 말고요. 사건 객체와 등록 구조가 늘면서 "가입하면 메일이 간다"는 사실이 코드 한 곳에서 안 보이게 됐을 텐데, 그 흐름을 디버깅으로 따라가는 수고가 얼마나 늘었는지까지 솔직하게 적어 두세요. 다음
단계 첫머리에서 같이 되짚겠습니다.
