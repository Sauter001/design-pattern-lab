# 레벨 3: 한 번 일어난 일을 여러 곳에 알리기

다시 봐요. 세 번째 단계예요. 컨테이너에 잠깐 다녀왔으니, 이번엔 다시 MVC로 올라옵니다. 시작 전에 지난 두 단계를 한 줄로 묶어 볼게요. 이번 단계의 모양은 그 둘과 견줄 때 가장 또렷해지거든요.

## 지난 단계 돌아보기

레벨 1에서는 "경로를 비교하는 방식"이라는 한 조각을 빼내 디스패처 바깥에서 골라 끼웠어요. 레벨 2에서는 "시작 뼈대"는 고정해 두고 가운데 한 칸만 컴포넌트가 채우게 했죠. 표현은 달랐지만 두 단계가 답한 질문은 사실 같았어요. **"한 자리에서 무엇이 달라지는가, 그리고 누가 그 자리를 정하는가."** 레벨 1은 바깥이, 레벨 2는 위(순서를 쥔 쪽)가 정했고요.

이번엔 질문 자체가 바뀝니다. 달라지는 건 "한 자리에서 무엇을 하느냐"가 아니라, **"한 가지 일이 일어났을 때, 거기에 반응해야 하는 곳이 몇 군데인가"** 예요. 그리고 그 군데 수가 자꾸 늘어요. 자리가 아니라 개수의 문제, 그게 이번 단계의 색깔입니다.

작업 공간은 다시 `mini-spring/mvc` 모듈(`lab.spring.mvc`)이에요.

## 우리가 만들고 싶은 것

레벨 1에서 만든 디스패처가 요청을 핸들러로 잘 넘긴다고 해 봐요. 이제 핸들러가 진짜 일을 합니다. 회원가입 핸들러를 예로 들게요. 하는 일은 단순해요. 들어온 요청으로 사용자를 하나 만들고, "가입 완료"라고 답해 주면 끝이죠.

문제는 가입이 "끝"이 아니라는 데 있어요. 가입이라는 사건 하나가 일어나면, 그걸 계기로 움직여야 하는 다른 일들이 줄줄이 딸려 옵니다. 환영 메일을 보내야 하고, 감사 로그를 남겨야 하고, 가입 수 통계를 올려야 하죠.

그래서 이번 목표는 이거예요. **하나의 사건(가입)이 일어났을 때, 거기에 따라붙는 여러 부수 작업이 빠짐없이 실행되게 만든다.**

## 가장 정직하게 먼저

핸들러가 직접 다 부르게 해 볼게요. 일어난 일 다음에, 해야 할 일들을 차례로 호출하는 거예요.

```java
class SignupHandler implements Handler {

    private final EmailSender emailSender;
    private final AuditLog auditLog;
    private final SignupStats stats;

    @Override
    public String handle(String path) {
        User user = createUser(path);          // 진짜 할 일: 가입 처리

        emailSender.sendWelcome(user);         // 곁다리 1: 환영 메일
        auditLog.write("SIGNUP", user);        // 곁다리 2: 감사 로그
        stats.increment();                     // 곁다리 3: 가입 수 집계

        return "가입 완료: " + user.name();
    }
}
```

동작은 합니다. 가입 한 줄 처리하고, 뒤따르는 셋을 순서대로 불러 주죠. **여기까지 첫 커밋.** before예요.

## 여기저기서 요청이 들어옵니다

서비스가 자리를 잡으니, 여러 팀이 저마다 "가입할 때 이것도 같이 해달라"며 찾아와요. 들어 보세요.

- 마케팅팀: "가입한 사람한테 환영 쿠폰도 하나 발급해 주세요. 첫 구매로 이어지게요."
- 운영팀: "가입이 평소보다 갑자기 몰리면 슬랙으로 알려 주세요. 이벤트인지 어뷰징인지 봐야 해서요."
- 추천팀: "가입 직후에 그 사람 추천 목록을 미리 데워 두면 첫 화면이 빨라져요. 그것도 끼워 주세요."

시키는 대로 핸들러에 넣어 보세요. 그럼 가운데가 순식간에 불어나요.

```java
@Override
public String handle(String path) {
    User user = createUser(path);

    emailSender.sendWelcome(user);
    auditLog.write("SIGNUP", user);
    stats.increment();
    slackNotifier.notifyIfSpike(stats);          // 운영팀
    couponService.issueWelcomeCoupon(user);      // 마케팅팀
    recommendationCache.warm(user);              // 추천팀

    return "가입 완료: " + user.name();
}
```

이 핸들러는 이제 메일, 로그, 통계, 슬랙, 쿠폰, 캐시... 여섯 군데를 의존하게 됐어요. 생성자도 그만큼 길어졌고요. 그리고 분기 회의 때 마케팅팀이 또 와요. "다음엔 가입 축하 푸시도 넣고 싶은데요." 가운데는 계속 자랍니다. **여기까지 두 번째 커밋.** 일단 박아 두세요.

## 그런데 이게 가입만의 일이 아니에요

여기서 끝났으면 그냥 "긴 메서드" 정도였을 거예요. 진짜 문제는 다음에 와요.

주문이 일어나도 똑같이 딸려 오는 일이 있죠. 주문 확인 메일, 주문 로그, 매출 통계 같은 것들요. 그래서 이번엔 **`OrderHandler`를 진짜로 하나 만들어 볼 거예요.** 사고 실험 말고, 실제 코드로요. 커밋 위치는 신경 쓰지 마세요. before 때 가입 핸들러 처음 짠 거랑 똑같이 naive하게 하나 더 만들어서 옆에 띄우기만 하면 돼요. 어차피 곧 가입이랑 같이 들어낼 거니까요.

지시는 명확하게 할게요. 욕심부리지 말고 딱 이만큼만 하세요.

- `OrderHandler implements Handler` 하나 새로 만들기.
- `handle` 본업은 한 줄(`Order order = createOrder(path);`)이면 됨.
- 곁다리는 **세 개만**. 방금 가입 핸들러에 한 거랑 똑같은 모양으로요. 예: 주문 확인 메일(`emailSender.sendOrderConfirm`), 주문 로그(`auditLog.write("ORDER", ...)`), 매출 통계(`stats.increment`).
- 의존성도 그 셋만 생성자로 주입. 그 이상은 안 해도 돼요. 통증은 셋이면 충분히 와요.

다 짜고 가입 핸들러랑 나란히 띄워 놓고 보세요. 같은 손동작을 두 번째로 반복했다는 게 보일 거예요. 그게 핵심이에요. 잠깐 멈춰서 봐요.

- 가입 핸들러도, 주문 핸들러도, **"내가 일을 끝냈으니 이제 너랑 너랑 너를 부른다"**는 똑같은 짐을 지고 있어요. 부르는 대상만 다를 뿐 구조가 같죠.
- 마케팅팀이 "주문할 때도 쿠폰 주세요" 하면? `OrderHandler`를 또 열어요. 가입 때 `SignupHandler` 열었던 거랑 똑같이요.
- 그러니까 결합이 핸들러 하나에 갇힌 게 아니에요. **핸들러를 늘릴 때마다 같은 결합이 복제돼요.** 일 시키는 쪽마다 받는 쪽 전부의 이름을 외우고 있는 거죠.

이게 이번 단계의 진짜 통증이에요. 한 메서드가 길어진 게 아니라, **"일어난 일을 아는 쪽"이 "거기에 반응할 모두"를 직접 손에 쥐고 있다**는 거. 그리고 그 쥐는 손이 핸들러마다 새로 생겨요.

## 혹시 문제를 느꼈나요?

한둘이 아닐 텐데요. 짚어 볼게요.

- 정작 이 핸들러의 본업인 "가입 처리"는 맨 윗줄 하나뿐인데, 그 아래로 남의 일이 잔뜩 쌓여 본업을 덮어 버렸어요.
- "가입하면 이것도 해주세요" 한 건이 들어올 때마다 가입 핸들러를 다시 열어야 해요. 마케팅팀 요구를 넣다가 추천팀 코드를 건드릴 수도 있고, 쿠폰 발급에서 터지면 멀쩡하던 가입까지 같이 죽죠.
- 가입만 따로 테스트하고 싶어도 메일, 슬랙, 쿠폰, 캐시를 전부 세워 줘야 해요. 본업 한 줄을 시험하려고 곁다리 여섯을 데려와야 하는 거예요.

레벨 1, 2와 견줘 보면 결이 다르다는 게 보일 거예요. 그때는 변하는 자리가 "하나"였어요. 비교 방식 한 조각, 시작 뼈대의 빈칸 하나. 지금은 변하는 게 **반응하는 곳의 개수**예요. 하나가 둘, 셋, 여섯으로 자꾸 늘고, 더 고약한 건 가입 핸들러가 그들 하나하나의 이름을 다 알고 직접 불러 준다는 점이에요. 일을 시키는 쪽이 일을 받는 쪽 전부를 손에 쥐고 있는 거죠.

## 이번 함정을 미리 말해 둘게요

이 단계, 다들 비슷한 데서 한 번 헛디뎌요. 미리 짚을게요.

제일 흔한 게, 곁다리 여섯을 `SignupNotificationService` 같은 묶음 클래스 하나로 싹 옮기는 거예요. 그러면 핸들러는 `signupNotifications.onSignup(user)` 한 줄로 깔끔해지죠. "오, 됐다" 싶고요.

근데 그 묶음 클래스를 열어 보세요. 안에 뭐가 있어요? 메일, 로그, 슬랙, 쿠폰, 캐시를 **여전히 누군가 이름으로 알고 직접 부르고 있어요.** 결합이 사라진 게 아니라 핸들러에서 옆 클래스로 이사 갔을 뿐이에요. 마케팅팀이 새 곁다리를 들고 오면? 그 묶음 클래스를 또 열어야 하고요.

**한쪽이 다른 쪽 전부의 이름을 알고 직접 부른다: 이 모양이 어디로 옮겨가든 남아 있으면 아직 안 풀린 거예요.** 목표는 부르는 코드를 한 군데로 모으는 게 아니라, "일어난 일을 아는 쪽"이 "반응하는 쪽이 누구인지"를 아예 모르게 만드는 거예요. 그 둘은 달라요.

## 방향만 던질게요

길이 몇 가지 떠오를 거예요. 이름은 끝까지 숨길 거고, 가는 방향만 짚어 줄게요.

1. 가입 핸들러가 해야 하는 말은 사실 딱 한마디예요. "방금 가입이 일어났습니다." 그 한마디(무슨 일이 누구에게 일어났는지 담은 작은 **사건** 한 덩어리)와, 그 일에 무엇으로 반응할지를 갈라놓을 수 있다면 어떨까요. 핸들러는 알리기만 하고, 반응은 핸들러 바깥의 일로요.
2. 그 한마디를 들어야 할 쪽들이 **스스로 손을 들어 등록**하게 하면요? 핸들러는 "가입에 관심 있는 이들"의 목록을 들고 있되 그게 누구누구인지는 모르고, 그냥 전부에게 "이 일이 일어났어요"라고 통지만 해요. 반응을 더하고 빼는 일이 곧 등록하고 빠지는 일이 되고, 가입 핸들러는 한 줄도 안 바뀌겠죠. 함정 섹션 다시 봐요. 묶음 클래스는 "부르는 쪽이 받는 쪽을 안다", 이건 "부르는 쪽이 받는 쪽을 모른다". 거기서 갈려요.
3. 레벨 1과 나란히 그려 보세요. 그때는 쓰는 쪽이 여럿 중 **하나**를 골라 끼웠어요(일대일). 지금은 반대로, 한쪽에서 일어난 일을 그쪽이 알지도 못하는 **여럿**에게 한꺼번에 흘려보내야 해요(일대다). 화살표가 누구에게서 누구로 향하는지, 그리고 보내는 쪽이 받는 쪽을 알아야 하는지를 다시 그려 보세요.

노리는 그림은 이거예요. **가입에 대한 반응을 추가하거나 빼도 가입 핸들러는 단 한 줄도 바뀌지 않는다. 그리고 가입 핸들러는 메일/쿠폰/슬랙이 무엇인지 더는 알지 못한다.** 거기까지 가면 after예요. 같은 해법이 주문 핸들러에도 그대로 통하면 제대로 푼 거고요.

## 검증: 결합이 진짜 끊겼는지

다 됐다 싶으면, 커밋 전에 이거 통과하세요.

> "가입할 때 문자 알림도 보내라"는 새 요구가 왔어요. 이 반응 하나를 추가하려고 **`SignupHandler`(또는 그게 부르는 묶음 클래스)를 열어요, 안 열어요?**

- 연다 -> 아직이에요. 일을 아는 쪽이 받는 쪽을 여전히 손에 쥔 거고, 함정에 머문 거예요.
- 안 연다(새 반응 하나 만들어서 "가입에 관심 있다"고 등록만 하면 끝난다) -> 됐어요. 그게 after예요. 일어난 일과 반응이 서로를 모르는 상태죠.

하나 더. 아까 만든 `OrderHandler`도 같이 보세요. before에선 가입이랑 주문이 똑같은 결합을 각자 짊어졌잖아요.

> after에서 `SignupHandler`랑 `OrderHandler`를 나란히 띄워 봐요. 둘 다 "내가 끝났다"고 알리는 그 한 줄 말고, 곁다리를 직접 부르는 코드가 남아 있어요?

둘 다 알리기만 하고 깨끗하면, 결합 복제가 풀린 거예요. before에서 핸들러마다 새로 생기던 그 "쥐는 손"이 사라진 거죠.

## 회고: 레벨 1을 꼭 옆에 펴고

다 되면 세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요.

이번엔 레벨 1을 꼭 옆에 펴 두고 한 줄 보태세요.

> 레벨 1도 변하는 걸 바깥으로 뺐고 이번에도 뺐는데, 무엇이 달랐나? 한쪽은 여럿 중 하나를 고르는 일이었고(일대일), 다른 쪽은 하나가 여럿에게 알리는 일이었어요(일대다).

그 차이를 스스로 말로 옮길 수 있으면 이번 단계는 제대로 닿은 거예요.

비용도 잊지 마요. 이번 비용은 좀 특이해요. 코드는 깔끔해졌는데 **"가입하면 메일이 간다"는 사실이 코드 한 곳에서 안 보이게 됐어요.** before에선 `handle` 한복판을 보면 무슨 일이 딸려 오는지 다 보였잖아요. 이제는 등록된 쪽들을 일일이 뒤져야 흐름이 보여요. 추가/삭제는 거저 됐지만, "지금 가입에 뭐가 반응하고 있나"를 한눈에 보는 능력을 내준 거예요. 디버깅으로 그 흐름을 따라가는 수고가 얼마나 늘었는지까지 솔직하게 적어 두세요. 다음 단계 첫머리에서 다시 꺼낼게요.

## 다 끝났다면, 답을 맞춰 보세요
