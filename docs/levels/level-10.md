# 레벨 10: 하나가 아니라, 어긋나면 안 되는 한 벌을 만든다

또 봐요. 열 번째예요. 두 자리 수 왔네요, 우리. 지난 단계 끝에 제가 그랬죠. 다음엔 하나가 아니라 한 벌을 만든다고. 그거 하러 왔어요. 레벨 5하고 6, 7하고 8이 그랬듯 9하고 10도 한 쌍이에요.
그러니 레벨 9를 옆에 펴 둔 채로 시작하세요. 자꾸 비교하라고 할 거예요.

동네는 안 옮겨요. 그대로 `mini-spring/container` 모듈(`lab.spring.container`)이에요. 레벨 9에서 빈 만드는 일을 위임했던 그 자리에서 한 발 더 갑니다.

## 지난 단계 돌아보기

레벨 9에서 우리는 컨테이너가 빈을 직접 `new` 하지 않게 했어요. 빈마다 자기 만드는 법을 쥐고, 컨테이너는 "만들어 줘"라고 시키기만 했죠. 핵심은 "빈 하나를 만드는 일을 위임한다"였어요. 딱 하나요.

오늘은 그림이 거의 똑같아요. 만드는 일을 위임하는 것까지요. 그런데 딱 두 가지가 달라집니다. 하나, 하나가 아니라 여러 개를 만들어요. 둘, 그 여러 개가 **서로 어긋나면 안 돼요.** 이 두 번째가 오늘의
전부예요. 무슨 말인지는 만들어 보면 알아요.

## 이번에 마주칠 상황

컨테이너가 인프라 빈 몇 개를 엮어야 한다고 해 봐요. 데이터소스, 캐시 매니저, 트랜잭션 매니저. 이 셋은 따로 노는 게 아니라 한 세트예요. 운영 환경이면 셋 다 운영용이어야 하고, 테스트 환경이면 셋 다
테스트용이어야 하죠. 운영 데이터소스에 테스트 트랜잭션 매니저가 물리면? 그 순간 데이터가 안 맞거나, 더 나쁘면 조용히 깨져요. 섞이면 안 되는 거예요.

이번 목표는 이거예요. **환경에 맞는 빈 한 벌을 통째로, 서로 어긋나지 않게 만든다. 그리고 환경을 갈아끼우는 일은 한 번에 끝난다.**

## 가장 정직하게 먼저

컨테이너가 제품마다 환경을 따져서 만들어 주게 해 볼게요. 제일 정직하게요.

```java
class Container {

    DataSource createDataSource(String env) {
        if (env.equals("prod")) return new PostgresDataSource(props);
        else return new InMemoryDataSource();
    }

    CacheManager createCacheManager(String env) {
        if (env.equals("prod")) return new RedisCacheManager(props);
        else return new LocalCacheManager();
    }

    TxManager createTxManager(String env) {
        if (env.equals("prod")) return new JdbcTxManager(props);
        else return new NoopTxManager();
    }
}
```

동작은 해요. `env` 보고 환경에 맞는 걸 만들어 주죠. **여기까지 첫 커밋.** before예요.

## 환경이 늘고, 식구도 늘어요

서비스가 자라면서 환경도 빈 종류도 같이 늘어요.

- 인프라팀: "스테이징 환경도 추가해요. 운영이랑 거의 같은데 캐시만 로컬로요."
- QA팀: "통합 테스트용 한 벌도요. 그건 DB는 인메모리인데 트랜잭션은 진짜 JDBC로 굴려야 해요."
- 플랫폼팀: "인프라 식구에 메시지 브로커도 들어와요. 환경마다 다른 거 쓰고요."

시키는 대로 넣어 보세요. 환경 하나 늘 때마다 `createDataSource`도 열고, `createCacheManager`도 열고, `createTxManager`도 열어서 똑같은 `else if`를 셋 다
똑같이 붙여요. 어, 이 손맛 어디서 봤죠? 레벨 2의 산탄총 수술이에요. 또 만났네요. 반갑죠? 전 안 반가워요.

```java
DataSource createDataSource(String env) {
    if (env.equals("prod")) return new PostgresDataSource(props);
    else if (env.equals("staging")) return new PostgresDataSource(stagingProps);  // 여기 한 줄
    else return new InMemoryDataSource();
}
// createCacheManager에도, createTxManager에도, 새로 생길 createMessageBroker에도... 같은 분기가 줄줄이
```

**여기까지 두 번째 커밋.** 일단 박아 두세요.

## 혹시 문제를 느꼈나요?

이번엔 거의 보일 거예요. 그래도 짚어 둘게요.

- "환경이 뭐냐"는 결정이 제품마다 흩어져 반복돼요. `env.equals("prod")`가 세 군데, 곧 네 군데. 환경 하나 늘면 그걸 전부 다시 열어야 하고요. 레벨 2에서 본 그 산탄총 수술 맞아요.
- 한 벌이 어긋나는 걸 아무도 안 막아요. 누가 `createDataSource("prod")`랑 `createCacheManager("test")`를 같이 부르면 운영 DB에 테스트 캐시가 물려요. 짬뽕이 되는데
  컴파일러도 그냥 통과시키죠. 제일 무서운 종류의 사고예요. 조용하거든요.
- 새 식구(메시지 브로커)를 한 벌에 추가하려면, 그 식구 만드는 메서드에 환경 분기를 처음부터 또 다 깔아야 해요.

레벨 9랑 나란히 놓고 보세요. 거기서도 "만드는 일"을 다뤘죠. 그런데 거긴 빈 하나를 **어떻게** 만드느냐가 문제였어요. 여기선 달라요. 하나하나를 어떻게 만드느냐가 아니라, **여러 개가 같은 환경에서 나온
한 벌로 묶여 있느냐**가 문제예요. 변하는 자리가 "만드는 법"이 아니라 "한 벌의 일관성"인 거죠.

## 방향만 던질게요

길이 떠오를 거예요. 답은 본인 몫이고, 방향만 대 줄게요.

1. 환경별로 흩어진 그 분기들을 한 군데로 모을 수 없을까요. "운영 한 벌을 만드는 자"와 "테스트 한 벌을 만드는 자"를 따로 두고, 각자가 데이터소스/캐시/트랜잭션을 자기 환경 것으로만 내놓게요. 그러면
   운영용을 만드는 자한테선 테스트용이 절대 안 나와요. 어긋날 수가 없죠.
2. 컨테이너는 환경을 보고 "한 벌 만드는 자" 하나만 고르면 돼요. 그다음부턴 그 자한테 "데이터소스 줘, 캐시 줘" 하고 받기만 하고요. 환경을 통째로 바꾸는 게 그 "만드는 자" 하나를 갈아끼우는 일이 되는
   거예요. 세 군데, 네 군데 안 돌아다녀도요.
3. 레벨 9랑 갈리는 데를 보세요. 레벨 9는 빈 하나를 만드는 일을 위임했어요. 오늘은 한 벌을 만드는 일을 위임하는데, 그 위임의 핵심이 "한 벌이 같은 출처에서 나온다"는 보장이에요. 하나냐 한 벌이냐,
   그리고 그 한 벌의 일관성을 누가 보장하느냐. 그 한 끗이 두 단계를 가릅니다.

도착점은 또렷해요. **새 환경을 추가해도 기존 제품 만드는 코드는 안 바뀐다. 환경 하나당 "한 벌 만드는 자" 하나를 새로 만들면 끝. 그리고 운영과 테스트가 섞인 짬뽕 한 벌은 구조적으로 못 만든다.**
거기까지 가면 after예요.

검증은 레벨 9랑 갈리는 그 한 끗이 곧 기준이에요. 커밋 전에 두 개만 자문해요. 첫째, 환경 하나 추가하는데 제품마다 메서드를 다 여나요, "한 벌 만드는 자" 하나 추가로 끝나나요? 둘째, 운영 데이터소스에
테스트 캐시를 물린 짬뽕 한 벌을 실수로라도 만들 수 있나요? 제품마다 다 열어야 하거나 짬뽕이 만들어지면 아직이에요. 한 곳만 갈아끼우면 되고 짬뽕이 구조적으로 막히면 after고요.

## 회고: 레벨 9를 옆에 펴고

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도 적어요. 환경마다 "한 벌 만드는 자" 클래스가 하나씩 생겨요. 환경 셋이면 셋.
그리고 이번엔 비용에 한 가지가 더 있어요. 한 벌에 식구를 새로 추가하는 게 비싸요. 모든 "만드는 자"에 그 식구 만드는 법을 다 더해야 하거든요. 환경 추가는 싸고 제품 추가는 비싼 구조라는 거, 적어 두세요.
레벨 9는 정확히 반대였고요. 새 제품 타입 추가가 쌌죠. 어느 쪽 확장을 싸게 만들지를 고른 거예요. 공짜는 없어요.

그리고 이번엔 꼭 레벨 9 회고를 옆에 펴 놓고 한 줄 적으세요.

> 빈 하나를 만드는 위임(레벨 9)과 빈 한 벌을 만드는 위임(레벨 10)은 무엇이 달랐나? 그리고 확장이 싼 방향이 둘에서 어떻게 갈렸나?

그 답이 이 쌍의 전부예요. 코드 그림은 닮았는데 만드는 단위가 다르거든요.
