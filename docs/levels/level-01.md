# 레벨 1: 요청을 알맞은 핸들러에게 보내기

후... 왔군요. 첫 단계예요.

가볍게 가죠. 몸 푸는 구간이니까. 우리가 만들 건 미니 스프링의 MVC고, 미리 말하면 새로울 거 하나도 없어요. 어디선가 한 번쯤 써 봤을 도메인이고, 뭘 발명하는 게 아니라 알던 걸 손으로 다시 짜는
거예요.

시시해 보이죠? 그럴 거예요. 근데 그 시시한 걸 끝까지 제대로 하는 사람이 의외로 드물어서, 여기서부터 시작합니다. 진짜 목적은 따로 있어요. before -> 고통 -> after, 이 리듬을 머리 말고 손에
새기는 거예요.

약속 하나 하고 가죠. 이 단계가 무슨 패턴인지 저는 끝까지 말 안 합니다. 알려달라고 해도 소용없어요. 그건 마지막에 본인이 직접 만나야 하는 거고, 김새게 하는 짓은 제가 제일 싫어하거든요.

작업 공간은 `mini-spring/mvc` 모듈(`lab.spring.mvc`)이에요.

## 우리가 만들고 싶은 것

웹 프레임워크라는 게 결국 뭐 하는 물건이냐면, 들어온 요청을 누가 처리할지 고르는 거예요. 그게 심장이에요. 클라이언트가 `GET /users`를 보내면 사용자 핸들러가, `GET /orders`면 주문 핸들러가
받아야겠죠. 당연한 얘기죠. 근데 그 당연한 걸 코드로 옮기는 게 또 일이라서요.

첫 목표는 이겁니다. **요청 경로를 받아서, 그 요청을 처리할 핸들러를 찾아 주는 작은 디스패처를 만든다.** 그게 다예요. 지금은요.

핸들러는 일단 단순하게 둬요. 뭘로 응답할지만 아는 객체면 됩니다. 그 이상은 욕심이에요.

```java
interface Handler {
    String handle(String path);
}
```

디스패처는 "어떤 경로를 어떤 핸들러가 맡는지" 등록해 두고, 요청이 오면 맞는 놈을 골라 주면 돼요. 제일 정직한 방법부터 가죠. 머리 굴리지 말고요.

```java
class DispatcherServlet {

    // 경로 패턴 -> 핸들러
    private final Map<String, Handler> registry = new LinkedHashMap<>();

    void register(String pattern, Handler handler) {
        registry.put(pattern, handler);
    }

    Handler getHandler(String path) {
        for (var entry : registry.entrySet()) {
            if (path.equals(entry.getKey())) {   // 등록한 경로와 정확히 같은가
                return entry.getValue();
            }
        }
        throw new IllegalStateException("핸들러 없음: " + path);
    }
}
```

`register("/users", ...)`, `register("/orders", ...)`로 등록하고, `getHandler("/users")`가 맞는 핸들러를 돌려주면 성공이에요. 작은 테스트로 초록불 한 번
확인하고요.

**여기까지가 첫 커밋이에요.** 이게 before입니다. 못생겼다고요? 동작하잖아요. 지금은 그걸로 충분해요.

## 그런데, 요구사항이 들어옵니다

며칠 지나면 옵니다. 늘 와요. 기획자가 기능을 하나씩 들고 오기 시작해요. 받은 말을 각색 없이 그대로 옮길게요.

- "정적 파일은 `/static/`으로 시작하는 경로면 전부 정적 핸들러로 내려가게 해주세요."
- "관리자 화면은 `/admin/*` 경로에서 처리되게요. 그 아래 한 칸은 뭐가 오든 관리자고요."
- "예약 기능도 곧 붙어요. `/reservations/*`, `/rooms/*`, `/payments/*`... 점점 늘 텐데 알아서 잘 받아 주세요."

받았으니 해야죠. 시키는 대로, 지금 `getHandler`에 그대로 욱여넣어 보세요. 십중팔구 저 `if (path.equals(...))` 한 줄이 이렇게 부풀어 오릅니다.

```java
Handler getHandler(String path) {
    for (var entry : registry.entrySet()) {
        String pattern = entry.getKey();
        boolean matched;
        if (pattern.endsWith("/*")) {
            matched = path.startsWith(pattern.substring(0, pattern.length() - 2));
        } else if (pattern.endsWith("/")) {
            matched = path.startsWith(pattern);
        } else {
            matched = path.equals(pattern);
        }
        if (matched) return entry.getValue();
    }
    throw new IllegalStateException("핸들러 없음: " + path);
}
```

동작은 해요. 동작은요. **여기까지가 두 번째 커밋입니다.** 마음에 드세요? 안 들 텐데. 그래도 일단 초록불 확인하고 커밋부터 박으세요. 다음이 진짜니까.

## 잠깐. 하나만 더 해 보죠

여기서 끝내면 안 돼요. 아까 기획자가 흘린 마지막 말, 기억나요? 안 적었을 텐데. "점점 늘 텐데"요. 그 "나중에"가 진짜로 나중에 안 끝난다는 거, 한 번 겪어 보면 압니다. 그래서 지금 미리 한 번 겪어
보죠.

요구사항 하나가 더 왔다고 칩시다. 자주 와요.

- "아, 그리고 경로 매칭을 정규식으로도 하고 싶어요. `regex:^/api/v[0-9]+/.*` 이런 식으로요."

자, 이거 넣으세요. **방법은 안 알려줄게요.** 방금 짠 그 코드 위에서, 본인이 알아서 정규식 매칭을 추가해 보세요. 지금 당장. (커밋은 따로 안 박아도 돼요. 어차피 곧 다 같이 들어낼 거라, 지금은 손으로 한 번 겪어 보는 게 목적이에요.)

...

넣었어요? 그럼 손 떼고, 질문 세 개에 답해 보세요. 머릿속으로 말고, 커밋 메시지든 메모든 어디든 **숫자로** 적으세요.

1. 정규식 하나 추가하려고 **연 파일이 몇 개**예요?
2. 그중에 `DispatcherServlet.java`가 있어요, 없어요?
3. 이거 추가하고 나서 **다시 돌린 테스트가 몇 개**예요? 정규식이랑 아무 상관 없는 정확매칭 테스트도 다시 돌렸죠?

십중팔구 답은 이래요. `DispatcherServlet`을 또 열었고, `getHandler` 한복판에 `else if`를 또 끼웠고, 그 메서드 건드렸으니 거기 딸린 테스트 전부를 다시 돌렸어요. 매칭 방식 하나
늘었을 뿐인데요.

이 단계, 다들 여기서 똑같아요. 매년 보는 풍경이에요. 새 매칭 방식이 올 때마다 심장(`getHandler`)을 열어요. 심장은 건드리면 안 되는 데인데요.

## 혹시 문제를 느꼈나요?

이제 좀 걸리죠. 안 걸리면 제가 짚죠.

- 매칭 "방식"이 하나 늘 때마다 디스패처의 핵심 메서드를 다시 열어서 `else if`를 끼워 넣어야 해요. 방금 본인 손으로 두 번 했잖아요. 건드리면 안 될 곳을 자꾸 건드리게 되죠.
- "경로를 순회하며 핸들러를 찾는 일"이랑 "경로를 어떻게 비교할지 정하는 일", 성격이 완전 다른 두 가지가 한 메서드에 엉켜 있어요.
- 접두사 매칭만 똑 떼서 따로 테스트하고 싶은데, 그러려면 디스패처 전체를 세워야 해요.

## 함정 하나, 미리 막고 가죠

이 단계, 다들 여기서 두 갈래로 갈려요. 매번 봐요.

한쪽은 저 `else if` 덩어리를 `private boolean matches(String pattern, String path)` 메서드 하나로 쏙 빼내요. 그럼 `getHandler`는 깔끔해 보이죠. "
리팩토링 끝!" 싶고요. 근데 매칭 방식 하나 더 늘면? 그 추출한 `matches` 안에 `else if`가 또 붙어요. `DispatcherServlet.java`를 또 열어야 하고요. **아무것도 안 옮긴
거예요. 줄만 바꾼 거죠.**

다른 한쪽은 그 덩어리를 아예 디스패처 **바깥으로** 들어내요. 그래서 새 매칭 방식이 와도 디스패처 파일은 안 열려요.

십중팔구 첫 번째 길로 가요. 그리고 다음 요구사항에서 다시 후회하죠. 안 봐도 알아요. 그러니까 본인이 어느 쪽으로 가고 있는지, 가는 도중에 한 번 멈춰서 확인해요.

## 방향만 던질게요

푸는 길이 두세 개쯤 보일 거예요. 정답이요? 말 안 해요, 알잖아요. 방향만 던질게요. 거기까진 해 드리죠.

1. 변하는 부분은 딱 하나예요. "이 경로가 이 패턴에 맞는가"를 판단하는 **비교 방식**. 이 한 조각만 따로 떼어내서, 공통된 약속(예:
   `boolean matches(String pattern, String path)` 하나짜리 인터페이스) 뒤에 숨길 수 있다면요. 디스패처는 "맞는지 봐 줘"라고 부탁만 하고, 어떻게 보는지는 모르게요.
2. 그 비교 방식 하나를 디스패처 **바깥에서 골라 끼워** 줄 수 있다면? 정확매칭에서 패턴매칭으로 갈아 끼우는데도 디스패처 코드는 한 줄도 안 바뀌겠죠.
3. 위 함정 얘기 다시 봐요. 메서드로 빼는 거랑 인터페이스 뒤로 들어내는 거, 뭐가 다른지가 이 단계의 전부예요. "어디에 `else if`가 또 붙느냐"로 자문해 보면 갈려요.

목표 지점은 분명해요. **새 매칭 방식을 추가할 때 디스패처의 핵심 메서드는 단 한 줄도 안 바뀐다.** 거기까지 가면 그게 after예요. 새 방식이 와도 본체가 멀쩡한 그 그림, 솔직히 꽤 깔끔하거든요. 제가
좋아하는 몇 안 되는 거고요.

## 검증: after가 진짜 after인지

다 됐다 싶으면, 커밋 박기 전에 이거 한 번만 통과하세요. 거짓말 못 하는 질문이에요.

> 매칭 방식을 하나 더 추가한다고 칩시다(아무거나, 예: 대소문자 무시 매칭). 그거 추가하려고 **`DispatcherServlet.java`를 열어요, 안 열어요?**

- 연다 -> 아직 after 아니에요. 메서드 추출 함정에 빠진 거예요. 위 함정 섹션 다시 보고 가요.
- 안 연다 -> 됐어요. 그게 after예요. 새 파일 하나 만들어서 끼우기만 하면 끝나는 그림이죠.

이 질문 통과하면 세 번째 커밋이에요.

## 회고: 비용을 꼭 적으세요

회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 이건 농담 안 하고 말할게요. 무엇이 어디로 옮겨졌는지보다, 그 대가로 **뭐가 늘었는지**를 꼭 같이
적으세요.

- 클래스(파일) 수가 before 대비 몇 개 늘었어요?
- 요청 하나 처리하는 데 따라가야 할 간접 단계가 몇 단계 늘었어요? (디스패처 -> 인터페이스 -> 구현체)
- before에서 정규식 추가할 때 건드린 파일 수 vs after에서 건드릴 파일 수. 이 둘을 나란히 적으세요. 아까 숫자로 세 둔 거 여기서 써먹어요.

패턴은 공짜가 아니에요. 복잡도를 한 곳에서 딴 곳으로 옮기는 거래일 뿐이거든요. 클래스 하나 더 생긴 만큼, 심장은 안 열리게 됐죠. 그 거래가 남는 장사였는지는 본인이 숫자로 적어 봐야 알아요.

## 다 끝났다면, 답을 맞춰 보세요

after 커밋이랑 회고까지 끝냈어요? 그럼 이제 본인이 다시 발견한 그 구조에 이름이 있었는지 확인할 차례예요. [정답과 이정표](answers/level-01.md)를
열어서, [refactoring.guru](https://refactoring.guru/ko/design-patterns) 카탈로그랑 본인 코드를 나란히 놓고 대조해 보세요. 이름이 같아도 본인 버전이랑 다른 데가
분명 있어요. 그 차이를 회고에 한 줄 보태면, 그게 진짜 재발견이에요.

아직 before -> 고통 -> after를 다 안 거쳤다고요? 그럼 지금은 닫아요. 이름을 미리 알면 "이 변경 압력을 어떻게 풀까"라는 고민이 "그 패턴을 어떻게 쓸까"로 변질돼요. 그 순간 이 단계에서 가져갈
게 반토막 나고요. 급할 거 없잖아요. 아마.
