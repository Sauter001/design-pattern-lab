# 레벨 19: 규칙이 줄줄이 느는데, 한 메서드에 다 엉키지 않게

또 봐요. 열아홉 번째예요. 동네 옮겨요. 레벨 17, 18에서 미니 스프링 컨테이너에 있었는데, 다시 미니 DB로 건너갑니다. 레벨 7, 11에서 갔던 그 엔진이에요. 작업 공간은 `mini-db/engine` 모듈(`lab.db.engine`)이고요.

오늘은 미리 흘려뒀던 걸 거두러 가요. 레벨 7 기억나요? 조인 방법을 갈아끼우면서, "어느 게 빠를지 정하는 옵티마이저가 붙기 시작한다"고 했죠. 오늘은 그 옵티마이저 안을 들여다봐요. 그게 실제로 무슨 일을 하는지요.

## 거슬러 올라가 돌아보기

레벨 7에서 옵티마이저는 바깥에서 조인 방법을 "골라 끼우는" 손이었어요. 여럿 중 하나를 골라 갈아끼웠죠. 한 번에 하나만 쓰고요.

오늘 그릴 그림은 그것과 닮았는데 결정적으로 달라요. 옵티마이저는 파싱된 쿼리 계획을 받아서, 실행 전에 여러 규칙으로 다듬어요. 그런데 이번엔 규칙 여럿 중 하나를 고르는 게 아니라, **여럿을 순서대로 줄줄이 다 거쳐요.** 레벨 7이 "갈아끼우기"였다면 오늘은 "이어달리기"예요. 이 차이를 머리에 걸어 두고 시작해요. 끝에서 다시 꺼낼 거예요.

그리고 레벨 11에서 우리가 지었던 그 `Query` 객체, 기억나죠? 절을 골라 얹어 만들던 거요. 오늘 규칙들이 주무르는 게 바로 그 계획이에요. 우리가 지은 걸 우리가 다듬는 거죠.

## 이번에 마주칠 상황

옵티마이저는 계획을 받아서 이리저리 손봐요. 규칙을 적용해서요. 몇 개만 들면.

- 상수 접기: `WHERE 1 = 1` 같은 건 미리 계산해서 치워요.
- 조건 밀어넣기: WHERE 조건을 가능한 한 스캔 가까이로 내려, 행을 일찍 걸러요.
- 안 쓰는 컬럼 자르기: SELECT에 없는 컬럼은 위로 안 올려요.

각 규칙은 계획을 보고, 자기가 손댈 게 있으면 고치고, 없으면 그냥 둬요. 그리고 다음 규칙으로 넘기고요.

이번 목표는 이거예요. **규칙을 적용하는 그 한 곳이, 규칙이 늘고 순서가 바뀌고 일부만 켜져도 헤집어지지 않게 한다.**

## 가장 정직하게 먼저

옵티마이저 메서드 하나에 규칙들을 차례로 박아 볼게요. 제일 정직하게요.

```java
Plan optimize(Plan plan) {
    // 1) 상수 접기
    if (hasConstantPredicate(plan)) plan = foldConstants(plan);
    // 2) 조건 밀어넣기
    if (hasPushablePredicate(plan)) plan = pushPredicatesDown(plan);
    // 3) 안 쓰는 컬럼 자르기
    if (hasUnusedColumns(plan)) plan = pruneColumns(plan);
    return plan;
}
```

동작은 해요. 계획이 세 규칙을 차례로 거쳐 다듬어져 나오죠. **여기까지 첫 커밋.** before예요.

솔직히 말할게요. 지금 이거, 규칙 셋뿐이면 나쁘지 않아요. 외려 깔끔하죠. 그러니 여기서 "이건 고쳐야 해" 하고 달려들지 마세요. 문제는 지금이 아니라 다음이에요. 규칙이 늘기 시작할 때요.

## 규칙이 자꾸 늡니다

옵티마이저팀이 또 와요. 레벨 7의 그 팀요. 이번엔 주문이 이래요.

- "규칙이 계속 늘어요. 인덱스 선택, 조인 순서 바꾸기, 중복 서브쿼리 제거... 올 때마다 이 메서드 한가운데에 `if` 한 칸씩 끼우고 있어요."
- "순서가 중요해요. 조건 밀어넣기는 컬럼 자르기보다 먼저 돌아야 하거든요. 순서를 바꾸려면 메서드 안에서 줄을 위아래로 옮겨야 하고, 옮기다 옆 줄을 건드려요."
- "배포마다 켜는 규칙이 달라요. 실험 중인 규칙은 운영에선 끄고 싶은데, 지금은 주석 처리하거나 플래그 `if`로 메서드를 떡칠해요."
- "규칙 하나가 계획을 어떻게 바꿨는지 따로 로깅하고 싶어요. 근데 한 메서드에 다 엉켜서 규칙별로 떼어 볼 수가 없어요. 하나만 떼서 테스트도 못 하고요."

시키는 대로 넣어 보세요. 그 메서드가 규칙 수만큼 부풀고, `if` 사이사이에 플래그가 끼고, 순서 의존이 오로지 "줄 위치"로만 표현돼요.

```java
Plan optimize(Plan plan, RuleFlags flags) {
    if (flags.constantFolding && hasConstantPredicate(plan)) plan = foldConstants(plan);
    if (flags.predicatePushdown && hasPushablePredicate(plan)) plan = pushPredicatesDown(plan);
    // 인덱스 선택이 오면? 여기 또 한 칸. 순서 맞춰서. 늘 그렇듯.
    if (flags.columnPruning && hasUnusedColumns(plan)) plan = pruneColumns(plan);
    return plan;
}
```

**여기까지 두 번째 커밋.** 고통이 드러난 상태예요.

## 혹시 문제를 느꼈나요?

이쯤이면 어디가 걸리는지 말 안 해도 보일 거예요. 그래도 짚어는 둘게요.

- 규칙이 하나 늘 때마다 옵티마이저 본체를 열어 한가운데에 `if`를 끼워요. 건드리면 안 될 곳을 매번 건드리죠.
- 순서가 코드 줄 위치로만 표현돼서, 순서를 바꾸려면 본체를 수술해야 해요.
- 켜고 끄기가 플래그 `if`로 본체에 박혀, 본체가 규칙 목록에다 온오프 정책까지 다 떠안아요.
- 규칙 하나만 똑 떼어 테스트하거나 계측할 수가 없어요. 다 엉켜 있으니까요.

레벨 7이랑 헷갈리지 마세요. 클래스 관계도만 그리면 닮아 보일 거예요. 근데 거긴 조인 방법 여럿 중 하나를 골라 갈아끼웠죠. 한 번에 하나만 썼고요. 오늘은 규칙 여럿을 순서대로 다 거쳐요. 갈아끼우기가 아니라 이어달리기예요. 그래서 푸는 모양도 달라요.

## 방향만 던질게요

길이 떠오를 거예요. 방향만 대 줄게요.

1. 규칙 하나하나를 독립된 조각으로 떼어내세요. 각 조각은 딱 두 가지만 알면 돼요. "이 계획에 내가 손댈 게 있나, 있으면 고친다" 그리고 "어쨌든 다음으로 넘긴다." 옆 규칙이 뭔지, 자기가 몇 번째인지는 몰라도 되고요.
2. 그 조각들을 한 줄로 꿰세요. 순서는 꿰는 곳에서 정하고요. 계획이 그 줄을 따라 흐르며 조각을 차례로 거쳐요. 새 규칙은 줄에 한 칸 끼우면 끝이라 본체는 안 열려요. 순서 바꾸기는 꿰는 순서만 바꾸면 되고, 끄기는 줄에서 빼면 되죠. 배포마다 다른 줄을 꿰면, 켜는 규칙도 배포마다 달라지고요.
3. 조각이 독립적이니, 규칙 하나만 떼어 테스트하고 계측할 수 있어요. 그 조각 하나에 입력 계획 주고 출력 계획 보면 끝이에요.
4. 균형추, 이게 오늘의 진짜 교훈이에요. 규칙이 끝까지 두세 개뿐이고 순서도 안 바뀌고 늘 다 켜져 있다면, 맨 처음 그 `if` 세 줄이 더 정직해요. 줄로 꿰는 건 공짜가 아니거든요. 규칙이 많고, 자주 늘고, 순서나 온오프가 출렁일 때라야 본전을 뽑아요.

이건 농담 안 하고 말할게요. 규칙이 셋인데 이걸 줄로 꿰면, 패턴을 쓰는 게 아니라 패턴한테 끌려다니는 거예요. "앞으로 늘 거다"라는 확신이 설 때만 미리 꿰세요. 그 확신이 없으면, 안 꿰고 그대로 두는 게 오늘의 정답이에요. 그리고 그 판단을 회고에 적는 게, 꿰는 법을 아는 것만큼 값나가요. 이 lab은 "안 쓴 이유"를 쓴 이유만큼 쳐요. 레벨 11, 15에서도 같은 말 했죠. 또 해요. 중요하니까요.

도착점은 또렷해요. **새 옵티마이저 규칙을 추가해도 본체는 한 줄도 안 바뀐다(줄에 한 칸 끼울 뿐). 순서는 꿰는 순서로, 온오프는 줄에 넣고 빼기로 정해진다. 규칙 하나를 똑 떼어 따로 시험할 수 있다.** 거기까지 가면 after예요. 단, 규칙이 적으면 안 가는 게 맞다는 것도 같이 기억하고요.

검증은 짧아요. 커밋 전에 자문해요. 새 규칙(예: 조인 순서 바꾸기)을 붙이는데 옵티마이저 본체를 여나요, 안 여나요? 그리고 두 규칙의 순서를 바꾸는데 본체 줄을 옮기나요, 꿰는 곳만 바꾸나요? 본체 안 열고 꿰는 곳만 손대면 after예요. 그리고 규칙이 적어서 안 꿰기로 했다면, 그것도 정답이고요.

## 회고: 줄로 옮겼다고 순서 문제가 사라진 건 아니에요

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도 꼭이요. 규칙마다 클래스가 하나씩 늘어요(조각 수만큼). 계획 한 번 다듬는 데 줄을 따라 여러 칸을 거치니 간접 단계가 늘고요. 디버깅할 땐 어느 칸에서 계획이 바뀌었는지 줄을 따라가야 해요.

그리고 함정 하나. 규칙 사이 숨은 순서 의존(조건 밀어넣기는 컬럼 자르기보다 먼저)은, 줄 꿰는 곳에 적어 두지 않으면 여전히 사고로 와요. 줄로 옮겼다고 순서 문제가 저절로 사라지는 게 아니거든요. 자리만 옮긴 거예요. 그 의존을 어디에 어떻게 적어 뒀는지 회고에 남기세요.

레벨 7 회고랑 나란히 놓고 한 줄 적어요. 레벨 7은 "여럿 중 하나를 바깥이 골라 갈아끼움", 오늘은 "여럿을 순서대로 다 거침"이죠. 클래스 그림은 비슷해도(공통 약속 하나에 구현 여럿) 계획이 흐르는 방식이 달라요. 같은 그림인데 또 다른 것, 레벨 5와 6, 7과 8에서 해 본 그 비교를 한 번 더 하는 거예요.
