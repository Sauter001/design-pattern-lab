# 레벨 13: 되돌리려고 남의 속을 뒤지지 않기

또 봐요. 열세 번째예요. 동네는 안 옮겨요. 레벨 8, 12에서 있던 미니 DB 저장 층 그대로예요. `mini-db/storage` 모듈(`lab.db.storage`)이고요.

레벨 8 기억나요? 트랜잭션이 상태를 ACTIVE에서 COMMITTED나 ABORTED로 넘겼죠. 그때 ABORTED, 그러니까 "엎는다"는 걸 만들긴 했는데, 정작 진짜로 되돌리는 건 `log.undo()` 한 줄로 얼버무렸어요. 오늘 그 얼버무린 걸 제대로 합니다. "되돌린다"를 진짜로 구현해요.

## 지난 단계 돌아보기

레벨 8에서 트랜잭션은 자기 상태를 스스로 넘겼어요. ACTIVE에서 abort를 받으면 ABORTED로요. 거기서 "되돌린다"는 건 그냥 있다 치고 넘어갔죠. 오늘은 그 되돌리기 자체를 들여다봐요. 엎으려면 엎기 전 상태를 어딘가에 떠 놨어야 하거든요. 그 "떠 놓고 되돌리기"를 어떻게 하느냐가 오늘 문제예요. 쌍은 아니고, 혼자 가는 단계고요.

## 이번에 마주칠 상황

트랜잭션이 데이터를 고쳐요. 페이지(`Page`) 하나를 바꾼다고 해 봐요. 그런데 트랜잭션이 엎어지면, 그 페이지를 고치기 전으로 되돌려야 해요. 그러려면 고치기 전에 그 상태를 어딘가 떠 놨어야 하고요.

이번 목표는 이거예요. **트랜잭션이 페이지를 고치기 전 상태를 떠 뒀다가, 엎어질 때 그 상태로 되돌린다.**

## 가장 정직하게 먼저

트랜잭션이 페이지 속을 통째로 베껴 두고, 되돌릴 때 도로 써넣게 해 볼게요. 제일 정직하게요.

```java
class Page {
    byte[] data;          // 트랜잭션이 스냅샷 뜨려고 속을 다 열어 둔다
    int version;
    long lastModified;
}

class Transaction {
    private byte[] savedData;        // 롤백하려고 페이지 속을 베껴 둔다
    private int savedVersion;
    private long savedModified;

    void begin(Page p) {
        savedData = p.data.clone();         // 남의 속을 직접 들여다보고
        savedVersion = p.version;
        savedModified = p.lastModified;
    }

    void rollback(Page p) {
        p.data = savedData;                 // 남의 속에 직접 써넣는다
        p.version = savedVersion;
        p.lastModified = savedModified;
    }
}
```

동작은 해요. begin에서 베껴 두고, rollback에서 도로 써넣죠. **여기까지 첫 커밋.** before예요.

## 되돌릴 게 자꾸 늘어납니다

저장 엔진이 자라면서 요구가 들어와요.

- 저장팀: "Page에 체크섬 필드를 넣어요. 무결성 검사용이요."
- 저장팀: "dirty 플래그도요. 변경됐는지 표시하게요."
- 복구팀: "세이브포인트도 넣어요. 트랜잭션 중간중간 되돌아갈 지점을 여러 개 잡게요."
- 그리고: "인덱스랑 카탈로그도 롤백돼야 해요. Page만이 아니에요."

시키는 대로 해 보세요. Page에 필드가 하나 늘 때마다 트랜잭션의 `begin`이랑 `rollback`을 둘 다 열어서 베끼는 줄을 더해야 해요. 빠뜨리면 그 필드만 안 되돌아가고요. 조용히요.

```java
class Transaction {
    private byte[] savedData;
    private int savedVersion;
    private long savedModified;
    private long savedChecksum;    // 필드 추가 -> begin에도 rollback에도 한 줄씩
    private boolean savedDirty;    // 또 추가 -> 또 양쪽에 한 줄씩
    // 세이브포인트가 여러 개면 이 묶음을 여러 벌. 인덱스/카탈로그까지 오면 통째로 또.
}
```

**여기까지 두 번째 커밋.** 일단 박아 두세요.

## 혹시 문제를 느꼈나요?

이쯤 되면 보이죠. 그래도 짚어 둘게요.

- 트랜잭션이 Page의 속을 전부 알아야 해요. 무슨 필드가 있는지, 어떻게 베끼고 되돌리는지까지요. Page 내부가 트랜잭션으로 다 새어 나간 거예요. 캡슐화가 깨졌죠.
- Page에 필드가 하나 늘 때마다 트랜잭션의 begin이랑 rollback을 둘 다 고쳐야 해요. 산탄총 수술이고, 한 줄 빠뜨리면 그 필드만 조용히 안 되돌아가요. 제일 나쁜 종류의 버그죠. 되돌렸다고 믿는데 안 됐으니까.
- 되돌릴 객체가 Page만이 아니라 인덱스, 카탈로그로 늘면, 트랜잭션이 그 모든 객체의 속까지 다 알아야 해요. 트랜잭션이 저장 층 전체의 내장을 떠안는 거예요.

## 방향만 던질게요

길이 떠오를 거예요. 방향만 대 줄게요.

1. 트랜잭션이 Page 속을 뒤지지 말고, Page한테 "네 지금 상태 좀 떠 줘"라고 부탁하면 어떨까요. Page가 자기 스냅샷을 직접 떠서 건네고, 트랜잭션은 그걸 속을 안 본 채 토큰처럼 들고만 있는 거예요. 되돌릴 땐 "이거 줄게, 네가 복원해" 하고 도로 건네고요. 스냅샷 안에 뭐가 들었는지는 Page만 알아요.
2. 그렇게 하면 Page에 필드가 늘어도 트랜잭션은 한 줄도 안 바뀌어요. 스냅샷 뜨고 복원하는 건 Page 안의 일이니까요. 트랜잭션은 토큰만 주고받죠.
3. 세이브포인트는 그 토큰을 여러 개 들고 있으면 끝이에요. 인덱스나 카탈로그도 각자 자기 스냅샷을 뜨게 하면, 트랜잭션은 누구 속도 모른 채 토큰 묶음만 쥐고요.

도착점은 또렷해요. **Page에 내부가 늘어도 트랜잭션은 한 줄도 안 바뀐다. 트랜잭션은 누구의 속도 모른 채 스냅샷 토큰만 들고 되돌린다. 세이브포인트는 토큰을 여러 개 쥐면 된다.** 거기까지 가면 after예요.

검증은 짧아요. 커밋 전에 자문해요. Page에 내부 필드를 하나 더 추가해요(예: 체크섬). 그걸 되돌리려고 트랜잭션의 begin이나 rollback을 여나요? 안 열고 Page만 고치면 끝이면 after예요. 트랜잭션이 스냅샷 속을 들여다보는 코드가 한 줄이라도 있으면 아직이고요.

## 회고: 비용은 메모리예요

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도요. 이번 비용은 메모리예요. 스냅샷을 들고 있어야 하고, 세이브포인트가 많으면 그만큼 쌓여요. 큰 Page를 통째로 복사하면 그 복사 비용도 만만치 않고요.

그래서 진짜 DB는 보통 상태를 통째로 뜨지 않고 변경분(undo 로그)만 남겨요. 같은 발상인데 "무엇을 스냅샷에 담느냐"를 가볍게 바꾼 거죠. 본인 건 통째로였나요, 변경분만이었나요? 그 선택이 메모리와 단순함을 맞바꾼 지점이에요. 회고에 적어 두세요. 레벨 8에서 만든 ABORTED, 이제 그게 진짜로 뭘 하는지 한 줄로 채워 넣을 수 있을 거예요.
