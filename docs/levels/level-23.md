# 레벨 23: 트리는 그대로 두고, 트리한테 시킬 일만 밖으로 빼기

또 봐요. 스물세 번째예요. 동네 안 옮겨요. `mini-db/frontend` 모듈(`lab.db.frontend`) 그대로고, 그 조건 트리도 그대로예요. 세 번째네요. 레벨 21에서 짓고, 레벨 22에서 행을 먹였던 그 트리요. 버리지 말랬죠. 오늘 또 써요.

그리고 레벨 22 끝에서 흘린 말, 기억나요? "새 일이 또 오면 노드마다 메서드가 셋, 넷으로 늘 거예요"라고 했죠. 왔어요. 늘 와요. 오늘이 딱 그날이에요.

## 지난 단계 돌아보기

레벨 21에서 우리는 노드마다 `render()`를 넣어 트리를 문자열로 그렸어요. 레벨 22에선 같은 노드마다 `evaluate(row)`를 넣어 행을 평가했고요. 그래서 지금 노드 하나가 메서드를 둘 들고 있어요. `render`랑 `evaluate`. 트리를 균일하게 다루기엔 그게 좋았죠. 잎이든 가지든 같은 약속만 부르면 됐으니까요.

그런데 그 "노드 안에 일을 넣는다"가, 오늘 정확히 발목을 잡아요.

## 이번에 마주칠 상황

조건 트리로 할 "새 일"이 계속 들어와요. 이번엔 노드 종류가 느는 게 아니라, 트리한테 시킬 연산이 늘어요.

- 분석팀: "조건 타입을 실행 전에 검사해 주세요. `age > 'kim'`처럼 숫자 컬럼에 문자열 비교한 거, 돌리기 전에 걸러야 해요." 그래서 `typeCheck`가 필요해져요.
- 플래너팀: "조건 트리를 실행 계획으로 바꿔 주세요. 인덱스 탈 수 있는 조건은 인덱스 스캔으로 내려 주고요." 그래서 `toPlan`이 필요하고요.
- 그리고 곧 또 와요. 선택도 추정(`estimateSelectivity`), 보기 좋게 들여쓰기(`prettyPrint`)... 트리로 할 일은 끝이 없어요.

이번 목표는 이거예요. **조건 트리에 새 연산(타입 검사, 실행 계획 생성)을 더하는데, 노드 클래스를 단 하나도 안 열고 끝낸다.**

## 가장 정직하게 먼저

습관대로 가요. 레벨 21, 22에서 했던 그대로요. 노드마다 메서드 하나 더 다는 거죠. 타입 검사가 필요하다니까 `typeCheck()`를 모든 노드에 넣어요.

```java
interface Condition {
    String render();              // 레벨 21
    boolean evaluate(Row row);    // 레벨 22
    Type typeCheck();             // 오늘 추가
}

class Comparison implements Condition {
    public String render()            { ... }
    public boolean evaluate(Row r)    { ... }
    public Type typeCheck()           { ... }   // 노드마다 하나씩
}
class BoolGroup implements Condition { /* 같은 세 메서드 */ }
class NotNode    implements Condition { /* 같은 세 메서드 */ }
```

동작은 해요. 트리 전체가 타입 검사를 받죠. **여기까지 첫 커밋.** before예요. 연산 하나 늘리는 게 노드 클래스 몇 개 여는 거지만, 아직은 견딜 만하죠.

## 연산이 더 늘어요

플래너팀이 `toPlan`을 들고 오고, 분석팀이 `estimateSelectivity`를, 리포트팀이 `prettyPrint`를 들고 와요. 하나 올 때마다 노드 클래스를 전부 엽니다. `Comparison` 열고, `BoolGroup` 열고, `NotNode` 열고, `BetweenNode` 열고. 새 연산 하나가 노드 클래스 전부를 건드려요.

```java
class Comparison implements Condition {
    render(); evaluate(r); typeCheck(); toPlan(); estimateSelectivity(); prettyPrint();  // 연산 올 때마다 한 줄씩
}
// BoolGroup, NotNode, BetweenNode... 전부 똑같이 부풀어요
```

**여기까지 두 번째 커밋.** 고통이 드러난 상태예요.

## 혹시 문제를 느꼈나요?

이쯤이면 보일 거예요. 그래도 짚어요.

- 산탄총, 또요. 근데 이번엔 쏘는 방향이 반대예요. 레벨 21에선 "새 노드"가 모든 메서드 분기를 건드렸죠. 오늘은 "새 연산"이 모든 노드 클래스를 건드려요. 같은 산탄총이 축만 바꾼 거예요.
- 노드 클래스 하나에 상관없는 일들이 다 모여 살아요. `Comparison` 안에 그리기, 평가, 타입 검사, 계획 생성이 한 데 섞여 있죠. 타입 검사 로직만 고치려고 여는데 그 옆에 렌더링 코드가 같이 있고요.
- 연산 하나를 통째로 보기가 어려워요. "타입 검사가 트리 전체에서 어떻게 도는지"를 읽으려면 노드 클래스 네 개를 다 열어서, 흩어진 `typeCheck`를 모아 읽어야 해요. 한 연산이 네 군데에 쪼개져 있으니까요.

여기서 레벨 22랑 헷갈릴 거예요. "이거 또 어제 거 아냐?" 트리도 같고, 노드에 메서드 넣는 손동작도 같았죠. 그런데 노리는 방향이 정반대예요. 레벨 22(인터프리터)는 **연산 하나(평가)를 고정해 두고 노드 종류를 늘리기 쉽게** 짠 거였어요. 새 조건 종류는 클래스 하나로 끝났죠. 대신 새 연산은 모든 노드를 수술해야 했고요. 오늘은 그 반대를 노려요. **연산을 늘리기 쉽게.** 새 연산이 노드를 안 건드리게요. 그 대신 뭘 내놓아야 하는지는, 회고에서 정직하게 셈할 거예요.

## 방향만 던질게요

길이 떠오를 거예요. 방향만 대 줄게요.

1. 연산을 노드 안에 두지 마세요. 연산 하나를 통째로 객체 하나로 떼어내요. "타입 검사기" 객체, "계획 생성기" 객체. 그 객체가 노드 종류별로 뭘 할지를 전부 알게요. 노드 종류마다 처리 메서드 하나씩, 그 객체 안에 모아서요.
2. 그러면 노드엔 메서드가 딱 하나만 남아요. "연산 객체야, 나한테 맞는 걸 해 줘" 하고 자기를 그 객체한테 넘기는 메서드요. 노드는 자기가 무슨 연산을 당하는지 몰라요. 그냥 자길 건네고, 받은 객체가 "아 너 `Comparison`이구나" 하며 맞는 처리를 골라 부르죠. 노드는 "나 이런 종류야"만 알리고, 처리는 연산 객체가 골라요. 부르기가 두 번 튕기는 셈이에요.
3. 가지(`BoolGroup`)는 그 하나의 메서드 안에서 자식들도 같은 연산 객체에 넘기면 돼요. 내려가기는 여전히 트리 안에 있고요. 레벨 21에서 `render`가 자식들의 `render`를 부르던 그 자리에, 이번엔 자식들을 연산 객체한테 넘기는 거예요.
4. 그러면 새 연산은? 연산 객체 하나 새로 만들면 끝이에요. 노드 클래스요? 한 줄도 안 열어요. 연산이 노드 바깥에 사니까, 새 연산을 더해도 노드는 자기가 더해진 줄도 몰라요.

도착점은 또렷해요. **새 연산(예: 비용 추정)을 더하는 게 객체 하나 추가로 끝나고, 노드 클래스는 하나도 안 열린다. 노드엔 "연산을 받아들이는" 메서드 하나만 남는다.** 거기까지 가면 after예요.

검증은 짧아요. 커밋 전에 자문해요. 새 연산 하나 붙이는데 `Comparison`이나 `BoolGroup`, `NotNode` 중 하나라도 여나요? 한 군데도 안 열고 연산 객체 하나로 끝나면 after예요.

## 회고: 비대칭을 뒤집은 값, 그리고 그 대가 (꼭)

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 이번엔 비용 칸이 진짜 핵심이에요.

뒤집힌 걸 적으세요. 레벨 22까진 "새 노드는 쉽고, 새 연산은 어렵다"였죠. 오늘 그게 "새 연산은 쉽고, 새 노드는 어렵다"로 뒤집혔어요. 이제 새 노드 타입(예: `IN` 조건)을 더하면? 모든 연산 객체를 다 열어서 "`IN`은 이렇게 처리한다" 메서드를 하나씩 더 달아야 해요. 산탄총이 사라진 게 아니에요. 노드 축에서 연산 축으로 갔다가, 오늘 다시 노드 축으로 돌아온 거죠.

이게 그 유명한 딜레마예요. 노드도 연산도 둘 다 쉽게 늘릴 순 없어요. 한쪽을 쉽게 만들면 다른 쪽이 어려워져요. 그러니 질문은 하나예요. 본인 WHERE 트리에선 뭐가 더 자주 늘어요? 노드 종류(새 조건)? 아니면 연산(트리로 할 새 일)? 더 자주 느는 쪽을 쉽게 만드는 게 정답이고, 그게 패턴 선택의 거의 전부예요. 회고 빈칸: "내 트리는 ___(노드 종류 / 연산)가 더 자주 는다. 그래서 ___(레벨 22 방식 / 레벨 23 방식)을 골랐다."

이건 농담 안 하고 말할게요. 패턴 책은 레벨 22랑 23을 멀찍이 떨어진 다른 장에 둬요. 근데 둘은 사실 한 트리를 앞에 두고 "뭘 포기할래"를 묻는 같은 질문의 두 답이에요. 그걸 같은 트리에 손으로 둘 다 올려 본 사람만 알아요. 책 두 장이 절대 못 주는 감각이고요. 이번 frontend 세 단계가 노린 게 통째로 그거였어요.
