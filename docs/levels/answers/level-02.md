# 레벨 2 답 맞추기: 정답과 이정표

스포일러 주의. **after 커밋과 회고를 끝낸 뒤에만 읽으세요.** 아직이라면 [레벨 2로 돌아가세요](../level-02.md).

## 당신이 다시 발견한 것

템플릿 메서드 패턴(Template Method)입니다.

변하지 않는 뼈대(로그, 검증, 빈칸, 정상 표시, 로그, 그리고 그 순서)를 상위에 고정해 두고, 컴포넌트마다 다른 가운데 한 칸만 하위가 채우게 했죠. 뼈대는 상위가 쥐고 빈칸만 하위에 위임하는 것, 그게 템플릿 메서드예요.

## 카탈로그와 대조하기

refactoring.guru: [템플릿 메서드 패턴](https://refactoring.guru/ko/design-patterns/template-method)

대조하며 확인할 것:

- 상위의 `start()`가 **템플릿 메서드**(순서를 박아 둔 뼈대), 하위가 구현한 가운데 한 칸이 **훅(추상 연산)**이에요.
- 레벨 1과 비교: 레벨 1은 **합성**으로 풀었어요(객체를 바깥에서 끼움). 이번엔 **상속**으로 풀었고요(상위가 순서를 쥐고 하위는 빈칸만 구현). "누가 순서를 쥐고 있었나?"라는 회고 질문의 답이 바로 이 차이예요.
- 카탈로그가 말하는 "할리우드 원칙"(Don't call us, we'll call you)이 우리 구조 어디에 나타났는지 짚어 보세요. 상위가 하위를 부르지, 하위가 상위를 부르지 않죠.

회고에 한 줄: "합성으로 풀 수도 있었는데 상속이 더 자연스러웠던 이유는 ___."
