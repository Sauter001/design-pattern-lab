# 레벨 12 답 맞추기: 정답과 이정표

스포일러 주의. **after 커밋이랑 회고까지 끝낸 다음에만 읽어요.** 아직이면 [레벨 12로 돌아가요](../level-12.md). 이번 것도 혼자 가는 단계라, 혼자 대조하면 돼요.

## 다시 발견한 것

브리지 패턴(Bridge)이에요.

서로 따로 자라던 두 갈래, "어떤 종류의 store인가"(추상화 축)와 "물리적으로 어떻게 저장하나"(구현 축)를 떼어 놨죠. store 종류 쪽이 백엔드를 하나 품고, 저장은 백엔드한테 넘기기만 하고요. 그래서 둘이 서로를 안 건드리고 따로따로 늘어나게 된 것. 그게 브리지예요. 두 축 사이에 다리를 놓고, 각자 자기 길로 자라게 한 거죠.

## 카탈로그와 대조하기

refactoring.guru: [브리지](https://refactoring.guru/ko/design-patterns/bridge)

맞춰 볼 것:

- **Abstraction** = `Store`(백엔드를 품는 쪽), **RefinedAbstraction** = 버전/캐시 store, **Implementor** = 백엔드 약속, **ConcreteImplementor** = 파일/메모리/원격 백엔드.
- 핵심은 둘이 **독립적으로** 확장되는지예요. 새 백엔드 하나에 클래스 하나, 새 store 종류 하나에 클래스 하나. 곱(M x N)이 합(M + N)으로 바뀌었으면 제대로 놓은 거예요.
- 이거, JDBC `Driver` 구조랑 똑같아요. JDBC API(추상화)와 각 DB 드라이버(구현)가 따로 자라죠. 같은 코드로 DB를 갈아끼우는 그 구조가 브리지예요. 실제로 매일 쓰던 거고요.

## 닮은 것들과 구분 (헷갈리기 쉬움)

- **어댑터(레벨 4)와:** 어댑터는 이미 안 맞는 남의 모양을 *사후에* 맞춰요. 브리지는 처음부터 두 축이 따로 자랄 걸 알고 *미리* 갈라놔요. 사후 수습이냐, 사전 설계냐.
- **전략(레벨 7)과:** 전략은 한 축(알고리즘 하나)을 갈아끼워요. 브리지는 두 축이 동시에 독립으로 자라고요. 백엔드를 품는 모양이 합성이라 전략이랑 닮아 보이지만, "축이 둘이고 둘 다 자란다"가 브리지의 표식이에요.

## 비용과 안 쓸 자리 (꼭)

간접성이 한 단계 늘어요. store가 직접 저장 안 하고 백엔드를 거치죠. 그리고 이건 **두 축이 진짜로 둘 다 변할 때만** 값을 해요. 백엔드가 영영 파일 하나뿐이면 이 분리는 과해요. 미리 갈라놓느라 클래스만 늘린 셈이고요. 회고에 "내 경우엔 두 축이 정말 둘 다 늘었나"를 적었으면, 그게 이 패턴을 *언제* 쓸지 아는 눈이에요.

회고 빈칸: "내 경우 백엔드는 ___개, store 종류는 ___개로 늘었고, 그래서 브리지가 ___했다."

여기까지 왔으면 구조 패턴도 슬슬 손에 잡힐 거예요. 자, 다음.
