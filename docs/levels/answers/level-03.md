# 레벨 3 답 맞추기: 정답과 이정표

스포일러 주의. **after 커밋과 회고를 끝낸 뒤에만 읽으세요.** 아직이라면 [레벨 3로 돌아가세요](../level-03.md).

## 당신이 다시 발견한 것

옵저버 패턴(Observer)입니다.

한 사건(가입)을 알리는 쪽과 거기에 반응하는 쪽을 갈라놓고, 반응하는 쪽이 스스로 등록하게 했죠. 알리는 쪽은 누가 듣는지 모른 채 "일이 일어났다"고 통지만 하고요. 일대다 발행/구독, 그게 옵저버예요.

## 카탈로그와 대조하기

refactoring.guru: [옵저버 패턴](https://refactoring.guru/ko/design-patterns/observer)

대조하며 확인할 것:

- **발행자(Subject/Publisher)** = 가입을 알리는 핸들러, **구독자(Observer/Subscriber)** = 메일/쿠폰/통계 같은 반응들, **구독(subscribe)** = 손 들어 등록하기.
- 실제 스프링의 `ApplicationEvent`는 한 발 더 나가요. 발행자가 구독자 목록을 직접 들지 않고, 가운데 **이벤트 발행기**(`ApplicationEventPublisher`)와 **이벤트 객체**를 두어 더 느슨하게 떼어 놓죠. 당신 버전은 발행자가 목록을 직접 들었나요, 아니면 가운데 중개자를 뒀나요? 그 차이가 결합도를 가릅니다.
- 통지 순서나 예외 전파(반응 하나가 터지면 나머지는?)를 당신이 어떻게 다뤘는지 보세요. 카탈로그는 거기까진 정해 주지 않아요. 구현자의 몫이에요.

회고에 한 줄: "발행자와 구독자를 떼어 놓은 대가로 잃은 것은 ___ (예: 흐름이 한눈에 안 보임)."
