# 레벨 16 답 맞추기: 정답과 이정표

스포일러 주의. **after 커밋이랑 회고까지 끝낸 다음에만 읽어요.** 아직이면 [레벨 16으로 돌아가요](../level-16.md). 이번 건 혼자 가는 단계라, 혼자 대조하면 돼요.

## 다시 발견한 것

퍼사드 패턴(Facade)이에요. 그리고 본인이 세운 그 정문, 웹 프레임워크에선 흔히 **Front Controller**라고 부르는 그거예요. 바로 스프링의 `DispatcherServlet`이고요.

매핑/어댑터/렌더/에러로 이어지는 복잡한 안쪽을 정문 하나 뒤로 숨기고, 손님에게는 "요청 처리해 줘" 한 줄짜리 입구만 내줬죠. 복잡한 서브시스템 앞에 단순한 창구 하나를 세워, 쓰는 쪽이 안쪽을 몰라도 되게 한
것. 그게 퍼사드예요.

## 이게 이 lab의 작은 분수령

오늘 만든 건 레벨 1부터 조각조각 빚어 온 것들의 종착점이에요. 경로 매핑(레벨 1), 핸들러 어댑터(레벨 4), 요청 감싸개(레벨 5), 문지기(레벨 6). 이 부품들이 전부 그 정문 하나 뒤로 들어갔어요. 실제
스프링 MVC가 정확히 이 구조예요. `DispatcherServlet`이 Front Controller(퍼사드)로 서서, 내부의 HandlerMapping, HandlerAdapter, ViewResolver를 다
가립니다. 본인이 그걸 손으로 다시 만든 거고요.

## 카탈로그와 대조하기

refactoring.guru: [퍼사드](https://refactoring.guru/ko/design-patterns/facade)

맞춰 볼 것:

- **Facade** = 정문(`DispatcherServlet` 같은), **Subsystem** = 매핑/어댑터/렌더/에러 부품들. 정문은 새 일을 하지 않고 부품 호출 순서만 알고 위임해요. 본인 정문 안에
  진짜 로직(매핑 알고리즘 같은)이 들어가 있으면, 그건 퍼사드가 일까지 떠안은 거예요. 순서만 엮고 일은 부품에 맡겼는지 보세요.
- 퍼사드는 안쪽을 **막지** 않아요. 그냥 쉬운 입구를 하나 더 줄 뿐이라, 필요하면 손님이 부품을 직접 쓸 수도 있어요. 이게 엄격한 계층(layer)과 다른 점이에요.

## 닮은 것들과 구분 (헷갈리기 쉬움)

- **어댑터(레벨 4)와:** 어댑터는 한 객체의 모양(인터페이스)을 다른 모양으로 바꿔요. 퍼사드는 여러 부품의 묶음 앞에 새 입구를 내요. 하나를 변환하느냐, 여럿을 가리느냐.
- **데코레이터(레벨 5)와:** 데코레이터는 같은 인터페이스를 유지하며 일을 한 겹 더해요. 퍼사드는 일을 더하지 않고 복잡함을 가리기만 해요. 게다가 퍼사드가 내주는 입구는 보통 안쪽보다 더 단순한 새
  인터페이스고요.
- **중재자(레벨 15)와:** 중재자는 동료들이 서로 맞물리는 규칙을 모아요(양방향 조율). 퍼사드는 손님 -> 서브시스템 단방향으로, 그냥 부르기 쉽게 해 줄 뿐이에요.

## 비용

정문 뒤로 다 숨기면 입구는 깔끔한데, 안쪽 흐름이 한 꺼풀 가려져요. 디버깅할 땐 정문을 열고 들어가야 보이죠. 그리고 정문이 안쪽 부품 전부를 알다 보니, 부품이 늘수록 정문도 무거워져요. 방치하면 모든 걸 아는
거대한 덩어리(god object)가 될 위험이 있고요. 그래서 정문은 "순서를 엮어 위임"하는 선에서 멈춰야 해요. 일을 직접 하기 시작하면 그때부터 비대해져요.

회고 빈칸: "정문 하나로 얻은 것은 ___(진입점마다 엮던 복제가 사라짐), 가려진 것은 ___(안쪽 흐름이 한 꺼풀 숨음)."
