# 레벨 11 답 맞추기: 정답과 이정표

스포일러 주의. **after 커밋이랑 회고까지 끝낸 다음에만 읽어요.** 아직이면 [레벨 11로 돌아가요](../level-11.md). 이번 건 쌍이 없어서 혼자 대조하면 돼요.

## 다시 발견한 것

빌더 패턴(Builder)이에요. 흔히 "쿼리 빌더"라고 부르는 그거죠.

쿼리를 생성자 한 방에 다 받지 않고, 필요한 절만 이름 붙여 하나씩 얹어 짓게 했죠. 안 쓰는 절은 안 부르면 그만이고, 다 얹은 뒤 마지막에 한 번 "지어 줘"로 완성하면서 거기서 잘못된 조합도 걸렀고요. 복잡한 객체의 조립 과정을, 객체 자신에서 떼어내 단계별로 만든 것. 그게 빌더예요.

## 카탈로그와 대조하기

refactoring.guru: [빌더](https://refactoring.guru/ko/design-patterns/builder)

맞춰 볼 것:

- **Builder** = 절을 하나씩 받아 모으는 조립기, **build()** = 모은 걸로 완성품을 내놓고 검증하는 마지막 단계, **Product** = `Query`.
- GoF 원형엔 **Director**(조립 순서를 정해 두는 역)도 있어요. 본인 코드엔 그게 있었나요, 아니면 호출부가 직접 순서대로 얹는 플루언트 방식이었나요? 요즘은 Director 없이 메서드 체이닝하는 플루언트 빌더가 흔해요. 둘 다 빌더의 정신은 같고, Director를 둘지가 트레이드오프예요.
- 잘못된 조합(OFFSET만, LIMIT 없이)을 `build()`에서 막고 있는지 보세요. 거기가 생성자 시절보다 분명해진 자리예요.

## 비용과 안 쓸 자리 (꼭)

빌더는 공짜가 아니에요. 조립기 클래스가 하나 더 생기고, 한 줄이면 되던 생성이 여러 줄이 돼요. **조각이 두세 개뿐인 객체엔 과해요.** 그냥 생성자나 정적 팩토리가 나아요. 빌더가 값을 하는 자리는 "선택 조각이 많고, 대부분 안 쓰고, 불변으로 만들고 싶을 때"예요. 회고에 "내 쿼리는 절이 몇 개였고 그래서 값을 했나"를 적었으면, 그게 이 패턴을 *언제* 쓸지 아는 눈이에요. 쓰는 법보다 그게 더 값나가요.

회고 빈칸: "내 객체는 선택 절이 ___개였고, 그래서 빌더가 ___했다(값을 했다 / 과했다)."

여기까지 혼자 와서 또 하나 발견했네요. 자, 다음.
