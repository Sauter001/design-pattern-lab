# 레벨 20: 지금 실행하지 말고, 실행할 일을 객체로 들고 있기

또 봐요. 스무 번째예요. 동네 옮겨요. 레벨 19까지 미니 DB 엔진에 있었는데, 미니 스프링 mvc로 잠깐 돌아옵니다. 레벨 1, 4, 5, 6, 16에서 요청 처리를 다지던 그곳, `mini-spring/mvc` 모듈(`lab.spring.mvc`)이에요.

미리 말해 두면, 미니 DB 맨 앞(쿼리를 읽어 들이는 파싱 층) 그 마지막 보상 구간이 바로 다음부터예요. 그 전에 mvc에 묵혀 둔 게 딱 하나 남아서, 그것만 털고 넘어가요. 오래 안 걸려요.

## 지난 단계 돌아보기

레벨 16에서 우리는 요청 하나가 들어와 응답이 나가기까지를 정문 뒤로 모았어요. 경로로 핸들러를 찾고, 어댑터로 부르고, 렌더링하고요. 그때 요청은 들어오는 즉시 처리돼서 바로 나갔죠. "받았으니 지금 실행한다"가 너무 당연해서, 그게 따로 정할 수 있는 선택인 줄도 몰랐고요.

오늘 깨질 게 그 "지금"이에요. 받은 시점과 실행하는 시점이 더는 같지 않아도 되는 요구들이 옵니다.

## 이번에 마주칠 상황

mvc가 요청을 받으면 알맞은 핸들러를 즉시 부르죠. 받자마자 실행, 끝. 그런데 서비스를 굴리다 보면 "지금 당장 실행" 말고 다른 게 필요해져요. 큐에 쌓았다 워커가 꺼내 실행하거나, 무슨 요청이 실행됐는지 기록하거나, 실패하면 다시 하거나요.

이번 목표는 이거예요. **요청을 받자마자 실행해 버리지 않고, "이 요청을 실행하라"는 일거리 자체를 손에 들 수 있게 한다. 그래야 미루고, 기록하고, 다시 할 수 있다.**

## 가장 정직하게 먼저

받으면 바로 핸들러를 부른다. 제일 정직하게요.

```java
void dispatch(Request req) {
    Handler h = mapping.find(req.path());
    Object result = h.handle(req);   // 받자마자 즉시 실행
    respond(result);
}
```

동작은 해요. 요청 오면 처리해서 응답하죠. **여기까지 첫 커밋.** before예요.

## "지금 말고"가 들어오기 시작해요

플랫폼팀이랑 운영팀이 하나씩 들고 와요.

- "피크 때 요청을 바로 실행하지 말고 큐에 쌓았다가, 워커가 순서대로 꺼내 실행하게 해주세요. 부하를 고르게요."
- "어떤 요청이 언제 실행됐는지 감사 로그를 남겨주세요. 실행 단위로요."
- "외부 호출이 실패하면 그 요청만 몇 번 재시도해주세요."
- "관리자 요청은 바로 실행 말고 승인 대기 큐에 넣었다가, 승인되면 실행하고요."

시키는 대로 해 보세요. 큐에 쌓으려면 "무엇을, 어떤 인자로 실행할지"를 한 덩어리로 들고 있어야 하는데, 지금은 `h.handle(req)`로 즉시 실행돼 버려서 들고 있을 게 없어요. 그래서 핸들러랑 요청을 임시로 묶어 큐에 넣고, 꺼낼 때 다시 부르는 코드를 짜기 시작하죠.

```java
// 큐에 쌓으려고 임시변통
record Pending(Handler h, Request req) {}
queue.add(new Pending(h, req));

// 워커: 꺼내서 다시 부른다
Pending p = queue.poll();
Object result = p.h().handle(p.req());   // 재시도? 감사 로그? 승인 대기? 여기저기 분기가 붙기 시작
```

그리고 재시도, 감사 로그, 승인 대기를 각각 이 언저리에 붙이다 보면, "실행을 다루는 코드"가 큐 처리에도, 재시도에도, 로깅에도 따로따로 흩어져요.

**여기까지 두 번째 커밋.** 일단 박아 두세요.

## 혹시 문제를 느꼈나요?

이쯤 왔으면 보일 거예요. 그래도 짚어요.

- "지금 실행"이 코드에 박혀 있어서, 미루거나 다시 하려면 핸들러와 인자를 임시 덩어리로 싸매 들고 다녀야 해요. 그 싸매는 코드가 큐, 재시도, 로깅마다 따로 생기고요.
- 실행을 미룬 곳마다 "무엇을 실행할지"를 다시 풀어 부르는 자리가 생겨요. 실행을 다루는 책임이 한곳에 안 모이고 흩어지죠.
- 받은 시점과 실행 시점이 갈라지는 모든 자리(큐잉, 재시도, 승인 대기)가 각자 실행 방법을 알아야 해요.

레벨 1이랑 7의 그 "갈아끼우기"랑 헷갈릴 수 있어요. 거기서도 할 일을 객체로 떼어 냈으니까요. 근데 결이 달라요. 그땐 "어떻게 할지(방법)"를 골라 **지금** 실행했어요. 오늘은 "실행할 일 한 건"을, 인자까지 통째로 객체에 담아 들고 있다가, **언제** 실행할지를 따로 정해요. 그 방법 객체는 보통 즉시 불리고 상태가 없지만, 오늘 담는 객체는 실행을 미뤘다 큐에서 꺼내고, 실패하면 다시 부르려고 자기 인자를 끝까지 품고 있어요. "어떻게"가 아니라 "무엇을, 언제"예요.

## 방향만 던질게요

길이 떠오를 거예요. 방향만 대 줄게요.

1. "이 요청을 실행하라"는 일거리 한 건을, 실행에 필요한 것(핸들러, 인자, 문맥)까지 통째로 담은 객체로 떠 보세요. 공통 약속은 딱 하나, `execute()` 정도면 돼요. dispatch는 그 객체를 만들기만 하고요.
2. 그렇게 떠 두면, 만든 객체를 즉시 `execute()` 해도 되고, 큐에 넣었다 워커가 꺼내 `execute()` 해도 되고, 승인 대기에 뒀다 나중에 해도 돼요. 큐도 워커도 승인기도 그 객체가 구체적으로 무슨 요청인지 몰라요. `execute()`만 부르면 되니까요.
3. 기록이나 재시도도 그 객체 단위로 깔끔해져요. "이 일거리를 실행했다"를 로깅하고, 실패하면 같은 객체를 다시 `execute()` 하면 되죠. 객체가 자기 인자를 품고 있어서 다시 부를 수 있고요.
4. 한 걸음 더 가도 돼요. 객체가 "실행"뿐 아니라 "되돌리기"까지 알면, 실행한 일을 취소하는 길도 열려요. 오늘 도메인에선 큐잉이랑 재시도가 먼저지만, 이 떠 두기가 되돌리기의 토대라는 것만 기억해 두세요.

도착점은 또렷해요. **요청 실행을 객체로 떠서, 즉시 실행 말고도 큐잉/기록/재시도/보류를 할 수 있다. 실행을 미루거나 다시 하는 자리들은 그 객체가 무슨 요청인지 모르고 execute()만 안다. 새 실행 동작(지연, 승인 대기)이 와도 실행 본체를 안 헤집는다.** 거기까지 가면 after예요.

검증은 짧아요. 커밋 전에 자문해요. 요청 하나를 받아서, 지금 실행하지 않고 큐에 넣었다가 나중에 꺼내 실행할 수 있나요? 그리고 실패한 그 요청을 똑같이 한 번 더 실행하는 데, 핸들러와 인자를 다시 수소문해야 하나요, 아니면 들고 있던 객체를 다시 부르면 되나요? 객체 하나 다시 부르면 끝이면 after예요.

## 회고: 즉시 실행이 정답일 때도 많아요

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도요. 요청 실행마다 객체가 하나씩 떠요. 요청 종류가 많으면 클래스가 늘거나, 람다로 가볍게 가거나 하죠. 그리고 "받아서 바로 실행"이던 한 줄이, 객체로 떴다가 어디선가 실행되는 두 단계가 돼서, 흐름을 좇을 때 "이게 대체 언제 실제로 실행되지?"를 한 번 더 따라가야 하고요.

이건 솔직히 말할게요. 큐잉도 재시도도 되돌리기도 영영 필요 없고 받는 족족 바로 실행만 하면, 오늘 이건 과해요. 그냥 직접 호출이 정직하죠. 떠 두기가 본전을 뽑는 건 실행을 미루거나(큐), 다시 하거나(재시도), 되돌리거나(취소), 기록해야(감사) 할 때예요. 본인 요청이 그중 몇 개를 정말 필요로 했는지 적어 두세요. 하나도 아니면 안 쓰는 게 맞고, 그 판단도 정답이에요.

콜백 하나. 레벨 19에서 옵티마이저 규칙을 객체로 떠서 줄에 꿰었죠. 오늘은 요청 실행을 객체로 떠서 큐에 쌓고요. 둘 다 "하던 일을 객체로 뜨기"인데, 떠서 뭘 하느냐가 달라요. 규칙은 줄줄이 잇고, 실행은 미뤘다 다시 하고. 같은 손동작, 다른 쓸모예요.
