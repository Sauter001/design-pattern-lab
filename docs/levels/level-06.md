# 레벨 6: 똑같이 감싸되, 이번엔 들여보낼지를 정한다

또 봐요. 여섯째 단계예요. 지난 단계 끝에서 제가 뭐랬죠. "다음에 이거랑 거의 똑같이 생긴 걸 또 만들 건데, 푸는 문제는 전혀 다르다"고요. 그거 하러 왔어요. 약속은 지키는 편이라.

## 지난 단계 돌아보기

레벨 5에서 우리는 핸들러를 품는 감싸개를 만들었죠. 바깥에선 똑같은 `Handler`인데, 안에서 로그/타이밍/압축 같은 일을 핸들러 둘레에 한 겹씩 덧입혔어요. 핵심은 두 가지였어요. 같은 인터페이스를 그대로 내보낸다. 그리고 **품은 진짜 핸들러를 무조건 불러 준 다음**, 그 앞뒤로 자기 일을 한다. 그래서 양파처럼 겹겹이 쌀 수 있었고요.

저 "무조건 불러 준다"를 기억해 두세요. 오늘 깨질 약속이거든요.

## 이번엔 문 앞이 문제예요

작업 공간은 그대로 `mini-spring/mvc`(`lab.spring.mvc`)고, 핸들러도 그대로예요. 달라지는 건 핸들러가 아니라 그 앞이에요.

어떤 핸들러는 아무나 부르면 안 돼요. 관리자 대시보드는 관리자만 들어가야죠. 그래서 이번 목표는 이겁니다. **핸들러가 진짜 일을 하기 전에, 그 요청을 들여보낼지 말지를 누군가 먼저 정한다.**

## 가장 정직하게 먼저

관리자 대시보드 핸들러를 봅시다. 본업은 대시보드를 그려 주는 거예요. 근데 아무나 보면 안 되니까, 본업 앞에 검사를 박아요.

```java
class AdminDashboardHandler implements Handler {

    @Override
    public String handle(String path) {
        if (!CurrentUser.get().hasRole("ADMIN")) {   // 곁다리: 권한 검사
            return "403 Forbidden";
        }
        return renderDashboard();                     // 진짜 할 일
    }
}
```

동작은 해요. 관리자 아니면 403, 맞으면 대시보드. **여기까지 첫 커밋.** before예요.

## 보안팀이 옵니다

이번엔 보안팀이에요. 하나씩 들고 와요.

- "관리자 화면만이 아니에요. 정산, 회원관리, 로그 조회 전부 ADMIN만 들어가게 해주세요."
- "외부에서 받아 온 리포트 핸들러도 막아야 하는데, 그쪽 코드는 우리가 못 고쳐요." (어디서 들어 본 얘기죠. 레벨 4요.)
- "권한 규칙이 자주 바뀌어요. 처음엔 ADMIN만, 다음엔 ADMIN이거나 본인, 점검시간엔 전부 차단. 핸들러마다 흩어져 있으면 못 따라가요."

시키는 대로 하면, 저 `if (!hasRole("ADMIN")) return "403";`을 보호 대상 핸들러마다 복붙하게 돼요. 정산 핸들러 위에도, 회원관리 위에도, 로그 조회 위에도요. 그리고 외부 리포트 핸들러? 거긴 코드를 못 여니까 검사를 넣을 자리가 아예 없어요.

```java
class SettlementHandler implements Handler {
    @Override
    public String handle(String path) {
        if (!CurrentUser.get().hasRole("ADMIN")) return "403 Forbidden";   // 또 복붙
        return renderSettlement();
    }
}
// 회원관리에도, 로그 조회에도... 같은 줄이 계속 따라다닌다
```

규칙이 한 번 바뀌면? 복붙해 둔 데를 전부 찾아다녀야죠. 또 하나는 빠뜨리고요. 늘 그래요. 근데 이번엔 빠뜨린 한 곳이 곧 뚫린 곳이라, 평소보다 덜 웃기죠.

**여기까지 두 번째 커밋.** 고통이 드러난 상태예요.

## 혹시 문제를 느꼈나요?

걸리는 게 있을 거예요. 없으면 제가 짚고요.

- 권한 검사가 핸들러 본업이랑 섞여서 보호 대상마다 복붙돼요. 본업은 한 줄인데, 검사가 그 앞을 막고 섰고요.
- 못 고치는 외부 핸들러는 보호할 방법이 없어요. 코드를 못 여니까.
- 규칙이 바뀌면 산탄총 수술. 그리고 빠뜨린 한 곳이 곧 보안 구멍이에요.

이쯤에서 레벨 5가 떠오를 거예요. 거기서도 핸들러 안 건드리고 둘레에 일을 더했으니, 검사도 그렇게 빼면 되겠네 싶고요. 맞아요. 거의 같아요. 근데 그 "거의"가 오늘의 전부예요.

길이 보일 거예요. 정답은 역시 말 안 해요. 방향만요.

1. 검사는 핸들러 본업이 아니에요. 핸들러는 그대로 두고, 그 앞에 "문지기"를 하나 세우면 어떨까요. 바깥에서 보면 똑같은 `Handler`인데, 안에서 통과 여부를 먼저 따져요. 통과면 품은 진짜 핸들러한테 넘기고, 아니면 안 넘겨요.
2. 방금 그 문장을 레벨 5랑 나란히 놓아 보세요. 레벨 5의 감싸개는 진짜를 **무조건** 불렀어요. 이 문지기는 진짜를 **부를지 말지를 정해요.** 안 부를 수도 있다는 거, 그 한 끗이 핵심이에요. 코드 모양은 쌍둥이처럼 닮았는데 하는 짓이 갈리는 지점이 정확히 거기예요. 한쪽은 기능을 더하려고 감싸고, 한쪽은 들여보낼지를 정하려고 감싸요.
3. 못 고치는 외부 리포트 핸들러요? 문지기는 그게 뭔지 몰라도 돼요. `Handler` 하나 품으면 그만이라, 레거시든 서드파티든 똑같이 앞에 세울 수 있어요. 레벨 4에서 모양 맞춰 둔 거, 여기서 또 덕 봐요.
4. 규칙이 자주 바뀐다고 했죠. 문지기 한 곳에만 규칙이 있으면, 바뀔 때 거기 한 곳만 고치면 돼요. 복붙한 데를 찾아다닐 일이 없어지고요.

그리고 하나 더. 들여보낼지 말지를 정하는 그 자리에선 다른 것도 정할 수 있어요. "아직 아무도 안 부르니, 무거운 진짜 핸들러를 아예 만들지도 말까"(첫 요청 전엔 존재하지도 않게), "방금 한 응답을 그대로 돌려줄까"(캐시). 전부 같은 자리예요. 바깥은 그냥 `Handler`를 부른다고 믿고, 그 앞에 선 누군가가 대신 판단해 주는 거죠. 굳이 지금 다 할 필욘 없고, 권한 검사 하나만 제대로 빼도 충분해요.

목표 지점은 분명해요. **핸들러도 디스패처도 안 건드리고 보호를 입힌다. 규칙이 바뀌면 문지기 한 곳만 고친다. 못 고치는 핸들러에도 똑같이 적용된다.** 거기까지 가면 after예요.

다 되면 세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적되, 이번엔 꼭 레벨 5 회고를 옆에 펴 놓고 적으세요. 이건 농담 안 하고 말할게요. 코드가 얼마나 닮았는지, 그런데 의도가 어디서 갈렸는지를 보세요. "한쪽은 ___ 하려고 감쌌고, 한쪽은 ___ 하려고 감쌌다." 이 빈칸 둘을 본인 말로 채우면, 두 단계를 같이 한 보람이 거기 다 있어요.
