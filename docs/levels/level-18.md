# 레벨 18: 똑같이 시작하는 새것을, 매번 따로 떠내기

또 봐요. 열여덟 번째예요. 동네는 안 옮겨요. 바로 전 레벨 17 그대로, `mini-spring/container` 모듈(`lab.spring.container`)이에요. 레벨 2, 9, 10, 15, 17을
다진 그곳요.

오늘은 레벨 17이랑 짝이에요. 바로 전에 우리는 "앱에 단 하나"를 푸는 법을 했죠. 오늘은 그 정반대예요. 어떤 빈은 하나면 안 되고, 쓸 때마다 새것이 필요하거든요. 같은 컨테이너인데 요구가 거꾸로예요.

## 지난 단계 돌아보기

레벨 17에서 우리는 설정 레지스트리 같은 걸 "앱에 하나만" 두게 했어요. 하나를 컨테이너가 보장하고, 쓰는 쪽은 그 하나를 주입받아 같이 썼죠. 모두가 같은 걸 보는 게 핵심이었고요.

오늘은 그 반대 요구가 와요. 모두가 같은 걸 보면 안 되는 물건이 있어요. 작업마다, 요청마다 자기만의 것을 들고 따로 놀아야 하는 거요. 그런데 그게 매번 빈 화면에서 새로 짓는 건 또 아니에요. 시작점은 다
똑같아야 하거든요. "똑같이 시작하되, 그다음부턴 따로." 이게 오늘 풀 모양이에요.

## 이번에 마주칠 상황

작업 컨텍스트라는 걸 하나 두려고 해요. 작업 하나가 도는 동안 그 작업이 들고 다니는 메모장 같은 거예요. 누가 요청했는지, 지금까지 뭘 거쳤는지, 예산이 얼마 남았는지. 작업이 시작될 때 기본값이 쫙 깔린 채로
손에 쥐어지고, 도는 동안 거기에 자기 흔적을 적어 나가요.

```java
class WorkContext {
    Map<String, Object> attributes = new HashMap<>();   // 작업이 적어 나가는 메모
    List<String> trail = new ArrayList<>();             // 거쳐 온 단계 기록
    Budget budget;                                      // 기본값 채우는 게 좀 무겁다
    // ...
}
```

기본값을 차리는 게 가벼운 일이 아니에요. 외부에서 기본 예산을 받아오고, 검증하고, 초기 attributes를 깔고요. 그래서 컨테이너가 잘 차려 둔 기본 컨텍스트 하나를 미리 만들어 둬요.

이번 목표는 이거예요. **작업마다 컨텍스트를 하나씩 쥐어 주되, 시작값은 다 똑같고, 그다음부터는 서로 완전히 따로 논다. 한 작업이 자기 메모를 고쳐도 옆 작업엔 안 샌다.**

## 가장 정직하게 먼저

레벨 17에서 막 배운 게 손에 있죠. "하나 잘 만들어 두고 다 같이 쓴다." 그 손으로 가 볼게요. 제일 정직하게요.

```java
// 컨테이너: 기본값 쫙 깔린 컨텍스트 하나를 미리 차려 둔다 (무거운 준비)
WorkContext template = buildDefault();

// 작업이 시작될 때마다: 그 하나를 받아 쓴다
WorkContext ctx = template;
ctx.attributes.put("user", currentUser);
ctx.trail.add("started");
```

동작은 해요. 첫 작업은요. **여기까지 첫 커밋.** before예요.

## 작업이 둘 이상 돌기 시작해요

서비스가 한가할 땐 안 보이던 게, 작업이 동시에 둘 이상 돌면 바로 터져요.

- 운영팀: "작업끼리 값이 섞여요. A 작업이 적은 메모가 B 작업 컨텍스트에 보여요. 둘이 격리가 안 돼요."

당연하죠. 같은 `template` 하나를 둘이 같이 쥐고 각자 적어 댔으니까요. 레벨 17에선 "다 같이 하나를 본다"가 정답이었는데, 여기선 그게 바로 병이에요. 그래서 손을 바꿔요. "그럼 작업마다 새로
만들자."

```java
WorkContext ctx = new WorkContext();
ctx.budget = loadBudget();           // 무거운 기본 준비를 또
ctx.attributes.putAll(DEFAULTS);     // 기본값을 또 깐다
ctx.attributes.put("user", currentUser);
```

이걸 작업 시작하는 자리마다 해요. 요청 핸들러에도, 배치 잡에도, 테스트 하니스에도요. 그리고 곧 이런 요구가 붙죠.

- 플랫폼팀: "컨텍스트에 `traceId` 필드 하나 추가해 주세요. 추적용이요."

`traceId` 하나 추가하면, 위처럼 `new` 해서 기본값 깔던 자리를 전부 찾아다니며 그 한 줄을 또 끼워야 해요. 한 군데라도 빠뜨리면 거기 작업만 추적이 안 되고요. 게다가 누군가는 이 복붙이 지겨워서
절충을 시도해요. 템플릿을 살짝 베끼는 거죠.

```java
WorkContext ctx = new WorkContext();
ctx.attributes = template.attributes;   // 그냥 참조만 복사. 빨라 보이죠?
```

**여기까지 두 번째 커밋.** 일단 박아 두세요. 마지막 저 줄이 특히요. 곧 다시 봐요.

## 혹시 문제를 느꼈나요?

이쯤이면 보이죠. 그래도 짚어 둘게요.

- 하나를 공유하면 격리가 깨져요. 작업끼리 같은 메모장에 적으니 값이 섞이고요.
- 매번 `new`로 처음부터 지으면, 기본값 차리는 코드가 만드는 자리마다 복붙돼요. 필드가 하나 늘면 그 자리를 전부 손봐야 하고요. 산탄총 수술이죠. 레벨 11에서 절 하나 늘 때마다 호출부를 다 열던 그
  느낌, 또 왔어요.
- 무거운 기본 준비(외부 fetch, 검증)를 작업마다 처음부터 다시 치러요.
- 그리고 저 마지막 줄. 얕게 베끼면 겉은 새 객체인데 속 맵은 `template` 거 그대로예요. 한 작업이 `attributes.put` 하면 그 변경이 템플릿을 거쳐 다른 작업에도 새요. 격리한 줄 알았는데
  안 됐죠. 이게 제일 고약해요. 안 터지는 척하다 동시성 붙으면 터지거든요.

레벨 11이랑 헷갈릴 수 있어요. 거기서도 객체 하나 짓는 얘길 했으니까요. 근데 그건 빈 화면에서 절을 골라 새로 조립하는 거였어요. 오늘은 달라요. **이미 잘 차려진 견본이 있고, 그걸 그대로 떠내는**
거예요. 짓는 게 아니라 베끼는 거죠. 단, 속까지.

## 방향만 던질게요

길이 떠오를 거예요. 방향만 대 줄게요.

1. 작업마다 처음부터 조립하지 마세요. 잘 차려 둔 견본 하나를 두고, 그 견본에게 "너랑 똑같은 걸 하나 떠내" 하고 시키면 어떨까요. 만드는 법(기본값 차리는 것까지)은 견본이 알아요. 쓰는 쪽은 "떠내 줘"
   한 번이면 끝이고요. 그러면 기본값 복붙도, 무거운 재조립도 사라져요.
2. 떠낸 건 견본과 완전히 따로 놀아야 해요. 그러니 겉만 베끼면 안 되고, 안에 든 가변 객체(맵, 리스트, 중첩 객체)까지 같이 새로 떠야 해요. 거기까지 해야 한 작업이 자기 메모를 고쳐도 옆 작업에 안
   새요.
3. 떠내는 책임이 견본 그 객체 한곳에 있으니, 필드가 늘어도 만드는 자리들을 헤집을 일이 없어요. `traceId`가 늘면 떠내는 그 한 군데만 챙기면 되죠.
4. 균형추 하나. 만약 그 객체가 진짜 단순하고 필드 몇 개뿐이고 안에 가변 객체도 없으면, 그냥 `new`가 더 정직해요. 떠내기가 본전을 뽑는 건 기본 조립이 비싸거나, 안쪽이 복잡해 손으로 베끼다 빠뜨리기
   쉬울 때예요.

이건 농담 안 하고 말할게요. 겉만 베낀 건 베낀 게 아니에요. 속을 공유하는 순간 둘은 여전히 한 몸이고, "격리됐겠지" 하고 믿는 그 순간이 제일 위험해요. 오늘 진짜로 떼어 낼 건 겉이 아니라 속이에요.

도착점은 또렷해요. **작업마다 받는 컨텍스트는 서로 완전히 따로 논다. 한쪽의 안쪽 맵을 건드려도 다른 쪽은 멀쩡하다. 기본값은 견본 한 곳에만 있고 만드는 자리에 복붙되지 않으며, 필드가 늘어도 만드는 자리는 안
바뀐다.** 거기까지 가면 after예요.

검증은 짧아요. 커밋 전에 자문해요. 작업 A의 컨텍스트에서 안쪽 맵이나 리스트를 건드렸을 때, 작업 B의 것은 그대로인가요? 그리고 새 필드 하나 추가하는데, 컨텍스트 만드는 자리를 여나요, 안 여나요? 안쪽까지
따로 놀고 만드는 자리를 안 열면 after예요.

## 회고: before도 after도 "새것"인데

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도요. 떠내는(복제하는) 로직을 그 객체가 직접 떠안아요. 그래서 필드를 추가하면
떠내는 코드도 같이 손봐야 하고, 깜빡하면 새 필드만 조용히 안 베껴져요. 그리고 깊게 뜨는 건 안쪽까지 다 새로 만드는 거라 그만큼 비용이 들고요.

레벨 17이랑 나란히 놓고 한 줄 적어요. 둘 다 컨테이너가 빈을 내주는 건 같아요. 그런데 레벨 17은 "매번 같은 하나", 오늘은 "매번 다른 새것"이죠. 같은 컨테이너의 두 얼굴이에요. 본인 빈이 둘 중 어느
쪽이어야 했는지, 그 판단이 이 두 레벨을 가르는 전부예요. 짧게 적어 두세요.
