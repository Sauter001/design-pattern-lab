# 레벨 9: 무엇을 만들지는 정하되, 만드는 법은 넘긴다

또 봐요. 아홉 번째예요. 쌍 다루는 거, 이제 손에 익었죠? 레벨 5-6, 7-8에서 두 번 해봤으니까. 이번이 세 번째 쌍이에요. 봐줄 만하게 따라오고 있어요, 뭐.

오늘부터 둘은 "만든다"를 다뤄요. 지금까지 우리는 이미 만들어진 객체를 고르고(전략), 감싸고(데코), 모양 맞췄죠(어댑터). 정작 "그 객체를 누가 어떻게 만드나"는 건드린 적이 없어요. 거기를 팝니다. 오늘은 그 첫 짝이고, 다음 단계가 나머지 짝이고요.

동네를 다시 옮겨요. 미니 DB에서 놀다가, 이번엔 미니 스프링으로 돌아갑니다. 작업 공간은 `mini-spring/container` 모듈(`lab.spring.container`)이에요. 레벨 2의 그 동네예요. 빈 생명주기 만들었던 곳요.

## 거슬러 올라가 돌아보기

레벨 2 기억나요? 컨테이너가 빈을 다 엮은 뒤 깨우고 재웠죠. 그땐 빈이 "이미 만들어져 있다"는 전제였어요. 누군가 `new`를 이미 해 둔 거죠. 오늘은 그 "누군가"가 우리예요. 컨테이너가 빈을 직접 만들어야 하는 상황으로 내려갑니다.

그리고 미리 하나 흘려둘게요. **다음 단계에서 오늘이랑 그림이 거의 똑같은 걸 또 만드는데, 거기선 하나가 아니라 한 벌을 만들어요.** 무슨 말인지는 끝에서요.

## 이번에 마주칠 상황

컨테이너가 빈 정의(`BeanDefinition`)를 받아서 인스턴스를 하나 만들어 준다고 해 봐요. 대부분은 기본 생성자로 `new` 하면 끝이에요. 그런데 그렇게 안 되는 녀석들이 있어요.

- 커넥션 풀 빈은 설정을 읽어서 만들어야 해요.
- 캐시 매니저 빈은 크기 같은 옵션을 받아 만들고요.

이번 목표는 이거예요. **빈을 만들어야 한다는 건 컨테이너가 알되, 어떤 구체 클래스를 어떻게 만드는지는 컨테이너가 모르게 한다.**

## 가장 정직하게 먼저

컨테이너가 빈 종류를 직접 알아보고, 종류마다 다르게 만들어 주게 해 볼게요. 제일 정직하게요.

```java
class Container {

    Object instantiate(BeanDefinition def) {
        Class<?> type = def.type();
        if (type == DataSource.class) {
            return new HikariDataSource(def.props());          // 풀: 설정 읽어 빌드
        } else if (type == CacheManager.class) {
            return new CaffeineCache(def.intProp("size"));     // 캐시: 크기 받아 생성
        } else {
            return type.getDeclaredConstructor().newInstance();// 기본: 기본 생성자
        }
    }
}
```

동작은 해요. 정의를 보고 알맞게 만들어 주죠. **여기까지 첫 커밋.** before예요.

## 만드는 법이 자꾸 늘어납니다

서비스가 자라면서 빈 종류가 더 들어와요. 그런데 만드는 법이 제각각이에요.

- 플랫폼팀: "메시지 큐 클라이언트 빈도 넣어 주세요. 그건 빌더로 만들어야 해요. `MqClient.builder()...build()`요."
- 결제팀: "결제 게이트웨이 빈은 외부 SDK의 정적 팩토리로 만들어요. `PayGateway.create(config)`. 생성자가 private이라 `new`를 아예 못 해요."
- QA팀: "테스트용 가짜 빈도 끼울 거예요. 같은 타입인데 만드는 법만 다르게요."

시키는 대로 `instantiate`에 넣어 보면 `else if`가 또 자라요. 빈 만드는 법이 하나 늘 때마다 컨테이너 심장을 다시 열고요.

```java
} else if (type == MqClient.class) {
    return MqClient.builder()./* ... */.build();             // 빌더
} else if (type == PayGateway.class) {
    return PayGateway.create(def.props());                   // 정적 팩토리
}
// 새 빈 만드는 법이 올 때마다 여기 한 칸 더. 늘 그렇듯.
```

**여기까지 두 번째 커밋.** 일단 박아 두세요. 이번엔 직접 안 시켜요, 알죠? 대신 눈으로만 따라가 봐요. 빈 만드는 법이 하나 늘 때마다 저 `else if`가 한 칸씩 자라죠. 그게 정상이에요.

## 혹시 문제를 느꼈나요?

이번엔 어디가 걸리는지 거의 보일 거예요. 그래도 짚어는 둘게요.

- 빈 만드는 법이 하나 늘 때마다 컨테이너의 `instantiate`를 다시 열어 `else if`를 끼워야 해요. 건드리면 안 될 심장을 자꾸 건드리죠.
- 컨테이너가 세상 모든 빈을 "어떻게 만드는지"까지 다 떠안았어요. 정작 컨테이너 본업은 빈을 등록하고 의존성 꽂고 생명주기 돌리는 거예요. "이 빈은 빌더로, 저 빈은 정적 팩토리로"라는 남의 사정이 거기 끼어든 거죠.
- 빌더로 만드는 그 빈 하나만 따로 시험하고 싶어도, 컨테이너 전체를 세워야 그 가지에 닿아요.

또 `else if`네, 레벨 1이랑 7이랑 똑같네 싶죠. 맞아요, 걷어내는 손동작은 닮았어요. 근데 이번에 갈리는 건 "비교 방식"도 "조인 방법"도 아니에요. **무엇을, 어떻게 `new` 하느냐**예요. 변하는 게 만드는 일 그 자체라는 게 이번 색깔이에요.

## 방향만 던질게요

길이 떠오를 거예요. 답은 본인이 부딪혀 찾는 거고, 저는 방향만 대 줄게요.

1. 컨테이너가 빈을 직접 `new` 하지 말고, "이 빈 좀 만들어 줘"라고 시키기만 하면 어떨까요. 빈마다 자기를 만드는 법을 따로 들고 있게 하고(예: `Object create()` 하나짜리 약속), 컨테이너는 그게 빌더인지 정적 팩토리인지 기본 생성자인지 모른 채 `create()`만 부르는 거예요. 무엇을 만들지는 알되, 어떻게 만드는지는 넘기는 거죠.
2. 그렇게 빼 두면 새 빈 만드는 법이 와도 컨테이너는 한 줄도 안 바뀌어요. 그 빈 전용 "만드는 법" 하나를 새로 만들어 끼우면 끝이고요. 컨테이너는 그게 생긴 줄도 몰라요.
3. 이번엔 빈을 하나 만들었어요. 딱 하나요. 다음 단계에서 거의 같은 그림을 또 그리는데, 거기선 빈 하나가 아니라 **서로 맞물려야 하는 빈 한 벌**을 통째로 만들 거예요. 하나를 만드는 거랑, 한 벌을 어긋나지 않게 만드는 건 또 다른 문제거든요. 오늘 만든 모양을 잘 기억해 두세요.

도착점은 또렷해요. **새 빈 만드는 법을 추가해도 컨테이너는 한 줄도 안 바뀐다. 빈마다 자기 만드는 법을 직접 쥔다. 컨테이너는 그 빈을 어떻게 만드는지 영영 모른다.** 거기까지 가면 after예요.

## 검증: 컨테이너를 여나 안 여나

커밋 전에 통과해요. 레벨 1, 7에서 본 그 질문이 또 나와요. 그림이 닮았으니 질문도 닮죠.

> 만드는 법이 또 다른 빈이 와요(예: 환경변수를 보고 골라 만드는 빈). 그걸 넣으려고 **컨테이너의 `instantiate`를 열어요, 안 열어요?**

- 연다 -> 아직이에요. 만드는 법이 아직 컨테이너 안에 `else if`로 박힌 거예요.
- 안 연다(그 빈 전용 만드는 법 하나 새로 만들어 등록만 하면 끝난다) -> after예요. 컨테이너는 그 빈을 어떻게 만드는지 끝까지 몰라요.

## 회고: 비용, 그리고 다음 짝

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도 꼭이요. 빈 종류마다 "만드는 법" 클래스가 하나씩 생겨요. 특수한 빈 다섯이면 작은 창조자 다섯. 클래스 수가 늘고, "이 빈 누가 어떻게 만들지"를 좇으려면 한 단계 더 거쳐 가요. 그 거래가 남는 장사였는지는 본인이 적어 봐야 알아요.

그리고 이건 꼭. 오늘은 다음 단계랑 한 쌍이에요. 오늘 만든 건 빈 "하나"를 만드는 위임이었어요. 다음엔 "한 벌"이고요. 회고에 한 줄 비워 두세요. "하나를 만드는 것과, 한 벌을 일관되게 만드는 것은 무엇이 다른가?" 지금 답이 안 나와도 돼요. 다음 단계 끝내고 둘을 나란히 놓으면 그제야 보여요. 다음 단계 들어가자마자 둘을 붙여 볼게요.
