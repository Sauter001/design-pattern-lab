# 레벨 15: 컴포넌트들이 서로의 상태에 맞춰 움직이게 하기

열다섯 번째 단계예요. 꽤 왔네요, 우리. 워밍업과 트레이드오프 쌍들을 지나오며 패턴 보는 눈이 꽤 자랐을 거예요. 이번엔 지금까지 따로 놀던 조각들이 처음으로 "서로를 의식"하기 시작합니다. 그래서 시작 전에, 꽤 거슬러
올라가 두 단계를 같이 떠올려 둘게요. 이번 고비는 그 둘을 겹쳐 봐야 선명해지거든요.

## 거슬러 올라가 돌아보기

레벨 2를 떠올려 보세요. 그때 우리는 컨테이너 컴포넌트들에 생명주기를 줬어요. 커넥션 풀, 캐시 워머, 스케줄러가 각자 깨어나고(start) 잠드는(stop) 골격을 갖췄죠. 그때 이 셋은 서로를 전혀 몰랐어요.
컨테이너가 차례로 깨워 주기만 하면 됐으니까요.

그리고 레벨 3에서 우리는 발행-구독을 만들었어요. 가입이라는 사건 하나가 일어나면 핸들러는 "가입이 일어났다"고 알리기만 하고, 메일과 쿠폰과 통계는 각자 손을 들어 그 알림을 받아 갔죠. 보내는 쪽은 누가 듣는지
몰라도 됐고, 받는 쪽은 각자 알아서 반응했어요. 한쪽에서 일어난 일을, 그쪽이 알지도 못하는 여럿에게 흘려보내는 일대다 통지였고요.

발행-구독, 바로 앞 레벨 14에서도 또 썼죠. 저장 층 WAL에서요. 페이지가 바뀌면 바뀌었다고 알리기만 하고, 로그와 복제와 통계가 각자 받아 갔어요. 두 번이나 손에 익은 도구라 오늘도 그쪽으로 손이 갈 거예요. 그리고 레벨 14 끝에 제가 "WAL은 순서가 걸린다"고 흘렸던 거, 같이 떠올려 두세요. 오늘 그 순서와 조건이 본격적으로 발목을 잡거든요.

이 두 가지를 손에 쥔 채로 이번 문제를 보면, 처음엔 "어, 레벨 3의 그 발행-구독으로 풀면 되겠는데?" 싶을 거예요. 그 직감을 일단 붙들어 두세요. 어디까지 통하고 어디서 깨지는지 직접 부딪혀 보는 게 이번
단계의 핵심입니다.

## 이번에 마주칠 상황

작업 공간은 레벨 2의 그곳, `mini-spring/container` 모듈(`lab.spring.container`)이에요.

레벨 2에서 컴포넌트들은 서로 독립적이었어요. 그런데 서비스를 실제로 굴려 보면, 이 녀석들의 "런타임 상태"가 서로에게 영향을 주기 시작합니다. 하나가 아프면 다른 하나가 일을 멈춰야 하고, 하나가 바쁘면 다른
하나가 속도를 늦춰야 하는 식으로요.

운영팀이 며칠에 걸쳐 부탁을 하나씩 들고 옵니다. 차례로 보죠.

- "커넥션 풀이 비정상이 되면 스케줄러는 새 작업을 그만 받게 해주세요. 죽은 DB에 일을 계속 밀어 넣어 봐야 쌓이기만 하니까요. 풀이 회복되면 다시 받고요."
- "캐시 워밍이 도는 동안엔 헬스 체크가 아직 준비 안 됨으로 응답하게 해주세요. 데우는 중에 트래픽을 받으면 첫 손님이 느린 응답을 맞거든요. 다 데우면 준비됨으로 바꾸고요."
- "스케줄러 큐가 꽉 차면 캐시 리프레시 주기를 좀 늦춰서 DB 부하를 덜어 주세요. 단, 풀이 정상일 때만요. 풀이 이미 죽어 있으면 리프레시는 어차피 멈춰 있을 테니 건드릴 필요 없어요."
- "이런 규칙은 앞으로도 계속 늘 거예요. 컴포넌트도 더 붙을 거고요."

## 가장 정직하게 먼저

시키는 대로, 각 컴포넌트가 "자기가 영향 줘야 할 상대"를 직접 들고 호출하게 해 볼게요. 커넥션 풀부터요.

```java
class ConnectionPool {
    private boolean healthy = true;
    private Scheduler scheduler;       // 내가 죽으면 멈춰야 할 상대
    private CacheWarmer cacheWarmer;

    void markUnhealthy() {
        this.healthy = false;
        scheduler.pauseNewJobs();      // 너, 새 작업 그만 받아
    }

    void markHealthy() {
        this.healthy = true;
        scheduler.resume();            // 이제 다시 받아도 돼
    }

    boolean isHealthy() {
        return healthy;
    }
}
```

스케줄러도요.

```java
class Scheduler {
    private ConnectionPool pool;        // 큐가 차면 풀 상태를 물어봐야 한다
    private CacheWarmer cacheWarmer;

    void onQueueFull() {
        if (pool.isHealthy()) {         // "풀이 정상일 때만"이라는 규칙이 여기 박혀 있다
            cacheWarmer.slowRefresh();
        }
    }
}
```

캐시 워머도요.

```java
class CacheWarmer {
    private HealthEndpoint health;      // 데우는 동안 준비 안 됨 표시

    void startWarming() {
        health.setReady(false);
        preload();
        health.setReady(true);
    }

    void slowRefresh() { /* 리프레시 주기를 늘린다 */ }
}
```

동작은 합니다. **여기까지 첫 커밋.** before예요.

## 그런데 규칙이 한두 개 더 붙으면

운영팀의 네 번째 말("앞으로도 계속 는다")이 곧 현실이 됩니다. "풀이 죽으면 캐시 워밍도 멈춰라", "스케줄러가 멈추면 헬스 체크에 사유를 남겨라" 같은 게 줄줄이 와요. 들어오는 족족 지금 구조에 넣어
보세요.

그러면 이렇게 됩니다. 풀은 스케줄러와 캐시를 들고, 스케줄러는 풀과 캐시를 들고, 캐시는 헬스와 스케줄러를 들고, 너도나도 서로를 참조하기 시작해요. 그리고 "풀이 죽으면 무슨 일이 벌어지나?"를 알고 싶으면
`ConnectionPool` 하나만 봐선 안 됩니다. 캐시도 열어 보고 스케줄러도 열어 봐야 전체 그림이 잡혀요. 규칙이 코드 곳곳에 흩어져 살거든요.

**여기까지 두 번째 커밋.** 고통이 드러난 상태예요. 그리고 잠깐 멈추세요.

## "레벨 3의 그 발행-구독으로 풀면 되잖아?"

아까 붙들어 둔 직감을 꺼낼 차례예요. 서로 직접 부르는 게 문제니까, 레벨 3에서처럼 상태가 바뀌면 이벤트만 뿌리고 관심 있는 쪽이 받아 가게 하면 깔끔하지 않을까요? 한번 그려 보세요.

풀이 "나 비정상"이라고 뿌리면 스케줄러가 받아서 멈추고, 캐시가 받아서 워밍을 멈추고요. 두 번째 요청까지는 정말 잘 풀려요. 직접 참조도 사라지고요.

그런데 세 번째 요청에서 발이 걸립니다. "큐가 차면 캐시 주기를 늦추되, **풀이 정상일 때만**." 캐시 워머가 "큐 꽉 참" 이벤트를 받았어요. 그런데 늦출지 말지 결정하려면 풀의 상태를 알아야 합니다. 그래서
캐시는 결국 다시 풀을 쳐다봐야 해요. 없애려던 그 상호 참조가 슬그머니 돌아오죠. 게다가 "풀이 정상일 때만"이라는 판단은 캐시 안에, 또 다른 판단은 스케줄러 안에, 정책은 여전히 사방에 흩어진 채예요.

알림을 뿌리는 것까진 레벨 3이 맞았어요. 그런데 지금 필요한 건 단순한 알림이 아니라 **여러 컴포넌트의 상태를 동시에 보고 내리는 결정**이에요. 거기서 발행-구독은 한 발 모자랍니다.

## 혹시 길이 보이나요?

방향만 드립니다.

1. 컴포넌트가 N개인데 서로를 직접 알면, 이어야 할 선이 몇 개가 되나요? 하나 늘 때마다 선은 그보다 훨씬 빠르게 늡니다. 이 그물부터 끊을 수 없을까요.
2. "A가 이러면 B는 저렇게"라는 규칙들이 지금 어디에 사나요? 한 컴포넌트에 한 조각씩 박혀 있죠. 이 규칙들이 모여 살 **한 채의 집**이 있다면, "풀이 죽으면 무슨 일이 벌어지나"를 그 집 한 곳만 열어
   보면 알 수 있을 거예요.
3. 그 집에게 모든 컴포넌트를 알게 하고, 컴포넌트는 거꾸로 그 집 하나만 알게 한다면? 각 컴포넌트는 자기 상태가 바뀔 때 "나 이렇게 됐어"라고 그 집에만 알리고, 누구에게 무엇을 시킬지는 그 집이 정합니다.
   세 번째 요청의 교차 조건도, 그 집은 풀과 스케줄러를 다 보고 있으니 자연스럽게 풀려요.
4. 마지막으로 균형추 하나. 만약 규칙이 끝까지 두세 개뿐이고 더 안 늘었다면, 사실 맨 처음의 직접 호출이 더 깔끔했을 거예요. 가운데 집은 컴포넌트와 규칙이 불어날수록 본전을 뽑습니다. 그러니 회고에 "내
   경우엔 규칙이 몇 개였고, 그래서 이 선택이 값을 했는가"를 꼭 솔직히 적어 두세요.

목표는 한 줄이에요. **새 조율 규칙을 넣을 때 한 곳만 고치면 되고, 어떤 컴포넌트도 다른 컴포넌트를 직접 참조하지 않는다.** 거기까지 가면 after예요.

세 번째 커밋 찍고요. 회고는 [docs/README-template.md](../README-template.md) 틀로, 이번엔 발행-구독을 다뤘던 레벨 3, 14와 나란히 놓고 한 줄 남기세요. "알림을
뿌리는 것"과 "결정을 한곳에 모으는 것"이 어떻게 다른지요. 그 한 줄이 이번 단계가 왜 따로 존재하는지를 설명해 줄 거예요.

비용이랑 숫자도 같이 적어요. before에서 컴포넌트끼리 직접 참조하던 선이 몇 개였고, after에서 몇 개로 줄었어요? 그 선이 0개가 됐으면 그물을 끊은 거예요. 새 조율 규칙 하나 넣는데 가운데 집 한 곳만 고치면 되던가요? 숫자로 확인하고 넘어가세요. 이건 다음 단계 들머리에서 다시 펴 볼게요.
