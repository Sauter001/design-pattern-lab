# 레벨 16: 복잡한 안쪽을, 정문 하나로 가린다

또 봐요. 열여섯 번째예요. 미니 DB를 네 바퀴 돌았으니, 이제 미니 스프링으로 돌아와요. mvc로요. 작업 공간은 `mini-spring/mvc` 모듈(`lab.spring.mvc`)이에요.

오늘은 좀 특별해요. 새 부품을 만드는 게 아니라, 레벨 1부터 우리가 만들어 온 부품들을 한자리에 모으는 단계거든요. 경로로 핸들러를 찾는 디스패처(레벨 1), 모양 다른 핸들러를 맞춰 부르는 어댑터(레벨 4),
둘레에 일을 입히는 감싸개(레벨 5). 다 따로 있죠. 요청 하나를 처리하려면 이것들을 순서대로 엮어야 해요. 그 "엮는 일"이 오늘 주제예요.

## 이번에 마주칠 상황

요청 하나가 들어와서 응답이 나가기까지, 거쳐야 할 안쪽 부품이 여럿이에요. 경로로 핸들러를 찾고, 그 핸들러에 맞는 어댑터를 골라 부르고, 결과를 응답 모양으로 렌더링하고, 도중에 터지면 에러로 바꿔 내보내고요.
이 순서를 누군가는 알고 엮어 줘야 해요.

이번 목표는 이거예요. **요청을 처리하려는 쪽이 안쪽 부품들을 일일이 엮지 않게 한다. 한 군데에 "이 요청 처리해 줘"라고만 부탁하면 끝나게.**

## 가장 정직하게 먼저

웹 서버의 요청 진입점이 그 부품들을 직접 순서대로 엮게 해 볼게요. 제일 정직하게요.

```java
// 웹 서버가 요청을 받는 진입점
void onRequest(String path, OutputStream out) {
    Handler handler = handlerMapping.find(path);              // 1) 핸들러 찾고
    if (handler == null) {
        out.write(notFound());
        return;
    }

    HandlerAdapter adapter = adapterRegistry.find(handler);  // 2) 맞는 어댑터 찾고
    try {
        String result = adapter.handle(handler, path);       // 3) 불러서 결과 받고
        String body = viewResolver.render(result);           // 4) 응답 모양으로 렌더링
        out.write(ok(body));
    } catch (Exception e) {
        out.write(error(e));                                 // 5) 터지면 에러로
    }
}
```

동작은 해요. 부품을 차례로 엮어 요청 하나를 끝까지 처리하죠. **여기까지 첫 커밋.** before예요.

## 진입점이 늘어나기 시작해요

요청을 처리해야 하는 자리가 하나둘 늘어요.

- 운영팀: "관리자 포털도 같은 요청 처리를 써요. 거기서도 똑같이 돌아가게요."
- QA팀: "테스트에서 요청을 흉내 내야 해요. 핸들러가 제대로 불리는지 보게요."
- 플랫폼팀: "비동기 처리 경로도 생겨요. 거기서도 같은 흐름이 필요하고요."
- 그리고: "처리 앞뒤에 인터셉터(로깅, 인증)를 끼울 거예요."

시키는 대로 해 보세요. 진입점이 하나 늘 때마다 저 1번부터 5번까지의 엮는 코드를 통째로 또 베껴요. 관리자 포털에도, 테스트 하니스에도, 비동기 경로에도요. 그리고 인터셉터를 끼우라고 하면? 그 베껴 둔 곳을
전부 찾아다니며 같은 줄을 또 끼워야 하고요.

```java
// 관리자 포털 진입점 - 똑같은 엮기를 또
void onAdminRequest(String path, OutputStream out) {
    Handler handler = handlerMapping.find(path);
    HandlerAdapter adapter = adapterRegistry.find(handler);
    // ... 1~5번이 그대로 또. 인터셉터까지 오면 여기도 저기도 손봐야 한다
}
```

**여기까지 두 번째 커밋.** 일단 박아 두세요.

## 혹시 문제를 느꼈나요?

이쯤 되면 보이죠. 그래도 짚어 둘게요.

- 요청을 처리하려는 쪽마다 매핑/어댑터/렌더/에러를 엮는 순서를 통째로 알아야 해요. 안쪽 부품 전부의 사용법이 진입점마다 복붙돼 있고요.
- 진입점이 하나 늘면 그 엮기 코드가 통째로 또 복제돼요. 산탄총 수술이죠. 한 군데서 순서를 틀리거나 에러 처리를 빠뜨리면 거기만 조용히 다르게 굴러가고요.
- 안쪽 흐름을 한 단계 바꾸려면(인터셉터 추가 같은) 복제된 진입점을 전부 찾아다녀야 해요.

이번엔 레벨 5, 6이랑 헷갈릴 수 있어요. 거기서도 뭔가를 감쌌으니까요. 근데 그건 핸들러 하나에 일을 더하거나 들여보낼지를 정한 거였죠. 오늘은 그게 아니에요. 부품을 새로 만들거나 감싸는 게 아니라, **이미
있는 여러 부품을 부르는 복잡한 순서를 한 군데로 치우는** 거예요. 일을 더하는 게 아니라 가리는 거죠.

## 방향만 던질게요

길이 떠오를 거예요. 방향만 대 줄게요.

1. 안쪽 부품 여럿(매핑/어댑터/렌더/에러)을 손님이 직접 엮게 하지 말고, "정문" 하나를 세워서 그 뒤에 다 숨기면 어떨까요. 손님은 정문에 "이 요청 처리해 줘" 한 번만 하고, 매핑이고 어댑터고는 몰라요.
2. 그러면 진입점이 여럿이어도(관리자, 테스트, 비동기) 전부 그 정문 하나만 부르면 돼요. 안쪽 순서가 바뀌거나 인터셉터가 끼어도 정문 안만 고치면 되고, 손님은 한 줄도 안 바뀌고요.
3. 단, 정문은 일을 새로 하는 게 아니에요. 안쪽 부품들을 어떤 순서로 부를지만 알고 위임할 뿐이죠. 그래서 레벨 5의 감싸개랑 달라요. 감싸개는 한 겹 일을 더했지만, 정문은 그냥 복잡한 안쪽을 가리고 쉬운
   입구 하나를 내주는 거예요.

도착점은 또렷해요. **진입점이 늘거나 안쪽 흐름이 바뀌어도 손님 코드는 안 바뀐다. 손님은 정문 하나만 알고, 매핑/어댑터/렌더가 안에서 어떻게 엮이는지는 모른다.** 거기까지 가면 after예요.

검증은 짧아요. 커밋 전에 자문해요. 새 진입점(예: gRPC 게이트웨이)을 하나 붙이는데, 매핑부터 에러까지 엮는 순서를 또 베껴 써야 하나요, 아니면 정문 하나 부르고 끝인가요? 정문 하나면 after예요.

## 회고: 정문이 너무 많이 가리진 않았나

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도요. 정문 뒤로 다 숨기면 입구는 깔끔한데, 그 안에서 무슨 일이 어떤 순서로
도는지가 한 꺼풀 가려져요. 디버깅할 때 정문을 열고 들어가야 흐름이 보이죠. 그리고 정문이 안쪽 부품 전부를 알다 보니, 부품이 늘수록 정문도 같이 무거워져요. 한곳에 모은 대가예요.

이건 그냥 흘려둘게요. 오늘 본인이 세운 그 정문, 사실 웹 프레임워크의 심장이에요. 레벨 1부터 조각조각 만들어 온 게 오늘 하나의 입구 뒤로 다 들어갔거든요. 그게 뭔지는 답에서 확인해요.
