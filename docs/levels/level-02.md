# 레벨 2: 컨테이너가 빈을 깨우고 재우기

안녕하세요. 두 번째 단계예요. 시작하기 전에 지난 걸 잠깐 같이 돌아볼게요. 이번 단계는 레벨 1과 나란히 놓고 보면 훨씬 선명해지거든요.

## 지난 단계 돌아보기

레벨 1에서 우리는 디스패처가 요청 경로를 핸들러로 잇게 만들었죠. before를 떠올려 보면, "경로가 맞는지 따지는 방식"이 디스패처의 핵심 메서드 안에 `else if`로 눌어붙어 있었어요. 정확 일치, 접두사, 와일드카드... 방식이 하나 늘 때마다 그 메서드를 다시 열어 수술해야 했고요.

after에서 한 일은 단순했어요. 변하는 그 한 조각, "비교하는 방식"만 공통 약속 뒤로 빼내고, 그걸 디스패처 바깥에서 골라 끼워 줬죠. 그러자 새 매칭 방식이 와도 디스패처 본체는 한 줄도 안 바뀌었어요. 한 문장으로 줄이면 이래요. **변하는 부분을 통째로 떼어내, 쓰는 쪽은 그게 뭔지 모르게 했다.** 디스패처는 그저 "맞는지 봐 줘" 하고, 바깥이 정해 준 것을 받아 썼을 뿐이고요.

거기까지 닿았나요? 아직이면 마저 끝내고 와도 좋아요. 이번 단계는 정확히 그 반대편을 보여줄 거예요.

## 이번엔 컨테이너로 내려갑니다

작업 공간을 옮길게요. 이번엔 `mini-spring/container` 모듈(`lab.spring.container`), DI 컨테이너입니다.

컨테이너가 빈을 다 만들어 의존성까지 꽂아 주고 나면 보통은 그걸로 끝이에요. 그런데 그냥 객체로 둬선 안 되는 녀석들이 있어요.

- 커넥션 풀은 쓰이기 전에 연결 몇 개를 미리 열어 둬야 해요.
- 로컬 캐시는 자주 찾는 값을 미리 채워(데워) 둬야 첫 요청이 안 느려요.
- 스케줄러는 등록되자마자 시계를 돌리기 시작해야 하고요.

그리고 애플리케이션이 내려갈 땐 거꾸로, 풀은 연결을 닫고 스케줄러는 멈춰야 해요. 안 그러면 자원이 샙니다.

즉 컨테이너에 **생명주기**가 필요해졌어요. "빈을 다 엮은 뒤 깨우고(start), 끝날 때 재운다(stop)." 이걸 만들어 봅시다.

## 가장 정직하게 먼저

컴포넌트마다 자기 `start()`를 직접 들고 있게 해 볼게요. 커넥션 풀부터요.

```java
class ConnectionPool {
    private boolean healthy;

    void start() {
        System.out.println("[ConnectionPool] 시작");        // 1) 시작 로그
        if (config == null)                                 // 2) 설정 검증
            throw new IllegalStateException("설정 없음");
        for (int i = 0; i < size; i++) openOne();           // 3) 진짜 할 일
        this.healthy = true;                                // 4) 정상 표시
        System.out.println("[ConnectionPool] 준비됨");      // 5) 완료 로그
    }
}
```

이제 캐시도 같은 식으로요.

```java
class CacheWarmer {
    private boolean healthy;

    void start() {
        System.out.println("[CacheWarmer] 시작");           // 1) 시작 로그
        if (source == null)                                 // 2) 설정 검증
            throw new IllegalStateException("소스 없음");
        preload();                                          // 3) 진짜 할 일
        this.healthy = true;                                // 4) 정상 표시
        System.out.println("[CacheWarmer] 준비됨");         // 5) 완료 로그
    }
}
```

스케줄러까지 만들면 1, 2, 4, 5번이 또 똑같이 반복될 거예요. 컨테이너는 그저 등록된 컴포넌트들의 `start()`를 차례로 불러 주기만 하면 되고요.

동작은 합니다. **여기까지 첫 커밋.** before예요.

## 운영팀의 요청이 들어옵니다

서비스가 커지면서 운영팀이 하나씩 부탁해 와요.

- "컴포넌트가 뜰 때마다 시작 시각이랑 걸린 시간을 남겨 주세요. 느린 놈을 찾아야 해서요."
- "start 도중에 터지면 그냥 죽지 말고, 어떤 컴포넌트가 왜 실패했는지 한 줄로 남기고 넘어가 주세요."
- "앞으로 이런 컴포넌트가 열 개, 스무 개로 늘 거예요. 그래도 규칙은 똑같이 지켜지게 해주세요."

이 요청들을 지금 구조에 넣어 보세요. 시작 시각 기록 하나를 넣으려고 해도 `ConnectionPool.start()`, `CacheWarmer.start()`, `Scheduler.start()`... 전부 열어서 같은 코드를 또 붙여야 해요. 실패 처리도 마찬가지고요. 컴포넌트가 스무 개면 똑같은 수술을 스무 번 하는 거죠.

**여기까지 두 번째 커밋.** 고통이 드러난 상태예요. 그리고 잠깐 멈춰서, 저 `start()`들을 나란히 놓고 보세요.

## 혹시 문제를 느꼈나요?

눈에 걸리는 게 있을 거예요.

- 모든 `start()`가 똑같은 뼈대를 반복해요. 1) 로그, 2) 검증, 4) 정상 표시, 5) 로그, 그리고 이 순서까지요. 정작 진짜로 다른 건 3) 한 칸뿐이고요.
- 그 공통 뼈대에 무언가를 더하거나 순서를 손보려 하면, 모든 컴포넌트를 일일이 찾아다녀야 해요. 하나라도 빠뜨리면 거기만 규칙이 깨지죠.
- 새 팀원이 컴포넌트를 추가하면서 정상 표시를 깜빡하거나, 검증을 진짜 할 일 뒤에 두면(순서를 어기면) 조용히 버그가 생겨요.

레벨 1과 묘하게 닮았는데, 방향이 정반대라는 걸 눈치챘나요? 그때는 "방식 전체"가 통째로 갈렸고 순서랄 게 없었어요. 지금은 반대예요. **전체 순서는 절대 안 바뀌어야 하고, 그 안의 딱 한 칸만 컴포넌트마다 달라요.**

길이 몇 가지 떠오를 거예요. 정답은 안 알려줄게요. 방향만 드립니다.

1. 변하지 않는 뼈대(로그 - 검증 - ??? - 정상 표시 - 로그, 그리고 이 순서)를 한 곳에 고정해 두고, 컴포넌트는 가운데 빈칸 하나만 채우게 한다면 어떨까요. 시작 시각 기록 같은 새 규칙이 와도 그 한 곳만 고치면 전부에 퍼지겠죠.
2. 그 "빈칸"을 누가 채우게 할지 보세요. 레벨 1에서는 바깥이 통째로 끼워 줬죠. 이번엔 누가 순서를 쥐고 있는지 다시 보세요. 순서를 쥔 쪽이 위에서 "자, 여기서 네 일을 해" 하고 아래로 빈칸만 내려보내는 그림이 더 자연스럽지 않나요?
3. 종료(`stop`)에도 똑같은 고민이 그대로 한 번 더 와요. 시작이 풀리면 종료는 거의 거저 풀릴 거예요.

목표 지점은 분명해요. **새 규칙을 넣을 때 단 한 곳만 고치면 모든 컴포넌트에 똑같이 적용된다. 그리고 새 컴포넌트는 "자기만의 그 한 칸" 말고는 어떤 뼈대도 다시 쓰지 않는다.** 거기까지 가면 after예요.
