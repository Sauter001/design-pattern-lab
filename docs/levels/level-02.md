# 레벨 2: 컨테이너가 빈을 깨우고 재우기

또 왔네요. 둘째 단계예요. 시작 전에 지난 걸 잠깐 돌아보고 가죠. 귀찮아도 이건 해야 돼요. 이번 단계는 레벨 1이랑 나란히 놓고 봐야 제맛이거든요.

## 지난 단계 돌아보기

레벨 1에서 우리는 디스패처가 요청 경로를 핸들러로 잇게 만들었죠. before가 어땠는지 기억나요? "경로가 맞는지 따지는 방식"이 디스패처 핵심 메서드 안에 `else if`로 눌어붙어 있었어요. 정확 일치, 접두사, 와일드카드... 방식 하나 늘 때마다 그 메서드를 또 열어서 수술했고요.

after에서 한 건 단순했어요. 변하는 그 한 조각, "비교하는 방식"만 공통 약속 뒤로 빼내고, 그걸 디스패처 바깥에서 골라 끼웠죠. 그랬더니 새 매칭 방식이 와도 본체는 한 줄도 안 바뀌었어요. 한 문장으로 줄이면 이래요. **변하는 부분을 통째로 떼어내, 쓰는 쪽은 그게 뭔지 모르게 했다.** 디스패처는 그냥 "맞는지 봐 줘" 하고, 바깥이 정해 준 걸 받아 썼을 뿐이고요.

거기까지 갔어요? 아직이면 마저 끝내고 와도 돼요. 안 기다릴게요. (농담이에요. 기다릴게요. 어차피 여기 있어요.) 이번 단계는 정확히 그 반대편을 보여줄 거라, 레벨 1을 손에 쥐고 있어야 비교가 돼요.

## 이번엔 컨테이너로 내려갑니다

작업 공간을 옮길게요. 이번엔 `mini-spring/container` 모듈(`lab.spring.container`), DI 컨테이너예요.

컨테이너가 빈을 다 만들어서 의존성까지 꽂아 주면 보통 거기서 끝이에요. 근데 그냥 객체로 두면 안 되는 녀석들이 있어요.

- 커넥션 풀은 쓰이기 전에 연결 몇 개를 미리 열어 둬야 해요.
- 로컬 캐시는 자주 찾는 값을 미리 채워(데워) 둬야 첫 요청이 안 느리고요.
- 스케줄러는 등록되자마자 시계를 돌려야 하죠.

그리고 애플리케이션이 내려갈 땐 거꾸로, 풀은 연결을 닫고 스케줄러는 멈춰야 해요. 안 그러면 자원이 새요. 새는 자원만큼 사람 피 말리는 것도 없고요.

즉 컨테이너에 **생명주기**가 필요해진 거예요. "빈을 다 엮은 뒤 깨우고(start), 끝날 때 재운다(stop)." 이걸 만들죠.

## 가장 정직하게 먼저

컴포넌트마다 자기 `start()`를 직접 들고 있게 해 볼게요. 커넥션 풀부터요.

```java
class ConnectionPool {
    private boolean healthy;

    void start() {
        System.out.println("[ConnectionPool] 시작");        // 1) 시작 로그
        if (config == null)                                 // 2) 설정 검증
            throw new IllegalStateException("설정 없음");
        for (int i = 0; i < size; i++) openOne();           // 3) 진짜 할 일
        this.healthy = true;                                // 4) 정상 표시
        System.out.println("[ConnectionPool] 준비됨");      // 5) 완료 로그
    }
}
```

이제 캐시도 같은 식으로요.

```java
class CacheWarmer {
    private boolean healthy;

    void start() {
        System.out.println("[CacheWarmer] 시작");           // 1) 시작 로그
        if (source == null)                                 // 2) 설정 검증
            throw new IllegalStateException("소스 없음");
        preload();                                          // 3) 진짜 할 일
        this.healthy = true;                                // 4) 정상 표시
        System.out.println("[CacheWarmer] 준비됨");         // 5) 완료 로그
    }
}
```

스케줄러까지 만들면 1, 2, 4, 5번이 또 똑같이 반복될 거예요. 보이죠? 컨테이너는 그냥 등록된 컴포넌트들의 `start()`를 차례로 불러 주기만 하면 되고요. 세 개 다 만들어 두세요. 이따 쓸 거예요.

동작은 해요. **여기까지 첫 커밋.** before예요.

## 운영팀의 요청이 들어옵니다

서비스가 커지면 옵니다. 이번엔 운영팀이에요. 하나씩 부탁해 와요.

- "컴포넌트 뜰 때마다 시작 시각이랑 걸린 시간 좀 남겨 주세요. 느린 놈을 찾아야 해서요."
- "start 도중에 터지면 그냥 죽지 말고, 어떤 컴포넌트가 왜 실패했는지 한 줄 남기고 넘어가 주세요."
- "앞으로 이런 컴포넌트가 열 개, 스무 개로 늘 거예요. 그래도 규칙은 똑같이 지켜지게요."

받았으니 해야죠. 첫 번째 요청부터 지금 구조에 넣어 보세요. "시작 시각이랑 걸린 시간 로그."

## 직접 해 보죠. 한 군데만 고쳐서 끝나나

이번엔 숫자를 세라고 안 할게요. 레벨 1에서 해 봤잖아요. 이번엔 다른 걸 직접 겪어 보죠.

운영팀 첫 요청, "시작 시각이랑 걸린 시간 기록"을 넣을 거예요. **`ConnectionPool.start()` 한 곳에만** 넣어 보세요. 이렇게요.

```java
void start() {
    long t0 = System.nanoTime();                        // 추가: 시작 시각
    System.out.println("[ConnectionPool] 시작");
    if (config == null)
        throw new IllegalStateException("설정 없음");
    for (int i = 0; i < size; i++) openOne();
    this.healthy = true;
    long ms = (System.nanoTime() - t0) / 1_000_000;     // 추가: 걸린 시간
    System.out.println("[ConnectionPool] 준비됨 (" + ms + "ms)");
}
```

자, 한 군데 고쳤어요. 그럼 운영팀 요청이 만족됐나요?

아니죠. `CacheWarmer`랑 `Scheduler`는 여전히 시간을 안 남겨요. 운영팀이 보고 싶은 건 "느린 놈"인데, 셋 중 하나만 시간을 찍으면 비교가 안 되잖아요. 그러니까 나머지 둘도 똑같이 열어서 똑같은 `t0`, 똑같은 `ms` 줄을 붙여야 해요.

**여기서 멈추지 말고, 진짜로 나머지 둘에도 붙여 보세요.** `CacheWarmer.start()` 열고, `Scheduler.start()` 열고, 같은 두 줄 복사해서요. 손으로요. 머릿속으로 "아 똑같이 하면 되겠네" 하고 넘어가지 말고, 실제로 복사-붙여넣기를 세 번 해 보세요. 그 세 번이 이 레벨의 전부예요.

## 방금 그게 무슨 느낌이었는지

붙였어요? 그럼 솔직하게 답해 보죠.

- 똑같은 두 줄을 **세 번** 복사했죠. 컴포넌트가 스무 개였으면 스무 번이고요.
- 운영팀 요청은 아직 **두 개** 남았어요. "실패해도 한 줄 남기고 넘어가기"는요? 그것도 또 세 군데(스무 군데) 돌면서 `try-catch`를 똑같이 붙여야 해요.
- 복사하다 보면 꼭 한 군데서 변수 이름을 안 바꾸거나, 한 컴포넌트를 통째로 빠뜨려요. 그럼 거기만 규칙이 깨지죠. 조용히요. 조용한 버그가 제일 무서워요.

이게 "산탄총 수술"이에요. 규칙 하나 바꾸는데 총알이 사방으로 흩어지듯 여러 군데를 똑같이 쏴 줘야 하는 거. **여기까지 두 번째 커밋.** 고통이 코드에 박힌 상태예요.

## 혹시 문제를 느꼈나요?

이제 보이죠. 안 보이면 짚을게요.

- 모든 `start()`가 똑같은 뼈대를 반복해요. 1) 로그, 2) 검증, 4) 정상 표시, 5) 로그, 그리고 이 순서까지요. 정작 컴포넌트마다 진짜로 다른 건 3) 한 칸뿐이에요.
- 그 공통 뼈대에 뭘 더하거나 순서를 손보려 하면, 방금처럼 모든 컴포넌트를 일일이 찾아다녀야 해요. 하나라도 빠뜨리면 거기만 규칙이 깨지고요.
- 새 팀원이 컴포넌트를 추가하면서 정상 표시를 깜빡하거나, 검증을 진짜 할 일 뒤에 두면(순서를 어기면) 조용히 버그가 나요.

레벨 1이랑 묘하게 닮았는데 방향이 정반대라는 거, 눈치챘어요? 그때는 "방식 전체"가 통째로 갈렸고 순서랄 게 없었어요. 지금은 반대예요. **전체 순서는 절대 안 바뀌어야 하고, 그 안의 딱 한 칸만 컴포넌트마다 달라요.**

## 이번 함정은 결이 달라요

레벨 1에선 "메서드로 빼면 끝났다고 착각하는" 함정이 있었죠. 이번 함정은 그거랑 결이 달라요. 매번 보는 거라 미리 말해 둘게요.

이번엔 다들 공통 두 줄을 `private void logStart()` 같은 **헬퍼 메서드**로 빼요. 그리고 각 `start()`에서 그 헬퍼를 불러요. 중복은 좀 줄죠. 근데 보세요. 각 컴포넌트가 여전히 **자기가 직접** `logStart()`를 부르고, 검증하고, `healthy = true`를 찍고, 순서를 지켜야 해요. 새 컴포넌트 만드는 사람이 그 호출 순서를 외워서 직접 나열해야 하는 거예요. 헬퍼를 부르는 걸 빠뜨리거나 순서를 틀리면? 그대로 조용한 버그고요.

**헬퍼로 빼는 건 "중복 제거"지 "순서 보장"이 아니에요.** 이 둘은 달라요. 우리가 운영팀한테 약속해야 하는 건 "스무 개가 전부 같은 순서로 돈다"는 보장이지, 단순히 줄 수 줄이는 게 아니거든요. 컴포넌트가 순서를 직접 쥐고 있는 한, 그 보장은 못 해요.

## 방향만 던질게요

길이 몇 개 보일 거예요. 답은 안 줘요. 어느 쪽으로 갈지만 가리킬게요.

1. 변하지 않는 뼈대(로그 - 검증 - ??? - 정상 표시 - 로그, 그리고 이 순서)를 한 곳에 고정해 두고, 컴포넌트는 가운데 빈칸 하나만 채우게 하면 어떨까요. 시작 시각 기록 같은 새 규칙이 와도 그 한 곳만 고치면 전부에 퍼지겠죠. 헬퍼랑 다른 점은, 컴포넌트가 순서를 부르는 게 아니라 **순서가 컴포넌트를 부른다**는 거예요.
2. 그 "빈칸"을 누가 채우게 할지 보세요. 레벨 1에서는 바깥이 변하는 조각을 통째로 끼워 줬죠. 이번엔 누가 순서를 쥐고 있는지 다시 보세요. 순서를 쥔 쪽이 위에서 "자, 여기까지 했으니 이제 네 일 해" 하고 아래로 빈칸만 내려보내는 그림이 더 자연스럽지 않아요?
3. 종료(`stop`)에도 똑같은 고민이 그대로 한 번 더 와요. 시작이 풀리면 종료는 거의 거저 풀릴 거예요. 거저 주는 거 흔치 않으니 챙기고요.

도착점은 분명해요. **새 규칙을 넣을 때 단 한 곳만 고치면 모든 컴포넌트에 똑같이 적용된다. 그리고 새 컴포넌트는 "자기만의 그 한 칸" 말고는 어떤 뼈대도, 어떤 순서도 다시 쓰지 않는다.** 거기까지 가면 after예요.

## 검증: 헬퍼냐 진짜 뼈대냐

다 됐다 싶으면, 커밋 전에 이거 통과하세요. 레벨 1 검증이랑 질문이 다르니까 헷갈리지 말고요.

> 새 컴포넌트를 하나 추가한다고 칩시다. 그 컴포넌트를 만드는 사람이 **"로그 -> 검증 -> 정상 표시" 순서를 알아야 하나요, 몰라도 되나요?**

- 알아야 한다 -> 아직이에요. 순서를 컴포넌트가 쥐고 있는 거고, 헬퍼 함정에 머문 거예요. 위 함정 섹션 다시 봐요.
- 몰라도 된다(자기 한 칸만 채우면 나머지는 알아서 굴러간다) -> 됐어요. 그게 after예요. 순서가 위에 고정됐고, 컴포넌트는 빈칸만 아는 거죠.

## 회고: 레벨 1이랑 나란히 놓고

다 되면 세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도 잊지 말고 적으세요. 뼈대를 한 곳에 고정한 대가로 뭐가 생겼는지: 추상 타입(혹은 상위 클래스)이 하나 생겼고, 컴포넌트가 그것에 묶였죠. 그 결합이 거래의 비용이에요.

이건 농담 안 하고 말할게요. 이번엔 꼭 레벨 1이랑 비교해서 한 줄 남기세요.

> 두 단계 다 "변하는 부분"을 다뤘어요. 그런데 레벨 1에선 변하는 조각을 **바깥에서 끼워 넣었고**, 레벨 2에선 변하는 빈칸을 **위에서 내려보냈어요.** 누가 순서를 쥐고 있었나요?

이 질문의 답이 두 단계의 차이를 한마디로 설명해 줘요. 코드 모양은 닮았는데 주도권이 정반대거든요. 그거 하나 건지면 이번 단계는 본전 뽑은 거예요.
