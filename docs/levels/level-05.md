# 레벨 5: 핸들러를 건드리지 않고 그 둘레에 일을 덧입히기

안녕하세요. 다섯 번째 단계예요. 여기서부터 결이 한 단계 까다로워집니다. 지금까지(레벨 1~4)는 이미 한 번쯤 겪어 본 걸 명시적으로 다시 짜는 워밍업이었어요. 이제부터는 **구조는 거의 똑같은데 의도만 다른**
것들을 둘씩 짝지어 보며, 그 미묘한 경계를 몸으로 잡는 구간이에요. 이번 단계엔 다음 단계에서 다시 만날 쌍둥이가 하나 숨어 있어요. 일단 지난 단계부터 짚고 갈게요.

## 지난 단계 돌아보기

레벨 4에서 우리는 모양이 제각각인 핸들러들을 디스패처가 알던 한 모양으로 부르려고, 핸들러마다 중간에 번역기를 끼웠죠. 그 번역기는 핸들러를 안에 품고, 바깥으로는 디스패처가 기대하는 `handle`인 척하면서,
안에서 `process`나 `service`로 바꿔 불렀어요. 한마디로 **감싸서 모양(인터페이스)을 바꾼** 거예요.

이 "감싼다"는 동작을 잘 기억해 두세요. 이번 단계에서도 우리는 무언가를 감쌀 거예요. 그런데 목적이 달라요. 지난번엔 모양을 바꾸려고 감쌌다면, 이번엔 모양은 그대로 두고 **하는 일을 더하려고** 감쌉니다.
끝나고 둘을 나란히 놓으면 "감싼다"가 두 얼굴을 가졌다는 게 보일 거예요.

작업 공간은 그대로 `mini-spring/mvc` 모듈(`lab.spring.mvc`)이에요.

## 우리가 만들고 싶은 것

핸들러는 자기 본업만 해요. 경로를 받아 응답을 만들죠. 그런데 요청 하나가 처리되는 길목에는 본업과 상관없이 거의 매번 끼고 싶은 것들이 있어요. 들어온 요청을 로그로 남기고, 처리 시간을 재고, 응답이 크면
압축하고, 민감한 경로면 응답을 가리는 것 같은 일들요.

이번 목표는 이거예요. **핸들러의 본업 코드는 그대로 둔 채, 그 앞뒤로 부가 행동을 자유롭게 덧입힌다.** 그리고 경로마다 어떤 행동을 어떤 순서로 입힐지 다르게 고를 수 있어야 해요.

## 가장 정직하게 먼저

디스패처가 핸들러를 부르는 길목에 그 부가 행동들을 직접 끼워 넣어 볼게요.

```java
class DispatcherServlet {

    String dispatch(Handler handler, String path) {
        long start = System.nanoTime();               // 시간 재기 시작
        System.out.println("[IN] " + path);           // 요청 로그

        String body = handler.handle(path);           // 진짜 할 일

        body = body.trim();                           // 응답 다듬기
        System.out.println("[OUT] " + body.length() + "자");   // 응답 로그
        System.out.println("took " + (System.nanoTime() - start) + "ns");
        return body;
    }
}
```

동작은 합니다. 핸들러 호출을 가운데 두고, 그 앞뒤로 로그와 타이밍이 붙죠. **여기까지 첫 커밋.** before예요.

## 요청들이 들어옵니다

서비스가 굴러가면서 여러 팀이 부가 행동을 하나씩 들고 와요. 받은 말을 그대로 옮기면 이래요.

- 보안팀: "민감한 경로(`/admin`, `/users/*`)는 응답에서 주민번호 같은 걸 가린 다음 내보내 주세요. 근데 공개 API엔 그거 켜지 마세요."
- 프론트팀: "응답이 일정 크기를 넘으면 gzip으로 압축해 주세요. 압축은 일부 경로에만요."
- 운영팀: "느린 요청만 따로 로그로 남기고 싶어요. 어떤 경로엔 타이밍을 켜고 어떤 경로엔 끄고요."

이걸 지금 구조에 넣어 보세요. 죄다 `dispatch` 한 메서드로 몰리는 데다, "경로마다 켜고 끄고"를 맞추려고 플래그와 `if`가 줄줄이 붙습니다.

```java
String dispatch(Handler handler, String path, boolean log, boolean timing,
                boolean mask, boolean gzip) {
    long start = timing ? System.nanoTime() : 0;
    if (log) System.out.println("[IN] " + path);

    String body = handler.handle(path);

    if (mask) body = mask(body);
    if (gzip && body.length() > 1024) body = gzip(body);
    if (log) System.out.println("[OUT] " + body.length() + "자");
    if (timing) System.out.println("took " + (System.nanoTime() - start) + "ns");
    return body;
}
```

**여기까지 두 번째 커밋.** 고통이 드러난 상태예요. 잠깐 멈춰서, 저 플래그 더미를 노려보세요.

## 혹시 문제를 느꼈나요?

걸리는 게 한둘이 아닐 거예요.

- 부가 행동이 하나 늘 때마다 `dispatch`의 인자와 본문이 같이 부풀어요. 본업 한 줄(`handler.handle`)이 켜고 끄는 `if` 더미에 파묻혔고요.
- 행동들의 순서가 본문에 박혀 버렸어요. 어떤 경로에선 압축을 마스킹보다 먼저 하고 싶어도, 지금 구조에선 코드를 다시 짜야 바뀌죠. 조합과 순서를 경로마다 다르게 주는 게 사실상 불가능해요.
- 압축 로직 하나만 따로 시험하고 싶어도, 디스패처 전체와 플래그 조합을 세워야만 그 가지에 닿을 수 있어요.

지난 단계가 떠오를 거예요. 거기서도 우리는 무언가를 감쌌죠. 그런데 그땐 안에 품은 것을 **다른 모양으로** 바꿔 내보냈어요. 지금 하고 싶은 건 달라요. 핸들러가 내놓는 모양(
`String handle(String)`)은 그대로 두고, 그 둘레에 일을 한 겹 더 입히고 싶은 거예요. 모양은 안 바뀌고, 하는 일만 늘어나는 거죠.

길이 떠오를 거예요. 방향만 드릴게요.

1. 저 부가 행동들은 전부 똑같은 모양이에요. 진짜 호출 앞에 뭘 좀 하고, 호출을 넘기고, 뒤에 뭘 좀 하죠. 그렇다면 부가 행동 하나하나를 **"핸들러를 품은 또 다른 핸들러"**로 만들면 어떨까요. 바깥에서
   보면 똑같이 `handle`을 받는 핸들러인데, 안에서 자기 몫(로그, 타이밍, 압축)을 하고 품은 핸들러에게 일을 넘기는 거예요.
2. 그렇게 만들면 양파처럼 겹겹이 쌀 수 있어요. 진짜 핸들러를 로깅으로 한 겹 싸고, 그 바깥을 타이밍으로 또 싸고요. 경로마다 어떻게 쌀지 조립만 다르게 하면, `/admin`엔 로깅+마스킹을,
   `/public`엔 압축만 입힐 수 있죠. 디스패처는 여전히 맨 바깥 한 겹에게 `handle`만 부르면 되고요.
3. 핵심은 "감싸되 모양은 그대로 둔다"예요. 레벨 4의 번역기는 감싸서 인터페이스를 바꿨지만, 이번 감싸개는 같은 인터페이스를 그대로 내보내면서 그 안에 책임만 한 겹 더해요. 들어오는 것도 핸들러, 나가는 것도
   핸들러여야 양파처럼 계속 쌀 수 있다는 점을 떠올려 보세요.

이번 결승선은 이겁니다. **새 부가 행동을 추가해도 디스패처도, 진짜 핸들러도 단 한 줄 안 바뀐다. 행동 하나가 감싸개 하나가 되고, 경로마다 감싸는 조합과 순서를 자유롭게 짠다.** 거기까지 가면 after예요.

다 되면 세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적되, 이번엔 레벨 4와 꼭 나란히 두고 한 줄 남겨 보세요. "둘 다 감쌌는데, 한쪽은
무엇을 바꿨고 다른 쪽은 무엇을 더했나?" 그리고 미리 귀띔 하나. **다음 단계에서 이것과 거의 똑같이 생긴 구조를 또 만들 거예요.** 코드만 보면 구분이 안 갈 정도로요. 그런데 그게 푸는 문제는 전혀
다릅니다. 그때 이번 회고를 다시 꺼내 두 개를 나란히 놓고 보면, "구조가 같아도 의도가 다르면 다른 것"이라는 감각이 생길 거예요. 다음 단계 첫머리에서 같이 되짚겠습니다.
