# 레벨 14: 페이지가 바뀌면, 알리기만 한다

또 봐요. 열네 번째예요. 이쯤이면 흐름은 설명 안 해도 알죠. 그래도 할게요. 습관이라.

동네는 미니 DB 저장 층 그대로예요. `mini-db/storage` 모듈(`lab.db.storage`). 레벨 8, 12, 13에서 머물던 그곳요.

오늘은 솔직히 미리 한마디 할게요. 이번 건 새 패턴이 아닐 수도 있어요. 만들다 보면 "어, 이거 어디서 했는데" 싶을 거예요. 그 느낌이 맞아요. 그게 오늘의 핵심이고요. 패턴이 도메인을 안 가린다는 걸 직접 보는 자리예요.

## 이번에 마주칠 상황

저장 엔진이 페이지를 쓸 때, 그 변경을 로그에 먼저 남겨야 해요. 죽었다 깨어나도 복구하려고요(write-ahead log, 변경보다 로그가 앞선다고 WAL이라 불러요). 그런데 페이지가 바뀌면 반응해야 할 게 로그만이 아니에요. 쓰기 통계도 올려야 하고, 곧 복제도, 캐시 무효화도 따라붙어요.

이번 목표는 이거예요. **페이지 하나가 바뀌면, 거기에 반응해야 하는 여러 곳이 빠짐없이 움직이게 한다.**

## 가장 정직하게 먼저

저장 엔진이 페이지를 쓰면서 반응할 것들을 직접 줄줄이 부르게 해 볼게요.

```java
class StorageEngine {

    void writePage(Page p, byte[] data) {
        wal.append(p.id(), data);      // write-ahead: 변경 전에 로그부터
        p.apply(data);                 // 진짜 할 일: 페이지 변경
        metrics.incWrites();           // 곁다리: 쓰기 통계
    }
}
```

동작은 해요. 로그 남기고, 바꾸고, 통계 올리죠. **여기까지 첫 커밋.** before예요.

## 반응할 게 자꾸 늘어납니다

저장 엔진이 자리를 잡으니, 여러 팀이 "쓰기 일어날 때 이것도" 하며 찾아와요.

- 복제팀: "쓰기를 다른 노드로 복제해야 해요. 변경 날 때마다 복제 스트림으로 보내 주세요."
- 캐시팀: "그 페이지 캐시된 거 있으면 무효화해 주세요. 안 그럼 옛날 값 읽어요."
- 모니터링팀: "느린 쓰기만 따로 추적도요."

시키는 대로 `writePage`에 넣어 보세요. 가운데가 순식간에 불어나요.

```java
void writePage(Page p, byte[] data) {
    wal.append(p.id(), data);
    p.apply(data);
    metrics.incWrites();
    replicator.send(p.id(), data);     // 복제팀
    cache.invalidate(p.id());          // 캐시팀
    slowTracker.record(p.id());        // 모니터링팀
    // 다음 분기엔 또 뭐가 붙겠죠
}
```

**여기까지 두 번째 커밋.** 일단 박아 두세요.

## 혹시 문제를 느꼈나요?

어, 이 그림 어디서 봤죠? 레벨 3이에요. 가입 핸들러가 메일이랑 쿠폰이랑 통계를 줄줄이 직접 불렀던 그거요. 똑같아요. 이름표만 바뀌었어요. "가입"이 "페이지 변경"으로, "메일"이 "WAL"로요. 그러니 걸리는 데도 그때랑 같을 거예요.

- 저장 엔진의 본업은 "페이지를 쓴다" 한 줄인데, 그 아래로 남의 일이 잔뜩 쌓여 본업을 덮었어요.
- "쓰기 날 때 이것도" 한 건 올 때마다 `writePage`를 다시 열어야 해요. 복제 넣다가 캐시 코드 건드리고, 복제에서 터지면 멀쩡한 쓰기까지 같이 죽고요.
- 페이지 쓰기만 따로 시험하고 싶어도 WAL, 복제, 캐시, 통계를 전부 세워 줘야 해요.

레벨 3에서 이걸 어떻게 풀었는지 기억나면, 오늘은 거의 다 온 거예요.

## 방향만 던질게요

레벨 3에서 푼 그대로예요. 그래서 길게 안 끌게요.

1. 저장 엔진은 "페이지 N이 바뀌었다"만 알리고, WAL/복제/캐시/통계가 각자 손 들어 구독하게 하면 어떨까요. 엔진은 누가 듣는지 몰라요. 반응을 더하고 빼는 게 곧 구독하고 빠지는 일이 되고, `writePage`는 한 줄도 안 바뀌겠죠.
2. 그러면 새 반응(감사 로그 같은)이 와도 저장 엔진은 안 건드려요. 구독자 하나 새로 만들어 등록하면 끝이고요.
3. 단, 레벨 3이랑 한 가지 달라요. WAL은 변경보다 **먼저**, 그리고 **확실히** 기록돼야 해요. 그냥 뿌리고 잊는 알림이 아니라 순서와 보장이 걸린 구독이에요. 레벨 3의 가입 알림은 순서가 상관없었지만, 여긴 아니죠. 그 제약을 어디서 어떻게 챙길지도 같이 생각해 보세요. 같은 모양이어도 똑같이 풀면 안 되는 지점이 거기예요.

도착점은 또렷해요. **쓰기에 반응을 더하거나 빼도 저장 엔진의 `writePage`는 한 줄도 안 바뀐다. 저장 엔진은 누가 반응하는지 모른다. 단, WAL이 먼저라는 순서는 지켜진다.** 거기까지 가면 after예요.

검증은 레벨 3이랑 같은 질문이에요. 커밋 전에 자문해요. 쓰기에 반응할 게 하나 더 와요(예: 감사 로그). 그거 넣으려고 `writePage`(저장 엔진)를 여나요? 안 열고 구독자 하나 추가로 끝나면 after예요.

## 회고: 레벨 3을 옆에 펴고

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용은 레벨 3이랑 같아요. "페이지를 쓰면 무슨 일이 따라 도는지"가 코드 한 곳에서 안 보이게 됐어요. 추가/삭제는 거저 됐지만 흐름은 흩어졌죠.

그리고 비용 하나 더. 레벨 3 땐 순서가 상관없었는데 여긴 WAL이 먼저여야 해요. 구독으로 풀면 그 순서 보장이 약해지기 쉬워요. 그걸 어떻게 지켰는지 꼭 적으세요.

이건 그냥 흘려둘게요. 여러 구독자 사이의 순서나 조건이 슬슬 거슬리기 시작했다면, 그 직감 붙들어 두세요. 다음 단계가 정확히 그 얘기거든요. 발행-구독으로는 못 푸는 게 나와요.
