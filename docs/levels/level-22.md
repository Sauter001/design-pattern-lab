# 레벨 22: 트리에게 직접 묻기, 너 이 행에 참이냐

또 봐요. 스물두 번째예요. 동네는 안 옮겨요. `mini-db/frontend` 모듈(`lab.db.frontend`) 그대로요. 레벨 21에서 만든 그 조건 트리, 오늘 그걸 씁니다. 버리지 말랬죠. 잘 들고 있었길 바라요.

레벨 21에서 우리는 잎과 가지를 똑같이 다루는 트리를 지었어요. 그땐 "모양"이었죠. 오늘은 "뜻"이에요. 그 트리로 실제 일을 시켜요. 행 하나를 주고 "이 조건에 맞아, 안 맞아?"를 답하게요.

## 이번에 마주칠 상황

WHERE 절의 진짜 용도가 뭐였죠. 테이블을 훑으면서 각 행이 조건에 맞는지 가려 내는 거예요. `age > 18 AND vip = true`라는 트리에 `{age: 20, vip: true}` 행을 대면 참, `{age: 15, vip: true}`면 거짓. 트리가 행을 받아 참/거짓 하나를 뱉어야 해요.

이번 목표는 이거예요. **조건 트리에 행을 주면 참/거짓을 돌려준다. 그리고 새 조건 종류(NOT, BETWEEN, IN)가 와도 평가하는 본체는 안 바뀐다.**

## 가장 정직하게 먼저

평가는 왠지 엔진이 하는 일 같죠. 그러니 트리 바깥에 평가기를 하나 두고, 노드 타입을 따져 계산하게 해 볼게요. 제일 정직하게요. 그리고... 익숙하게요.

```java
boolean eval(Condition node, Row row) {
    if (node instanceof Comparison c) {
        Object actual = row.get(c.column);
        return compare(actual, c.op, c.value);          // >, =, < ...
    } else if (node instanceof BoolGroup g) {
        if (g.op.equals("AND")) {
            for (Condition ch : g.children) if (!eval(ch, row)) return false;
            return true;
        } else {
            for (Condition ch : g.children) if (eval(ch, row)) return true;
            return false;
        }
    }
    throw new IllegalStateException("모르는 노드네요");
}
```

동작은 해요. 행을 대면 참/거짓이 나오죠. **여기까지 첫 커밋.** before예요.

그런데 잠깐. 이 instanceof 사슬, 어디서 봤죠? 레벨 21에서 우리가 막 걷어낸 그거예요. 또 만났네요. 반갑죠? 전 안 반가워요. 평가라는 새 일이 생기니까, 그게 "엔진 로직" 같아서 슬그머니 트리 바깥으로 빼고 분기를 또 짠 거예요. 손이 기억하는 대로요. 일단 박아 두고 계속 가요.

## 평가할 조건이 늘어요

- 분석팀: "NOT, BETWEEN, IN 조건도 평가해야 해요."
- 리포트팀: "함수도요. `LOWER(name) = 'kim'` 같은 거요."

넣어 보세요. 조건 종류가 하나 늘 때마다 저 `eval`의 instanceof 사슬에 `else if`를 한 칸씩 더 답니다. 평가기 한가운데를 자꾸 열고요.

```java
boolean eval(Condition node, Row row) {
    if (node instanceof Comparison c)      { ... }
    else if (node instanceof BoolGroup g)  { ... }
    else if (node instanceof NotNode n)    { return !eval(n.child, row); }   // 새 조건 올 때마다
    else if (node instanceof BetweenNode b){ ... }                           // 여기 한 칸씩
    throw new IllegalStateException("모르는 노드네요");
}
```

**여기까지 두 번째 커밋.** 고통이 드러난 상태예요.

## 혹시 문제를 느꼈나요?

- 레벨 21에서 분명히 걷어낸 instanceof 사슬이, 평가기에 그대로 부활했어요. 모양만 바뀌어서요.
- 노드 타입이 하나 늘면 평가기 본체를 열어요. 산탄총, 또요.
- 트리 구조(레벨 21에서 잎/가지 똑같이 다루던)랑 평가 로직이 따로 놀아요. 그래서 새 조건은 두 군데에 손대야 해요. 노드 클래스 하나, 그리고 평가기 분기 하나.

이쯤에서 레벨 21이랑 헷갈릴 거예요. "이거 어제 거랑 같은 거 아냐?" 거의 같아요. 트리도 그대로고, 노드 안에 메서드 넣어 푸는 것도 곧 그대로일 거고요. 그런데 결이 달라요. 레벨 21은 "잎과 가지를 똑같이 다루는 구조" 자체가 목적이었어요. render는 그걸 보여줄 핑계였고요. 오늘은 그 트리를 **하나의 작은 언어로 보고, 행이라는 입력을 먹여서 그 언어를 실행**해요. 트리는 문장, 행은 입력, 평가는 실행이에요. 같은 트리 위에 "뜻"이 한 겹 올라탄 거죠. 그래서 이 둘은 따로면서 한 몸이에요.

## 방향만 던질게요

길이 떠오를 거예요. 방향만 대 줄게요.

1. 평가를 트리 바깥에 두지 마세요. 레벨 21에서 render를 노드 안에 넣었던 것처럼, 각 노드가 자기 자신을 평가하게 해요. 단 이번엔 입력(행)을 받아서요. `boolean evaluate(Row row)`를 공통 약속에 넣는 거예요.
2. 잎(Comparison)은 행에서 자기 컬럼 값을 꺼내 비교하면 돼요. 가지(AND/OR)는 자식들의 `evaluate(row)`를 부르고 결과를 논리로 합치면 되고요. 레벨 21의 그 내려가기 그대로인데, 이번엔 참/거짓을 모아 올리는 거예요.
3. 그러면 새 조건(NOT, BETWEEN)은 `evaluate`를 아는 클래스 하나 더예요. 평가기 본체요? 없어요. 평가가 노드들에 흩어져 사니까, 열어서 고칠 본체 자체가 없는 거죠.
4. 그림으로 보면 이래요. 트리는 작은 언어로 쓴 한 문장, 행은 그 문장에 먹이는 입력. 트리에게 "이 행에 대해 너 참이야?"라고 물으면, 트리가 스스로 답해요. 잎부터 답이 위로 모여 올라오고요. 묻는 쪽은 트리 속 생김새를 하나도 몰라요.

도착점은 또렷해요. **조건 트리에 행을 주면 트리가 스스로 참/거짓을 답한다. 새 조건 종류가 와도 열어 고칠 평가 본체가 없고, 클래스 하나만 는다. 노드 타입을 따지는 instanceof 사슬이 트리 바깥 어디에도 없다.** 거기까지 가면 after예요.

검증은 짧아요. 커밋 전에 자문해요. 새 조건(예: BETWEEN)을 추가하는데, 평가하는 분기 사슬을 여나요? 안 열고 클래스 하나 추가로 끝나면 after예요. 그리고 트리에 행을 물었을 때, 트리 바깥 어딘가에 노드 타입 따지는 instanceof가 남아 있나요? 한 군데도 없으면 after고요.

## 회고: 같은 트리, 다른 무게 (그리고 슬슬 불안할 거예요)

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도요. 노드마다 이제 메서드가 둘이에요. render(레벨 21)에 evaluate(오늘)까지요. 

그리고 이건 농담 안 하고 말할게요. 슬슬 불안하지 않아요? 새 일이 또 생기면, 그러니까 "타입 체크해라", "실행 계획으로 바꿔라" 같은 게 오면, 노드마다 메서드가 셋, 넷으로 늘 거예요. 노드 클래스가 점점 무거워지고, 새 연산 하나 추가하는 게 모든 노드 클래스를 다 건드리는 일이 되죠. 오늘 방식의 약점이 정확히 거기예요. 이 불편함, 회고에 또렷이 적어 두세요. **다음 단계가 바로 그 약점을 정면으로 건드리거든요.** 오늘 게 틀려서가 아니에요. 트리에 일을 시키는 방식이 하나가 아니라서 그래요.

레벨 21 회고랑 나란히 놓고 한 줄. 레벨 21은 "모양을 똑같이", 오늘은 "그 모양에 뜻을 실어 실행". 같은 트리예요. 패턴 책은 이 둘을 다른 장에 두지만, 사실 한 트리 위에 쌓은 두 겹이에요. 그걸 손으로 겪는 게 이 구간의 절반이고요.
