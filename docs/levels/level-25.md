# 레벨 25: 두 시스템을 붙이되, 한쪽 사정에 다른 쪽이 휘둘리지 않게

또 봐요. 스물다섯 번째예요. 어, 레벨 24에서 끝인 줄 알았죠? 패턴을 새로 발견하는 건 거기까지가 맞아요. 그런데 레벨 0에서 제가 약속한 거 하나, 아직 안 지켰어요. 기억나요? "작은 프로젝트 두 개를 만드는데, 이 둘은 한 군데서 만난다"고요. 그거 거두러 왔어요. 거 봐요, 흘린 말은 언젠가 돌아와요. 늘 그래요.

동네는 새 곳이에요. 두 프로젝트가 맞닿는 자리, `integration` 모듈(`lab.integration`)이고요. 미니 스프링이 미니 DB를 감싸는 데예요. 실제 스프링이 DB를 다루는 방식의 축소판이고요.

## 거슬러 올라가 돌아보기

미니 DB는 이제 다 됐어요. 쿼리를 읽고(frontend), 실행하고(engine), 저장하고(storage). 미니 스프링도 됐고요. 빈을 등록해 주입하고(container), 요청을 처리하고(mvc). 그런데 이 둘은 아직 서로 모르는 남남이에요. 오늘 붙여요.

그림은 이래요. 웹 요청이 mvc로 들어와요. 처리하다 데이터가 필요하면, 컨테이너가 주입해 준 리포지토리한테 물어요. 그 리포지토리가 미니 DB에 쿼리를 날려 행을 받아오고요. 스프링 데이터 써 봤으면 익숙한 그림이죠. 그 속을 직접 엽니다.

## 이번에 마주칠 상황

`UserRepository`라는 빈이 있다 칩시다. 얘가 미니 DB로 쿼리를 돌려야 해요. 그런데 미니 DB의 진짜 API는 미니 DB 사정대로 생겼어요. 문자열을 파싱하고(frontend), 계획을 실행하고(engine), 행을 받고요. 리포지토리가 그걸 직접 불러요. 제일 정직하게요.

이번 목표는 이거예요. **리포지토리가 미니 DB의 구체적 API를 직접 부르지 않게 한다. 그래서 뒤에 깔린 저장소를 갈아끼워도 리포지토리는 안 바뀐다.**

## 가장 정직하게 먼저

리포지토리가 미니 DB API를 그대로 호출해요.

```java
class UserRepository {

    User findById(int id) {
        Query q = frontend.parse("SELECT * FROM users WHERE id = " + id);  // 미니 DB 전용 API
        List<Row> rows = engine.execute(q);                                // 미니 DB 전용 API
        return toUser(rows.get(0));
    }

    List<User> findActive() {
        Query q = frontend.parse("SELECT * FROM users WHERE active = true");
        return engine.execute(q).stream().map(this::toUser).toList();
    }
}
```

아, 하나 짚고요. 저 `frontend.parse`, 우리한테 진짜 문자열 파서는 없죠. 레벨 21에서 23까지 조건 트리만 지었지, SELECT 문장 전체를 읽어 내는 파서는 안 만들었으니까요. 괜찮아요. 오늘 주제는 파싱이 아니라 붙이기라서요. `parse`는 최소 스텁이면 돼요. 정해진 쿼리 몇 개를 알아보고 레벨 11에서 지은 `Query`로 조립해 주는 정도요. `User`도 필드 두엇짜리 record면 충분하고요. 파서를 제대로 짓는 건 이 lab 바깥 일이에요.

동작은 해요. 리포지토리가 미니 DB에서 사용자를 꺼내 오죠. **여기까지 첫 커밋.** before예요.

## 저장소를 갈아끼울 일이 옵니다

팀들이 하나씩 들고 와요.

- "테스트할 때마다 진짜 미니 DB를 띄우기 무거워요. 인메모리 가짜 저장소로 바꿔 끼우게 해주세요."
- "외부 데이터소스가 하나 더 붙어요. 걘 API가 완전히 달라요. `parse`/`execute`가 아니라 자기만의 호출 방식이 있고요."
- "미니 DB 엔진 API 시그니처가 바뀌었어요. `execute`가 인자를 하나 더 받게요."

시키는 대로 해 보세요. 저장소를 바꾸거나 API가 바뀔 때마다, 그걸 직접 부르는 리포지토리를 전부 찾아다니며 고쳐야 해요. `UserRepository` 열고, `OrderRepository` 열고, `ProductRepository` 열고. 미니 DB 사정이 바뀌었을 뿐인데, 스프링 쪽 리포지토리가 죄다 흔들려요.

```java
User findById(int id) {
    // 인메모리로 바꾸면 이 두 줄이 통째로 다른 호출로 바뀐다. 모든 리포지토리에서.
    Query q = frontend.parse("SELECT * FROM users WHERE id = " + id);
    List<Row> rows = engine.execute(q, defaultOptions);   // 시그니처 바뀜? 여기도 다 손봄
    return toUser(rows.get(0));
}
```

**여기까지 두 번째 커밋.** 고통이 드러난 상태예요.

## 혹시 문제를 느꼈나요?

이쯤이면 보일 거예요. 그래도 짚어요.

- 리포지토리마다 미니 DB의 구체적 API가 박혀 있어요. 미니 DB가 기침하면 스프링 쪽이 다 같이 감기 걸려요.
- 저장소를 갈아끼우려면(인메모리, 다른 데이터소스) 리포지토리를 전부 열어야 해요. 산탄총, 또요.
- 스프링 쪽 코드가 미니 DB 내부 모양(`Query`, `parse`, `execute`)을 알아야 해요. 두 시스템이 꽉 붙어서, 한쪽만 떼어 테스트하기도 어렵고요.

레벨 4 기억나요? 디스패처가 바라는 모양이랑 남의 핸들러 모양이 안 맞아서, 둘 사이에 번역기를 끼웠죠. 오늘이 딱 그 모양이에요. 한쪽(스프링 리포지토리)이 바라는 모양과, 못 바꾸는 남의 모양(미니 DB API) 사이가 안 맞아요. 그래서 푸는 길도 그때랑 같고요.

## 방향만 던질게요

길이 떠오를 거예요. 방향만요.

1. 스프링 쪽이 바라는 균일한 약속 하나를 정하세요. 이를테면 `List<Row> run(String sql)` 하나짜리 인터페이스(이름은 뭐, `Driver`나 `Connection` 같은 거요). 리포지토리는 오직 이 약속만 알게요. 미니 DB가 뭔지는 모르고요.
2. 미니 DB의 진짜 API를 그 약속 뒤로 번역하는 조각 하나를 만들어요. 그 조각이 `run(sql)`을 받으면 속으로 `frontend.parse` 부르고 `engine.execute` 불러서, 결과를 약속된 모양으로 돌려주고요. 미니 DB의 울퉁불퉁한 API가 이 조각 한 곳에만 갇혀요.
3. 그러면 인메모리 저장소는? 같은 약속을 따르는 조각 하나 더예요. `run(sql)`을 받아 자기 식으로 처리하면 되고요. 리포지토리는 둘 중 뭐가 끼워졌는지 몰라요. 갈아끼워도 한 줄도 안 바뀌고요. 어느 걸 끼울지는 컨테이너가 주입으로 정하면 돼요. 레벨 9, 10에서 만든 그 주입이요.
4. 이게 사실 JDBC가 하는 일이에요. 자바 코드는 `Connection`/`Statement`라는 약속만 알고, 오라클이든 MySQL이든 각자의 드라이버가 그 약속 뒤에서 자기 DB로 번역하죠. 본인이 지금 그 구조의 축소판을 짓는 거예요.

도착점은 또렷해요. **리포지토리는 균일한 약속 하나만 안다. 저장소를 갈아끼워도(미니 DB -> 인메모리) 리포지토리는 안 바뀐다. 미니 DB 전용 API는 번역 조각 한 곳에만 산다.** 거기까지 가면 after예요.

검증은 짧아요. 커밋 전에 자문해요. 저장소를 인메모리로 바꾸는데 `UserRepository`를 여나요, 안 여나요? 안 열고 번역 조각 하나 갈아끼우면 끝이면 after예요.

## 회고: 균일하게 맞춘 대가 (꼭)

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도요.

- 번역 계층이 한 겹 생겼어요. 호출 하나가 리포지토리 -> 약속 -> 번역 조각 -> 미니 DB로 한 칸 더 거쳐요. 디버깅할 때 그 칸을 따라가야 하고요.
- 균일 약속은 보통 최소공통분모로 좁아져요. 미니 DB만의 특별한 기능(이를테면 미니 DB 고유의 힌트)이 있어도, 모두에게 공통인 약속에 안 들어가면 못 쓰거나 따로 빼야 해요. JDBC로 짜면 특정 DB 전용 기능 쓰기 번거로운 거, 그거랑 같은 값이에요.

레벨 4(어댑터) 회고를 옆에 펴고 한 줄 적어요. 둘 다 "안 맞는 모양을 원하는 모양으로 번역"한 거예요. 한쪽은 핸들러, 한쪽은 DB 드라이버. 도메인만 다르지 같은 손동작이죠. 레벨 14에서 옵저버를 두 도메인에서 만났을 때처럼, 패턴은 자리를 안 가린다는 그 감각 또 한 번이에요.
