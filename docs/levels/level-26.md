# 레벨 26: 커넥션은 보이지 않게, 호출자는 쿼리만

진짜 마지막이에요. 스물여섯 번째. 동네는 그대로 `integration` 모듈(`lab.integration`)이고요. 레벨 25에서 미니 DB를 균일한 약속(`Driver`) 뒤로 감쌌죠. 오늘 그 위에 마지막 한 겹을 씌워요. 이거 끝나면, 레벨 0에서 약속한 그림이 통째로 완성돼요.

## 거슬러 올라가 돌아보기

레벨 25에서 리포지토리는 이제 `Driver`로 쿼리를 날려요. 미니 DB가 뭔지 몰라도 되게 됐고요. 그런데 한 가지가 빠졌어요. **커넥션.** 쿼리를 날리려면 어딘가에 연결을 열어야 하고, 다 쓰면 닫아야 하고, 트랜잭션이 걸리면 그동안 같은 연결을 붙들고 있어야 해요.

그래서 약속이 한 칸 자랐다 치고 가요. 레벨 25에선 `driver.run(sql)` 한 줄이었는데, 연결이라는 현실이 들어오면서 `driver.connect()`로 커넥션을 얻고 쿼리는 그 커넥션 위에서 돌리는 모양(`c.run(sql)`)이 된 거예요. 그 진화 자체는 손에 익을 테니 먼저 반영해 두고요. 문제는 그 커넥션을 누가 챙기느냐예요. 지금은 리포지토리가 직접 해요.

## 이번에 마주칠 상황

커넥션은 공짜가 아니에요. 여는 데 비용이 들고, 안 닫으면 새고, 트랜잭션은 시작부터 끝까지 한 커넥션 위에서 굴러야 해요. 리포지토리 메서드마다 이걸 손으로 하면 어떻게 되는지 봐요. 제일 정직하게요.

이번 목표는 이거예요. **리포지토리에서 커넥션을 열고 닫는 코드를 걷어낸다. 호출자는 쿼리만 부르고, 커넥션이 열렸는지 닫혔는지도 모른다. 그리고 여러 호출을 한 트랜잭션으로 묶을 수 있다.**

## 가장 정직하게 먼저

리포지토리 메서드마다 커넥션을 열고, 쓰고, finally로 닫아요.

```java
class UserRepository {

    User findById(int id) {
        Connection c = driver.connect();          // 매번 연다
        try {
            List<Row> rows = c.run("SELECT * FROM users WHERE id = " + id);
            return toUser(rows.get(0));
        } finally {
            c.close();                            // 매번 닫는다. 빼먹으면 샌다.
        }
    }

    void deactivate(int id) {
        Connection c = driver.connect();          // 또 연다
        try {
            c.run("UPDATE users SET active = false WHERE id = " + id);
        } finally {
            c.close();
        }
    }
}
```

동작은 해요. 사용자도 꺼내고 비활성화도 하죠. **여기까지 첫 커밋.** before예요.

## 트랜잭션이 들어옵니다

서비스가 자라면서 요구가 와요.

- "주문 한 건 처리하는 데 리포지토리 호출이 세 번이에요. 재고 줄이고, 주문 만들고, 결제 기록하고. 이 셋이 다 성공 아니면 다 취소돼야 해요. 지금은 각자 다른 커넥션이라 중간에 실패하면 앞엣것만 남아요."
- "커넥션 여는 게 느려요. 풀에 두고 빌려 쓰게 해주세요."
- "누가 자꾸 `close`를 빼먹어서 커넥션이 새요. 아예 까먹을 수 없게 해주세요."
- "`@Transactional`처럼, 메서드에 표시만 하면 트랜잭션으로 묶이게 하고 싶어요."

시키는 대로 해 보세요. 트랜잭션으로 묶으려면 세 호출이 같은 커넥션을 써야 하는데, 지금은 메서드마다 따로 열어 버려서 묶을 수가 없어요. 그래서 커넥션을 인자로 넘기기 시작하고, 트랜잭션 시작/커밋/롤백 코드가 서비스 여기저기로 번져요.

```java
// 트랜잭션으로 묶으려고 커넥션을 손으로 넘기기 시작한다
Connection c = driver.connect();
try {
    inventory.reduce(c, itemId);     // 같은 c를 넘기고
    orders.create(c, order);         // 또 넘기고
    payments.record(c, payment);     // 또 넘긴다
    c.commit();
} catch (Exception e) {
    c.rollback();                    // 실패하면 되돌리고
    throw e;
} finally {
    c.close();
}
```

**여기까지 두 번째 커밋.** 고통이 드러난 상태예요.

## 혹시 문제를 느꼈나요?

이쯤이면 보일 거예요. 그래도 짚어요.

- 커넥션을 열고 닫는 보일러플레이트가 리포지토리마다 복붙돼요. 하나라도 `close`를 빼먹으면 조용히 새고요.
- 트랜잭션으로 묶으려니 커넥션을 인자로 줄줄이 넘기게 돼요. 리포지토리 시그니처가 커넥션에 오염되고, 트랜잭션 경계 코드가 서비스에 번져요.
- "진짜 일(쿼리)"이랑 "커넥션 생명주기 관리"가 한 메서드에 엉켜 있어요. 리포지토리가 자기 일도 하고 자원 관리도 하고요.

레벨 5하고 6, 그 쌍둥이 기억나요? 감싸개로 같은 인터페이스를 유지하면서, 한쪽은 기능을 덧입혔고(데코레이터) 한쪽은 접근을 통제했죠(프록시). 오늘은 그중 어느 쪽일까요? 우리가 씌우려는 건 새 기능이 아니에요. 쿼리 결과는 그대로예요. 우리가 하려는 건 "언제 커넥션을 열고 닫을지, 트랜잭션을 언제 시작하고 끝낼지"를 가로채 통제하는 거예요. 기능 추가가 아니라 접근과 생명주기 제어. 그 한 끗이 오늘을 가릅니다.

## 방향만 던질게요

길이 떠오를 거예요. 방향만요.

1. 리포지토리(또는 `Driver`) 앞에 문지기 하나를 세우세요. 리포지토리랑 똑같은 인터페이스를 쓰는, 그래서 호출자는 진짜 리포지토리를 부르는 줄 아는 그런 거요. 레벨 6에서 핸들러 앞에 세운 그 문지기처럼요.
2. 그 문지기가 호출을 가로채요. 진짜 메서드를 부르기 전에 커넥션을 열거나 풀에서 빌리고, 트랜잭션 범위에 묶고, 진짜 메서드를 부르고, 끝나면 커밋하고 커넥션을 반납해요. 도중에 터지면 롤백하고요. 호출자는 이게 일어나는 줄도 몰라요. 그냥 `findById`를 불렀을 뿐이고요.
3. 그러면 리포지토리에서 커넥션 코드가 통째로 사라져요. 리포지토리는 "이 커넥션으로 쿼리를 돌린다"만 알고, "그 커넥션을 언제 열고 닫나"는 문지기가 알고요. 트랜잭션으로 묶기? 문지기가 한 번 연 커넥션을 그 범위 안 호출들이 같이 쓰게 하면 돼요. 인자로 넘기던 거, 사라지죠.
4. 메서드에 표시만 하면 트랜잭션으로 묶이는 그것(`@Transactional`), 그게 바로 이 문지기예요. 레벨 6에서 권한 검사 문지기를 세울 때, 진짜 쓸모는 다른 데 있다고 흘렸던 거 기억나요? 그게 여기였어요. 트랜잭션 관리요.

도착점은 또렷해요. **리포지토리에 커넥션을 열고 닫는 코드가 없다. 호출자는 평범한 리포지토리 인터페이스만 부르고 커넥션을 모른다. 여러 호출을 한 트랜잭션으로 묶을 수 있고, 실패하면 통째로 롤백된다.** 거기까지 가면 after예요.

검증은 짧아요. 커밋 전에 자문해요. 리포지토리 메서드 안에 `connect`나 `close`가 남아 있나요? 한 군데도 없으면 됐어요. 그리고 서로 다른 리포지토리 호출 두 개를 한 트랜잭션으로 묶어서, 둘째가 실패할 때 첫째도 되돌아가나요? 되돌아가면 after예요.

## 회고: 두 시스템이 드디어 한 줄로 흐릅니다

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도요. 문지기를 한 겹 세웠으니 간접성이 늘어요. 호출이 진짜 리포지토리에 닿기 전에 문지기를 거치니, "실제 실행이 어디서 일어나지"를 한 번 더 따라가야 하고요. 그리고 트랜잭션 경계를 문지기가 쥔 만큼, 그 경계가 어디까지인지가 코드에 안 보이고 표시(애너테이션)에 숨어서, 잘못 그으면 찾기 어려워져요.

레벨 5(데코레이터), 6(프록시) 회고를 옆에 펴고 한 줄 적어요. 오늘 씌운 게 기능 추가였나요, 접근/생명주기 제어였나요? 구조는 셋 다 "같은 인터페이스로 감싸기"인데, 의도가 갈라요. 그 의도의 차이가 이름을 가르고요. 레벨 6에서 흘린 그 트랜잭션 떡밥, 오늘 거뒀고요.

그리고 이걸로 그림이 완성됐어요. 한번 따라가 봐요. 웹 요청이 mvc로 들어와서(레벨 1, 16, 20), 컨테이너가 주입한 리포지토리한테 가고(레벨 9, 10, 17, 18), 그 앞 문지기가 커넥션과 트랜잭션을 챙기고(오늘), 균일한 드라이버로 번역돼서(레벨 25), 미니 DB 엔진이 행을 한 줄씩 당겨 올려요(레벨 24). 두 시스템이 진짜로 한 줄로 흐르죠. 레벨 0에서 "이 둘은 한 군데서 만난다"고 약속한 그게, 지금 여기예요.
