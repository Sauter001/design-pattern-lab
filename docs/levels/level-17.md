# 레벨 17: "하나만"을 전역 변수로 풀지 않기

또 봐요. 열일곱 번째예요. 동네는 미니 스프링 컨테이너로 돌아와요. `mini-spring/container` 모듈(`lab.spring.container`)이에요. 레벨 2, 9, 10, 15에서 다지던 그곳요.

오늘은 좀 짓궂게 갈게요. 미리 말해 두는데, 이번 before는 십중팔구 여러분이 "어, 이거 그 유명한 패턴 아냐?" 할 바로 그 모양일 거예요. 맞아요. 근데 그게 함정이에요. 가장 유명한 게 가장 많이
욕먹기도 하거든요. 왜 그런지 손으로 겪어 봐요.

## 이번에 마주칠 상황

앱 전체에 딱 하나만 있어야 하는 것들이 있어요. 설정 레지스트리, 메트릭 수집기, 커넥션 풀 같은 거요. 둘이 생기면 곤란하죠. 설정이 두 벌이면 어느 게 진짜인지 모르고, 풀이 두 개면 자원이 두 배로 새요.

이번 목표는 이거예요. **앱 전체에 그 인스턴스가 하나만 존재하게 한다.** 단순해 보이죠. 그래서 다들 같은 길로 가요.

## 가장 정직하게 먼저

설정 레지스트리를 예로 들게요. "하나만"을 보장하는 제일 익숙한 방법, 다들 아는 그거요.

```java
class ConfigRegistry {
    private static final ConfigRegistry INSTANCE = new ConfigRegistry();  // 단 하나

    private ConfigRegistry() {
        load();
    }                                  // 생성자를 막고

    static ConfigRegistry getInstance() {
        return INSTANCE;
    }             // 전역 접근점

    String get(String key) { /* ... */ }
}

// 쓰는 쪽: 앱 어디서나
String dbUrl = ConfigRegistry.getInstance().get("db.url");
```

동작은 해요. 어디서 불러도 같은 하나가 나오죠. 둘이 생길 수가 없고요. **여기까지 첫 커밋.** before예요.

## 요구가 들어옵니다

서비스가 자라면서 요구가 와요.

- QA팀: "테스트마다 설정을 다르게 줘야 해요. 근데 지금은 한 테스트가 바꾼 값이 다음 테스트까지 새요. 격리가 안 돼요."
- 신사업팀: "테넌트마다 설정이 달라요. 하나가 아니라 테넌트별로 한 벌씩 필요해요."
- 플랫폼팀: "설정 만들 때 비밀 저장소에서 키를 받아와야 해요. 근데 언제 어떻게 로드되는지 통제가 안 돼요. 앱 뜨기도 전에 만들어지더라고요."

시키는 대로 풀어 보려고 하면, 하나씩 다 막혀요.

```java
// 테스트에서 설정을 갈아끼우고 싶은데
ConfigRegistry.getInstance().

set("db.url","test-db");  // 전역을 건드림 -> 다음 테스트로 샌다
// 가짜 레지스트리로 통째로 바꾸고 싶어도, getInstance()를 직접 부르는 코드가
// 사방에 박혀 있어서 손댈 데가 없다
```

**여기까지 두 번째 커밋.** 일단 박아 두세요.

## 혹시 문제를 느꼈나요?

이쯤 되면 보이죠. 그래도 짚어 둘게요.

- 전역 하나라 테스트 격리가 안 돼요. 한 테스트가 바꾼 상태가 다음으로 새고, 가짜로 갈아끼우려 해도 `getInstance()`가 코드 곳곳에 직접 박혀 손댈 수가 없어요.
- 누가 이걸 쓰는지가 안 보여요. 생성자로 받지 않고 아무 데서나 `getInstance()`로 꺼내니까, 이 클래스에 대한 의존이 숨어 버려요. 숨은 결합이죠.
- "하나만"이 "영영 하나만"이 돼 버렸어요. 정작 둘이 필요할 때(테넌트별로 한 벌씩) 길이 없어요.
- 초기화 시점이 클래스 로딩에 묶여서, 언제 무슨 설정으로 만들어지는지 통제가 안 돼요.

잠깐. before 저거, 사실 그 유명한 "딱 하나" 패턴 그 자체예요. 교과서에 그대로 나오는. 그런데 방금 그 고통들이 전부 어디서 나왔는지 보세요. "하나만"에서 나온 게 아니에요. "전역에서 꺼내 쓴다"에서
나왔어요. 이 둘은 같은 게 아닌데, 저 흔한 방법은 둘을 한 덩어리로 묶어 버렸거든요. 그래서 오늘 할 일은 그 패턴을 쓰는 법이 아니라, 좋은 점(하나만 보장)은 살리고 나쁜 점(전역)은 떼어 내는 법이에요.

## 방향만 던질게요

길이 떠오를 거예요. 방향만 대 줄게요.

1. "하나만 있어야 한다"는 요구와 "전역에서 꺼내 쓴다"는 방식을 갈라놓아 보세요. 하나만 만드는 책임을 이미 우리가 만든 그 컨테이너에 맡기고, 쓰는 쪽은 그걸 생성자로 받기만 하면요? 인스턴스는 여전히
   하나인데, `getInstance()`로 전역에서 꺼낼 필요가 없어지죠. 하나를 보장하는 게 static 필드가 아니라 컨테이너가 되는 거예요.
2. 그렇게 하면 테스트에선 가짜를 만들어 넣어 주면 돼요. 하나를 컨테이너가 보장하니까, 테스트에선 다른 걸 끼울 수 있고요. 테넌트별로 한 벌씩 필요해지면 그 보장의 범위(스코프)만 바꾸면 되고요.
3. 누가 이걸 쓰는지도 생성자에 드러나요. 숨어 있던 결합이 보이는 결합이 되는 거죠. 의존이 코드에 적혀 있으면 따라가기도 쉽고요.

도착점은 또렷해요. **인스턴스는 여전히 앱에 하나다. 그런데 그 "하나"는 전역 static이 아니라 컨테이너가 보장한다. 그래서 테스트에선 갈아끼울 수 있고, 의존은 생성자에 드러나며, 필요하면 둘로도 늘릴 수
있다.** 거기까지 가면 after예요.

검증은 짧아요. 커밋 전에 자문해요. 테스트에서 이 설정을 가짜로 갈아끼울 수 있나요? 그리고 `getInstance()`를 직접 부르는 코드가 한 군데라도 남아 있나요? 전부 주입받아 쓰고 한 군데도 안 남았으면,
그리고 인스턴스는 여전히 하나면 after예요.

## 회고: 진짜로 떼어 낸 게 뭔지

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 이건 농담 안 하고 말할게요. 오늘 걷어 낸 건 "하나만"이 아니에요. "전역"이에요. 그
둘을 떼어 낸 게 이 단계의 전부고요. 회고에 "before도 하나, after도 하나인데, 무엇이 달라졌나"를 한 줄로 적어 보세요. 답이 "하나를 누가 보장하느냐"로 나오면 제대로 잡은 거예요.

비용이랑 균형추도요. 컨테이너가 관리하는 하나도 결국 공유 상태라, 여러 곳이 동시에 건드리면 동시성 문제는 그대로 있어요. 그리고 이건 솔직히 말할게요. 정말로 상태가 없고(예: 순수 계산만 하는), 테스트에서
갈아끼울 일도 없는 거라면 그냥 처음의 그 전역 static이 더 단순하고 정직해요. 함정은 "하나"가 아니라 "전역 가변 상태"예요. 본인 경우가 어느 쪽이었는지 적어 두세요.
