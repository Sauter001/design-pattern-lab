# 레벨 11: 인자 지옥 말고, 필요한 것만 골라 짓기

또 봐요. 열한 번째예요. 동네 옮겨요. 레벨 9, 10에서 미니 스프링 컨테이너에 있었는데, 이제 다시 미니 DB로 건너갑니다. 레벨 7, 8에서 갔던 그 동네예요. 작업 공간은 `mini-db/engine`
모듈(`lab.db.engine`)이고요.

지난 두 단계, "만드는 일"을 다뤘죠. 빈 하나를, 그리고 빈 한 벌을. 오늘도 만드는 얘긴데 결이 또 달라요. 이번엔 "넣을 거리가 너무 많은 객체 하나"를 어떻게 짓느냐예요. 쌍은 아니에요. 오늘은 혼자 가는
단계예요.

## 지난 단계 돌아보기

레벨 9, 10에서 컨테이너가 빈을 직접 `new` 하지 않고 만드는 일을 위임했어요. 하나를, 그리고 어긋나면 안 되는 한 벌을요. 거기서 "무엇을, 어떻게 만드느냐"를 다뤘죠.

오늘 마주칠 건 좀 달라요. 만드는 법이 어려운 게 아니라, **만들 객체에 들어갈 선택 조각이 너무 많아서** 짓는 입구가 지저분해지는 경우예요. 한번 보죠.

## 이번에 마주칠 상황

미니 DB가 쿼리를 실행하려면, 먼저 그 쿼리를 객체로 들고 있어야 해요. `SELECT id, name FROM users WHERE age > 18 ORDER BY name LIMIT 10` 같은 걸
`Query` 객체 하나로요.

문제는 이 쿼리라는 게 들어갈 수 있는 조각이 너무 많다는 거예요. 고를 컬럼, 테이블, WHERE 조건, ORDER BY, LIMIT, OFFSET, GROUP BY, HAVING. 그런데 대부분은 안 쓸 때가 더
많아요. SELECT랑 FROM만 있고 나머진 다 비는 쿼리가 수두룩하죠.

이번 목표는 이거예요. **선택 조각이 많은 쿼리 객체를, 필요한 것만 골라서 깔끔하게 짓는다.**

## 가장 정직하게 먼저

들어갈 수 있는 건 다 받는 생성자 하나. 안 쓰는 건 `null`로요. 제일 정직하게요.

```java
class Query {
    Query(String table, List<String> columns, String where,
          String orderBy, Integer limit, Integer offset) {
        // ... 다 받아서 필드에 넣는다
    }
}

// 호출부: WHERE랑 LIMIT만 쓰고 싶은데
Query q = new Query("users", List.of("id", "name"), "age > 18", null, 10, null);
```

생성자를 여러 개 두는 길도 있죠. `Query(table)`, `Query(table, columns)`, `Query(table, columns, where)`... 조각 수만큼 생성자가 늘어요. 둘 중 뭘
골라도 곧 똑같이 막혀요.

동작은 해요. 인자 넣으면 쿼리 하나 나오죠. **여기까지 첫 커밋.** before예요.

## 절이 자꾸 늘어납니다

쿼리가 받아야 할 절이 점점 늘어요.

- 분석팀: "GROUP BY랑 HAVING도 받아야 해요. 집계 쿼리 쓸 거라서요."
- 리포트팀: "DISTINCT 옵션도요. 중복 제거요."
- 그리고 곧: "조인도 들어와요. 두 테이블이요."

시키는 대로 넣어 보세요. 저 생성자 인자가 여섯에서 아홉, 열로 불어요. 호출부는 이렇게 되고요.

```java
Query q = new Query("users", List.of("id"), "age > 18",
        null, 10, null, null, null, true);
//                   WHERE만 쓰고 ORDER/OFFSET/GROUP/HAVING은 null, 맨 끝 true는 DISTINCT... 한눈에 보여요?
```

**여기까지 두 번째 커밋.** 일단 박아 두세요.

## 혹시 문제를 느꼈나요?

이쯤 되면 보이죠. 그래도 짚어 둘게요.

- 호출부가 `null, 10, null, null` 위치 인자 지옥이에요. 다섯 번째 인자가 LIMIT인지 OFFSET인지 세어 봐야 알고, 한 칸 밀리면 조용히 틀려요. 그리고 꼭 한 번은 밀려요.
- 선택 절이 하나 늘 때마다 생성자 시그니처가 부풀고, 그 생성자를 부르던 **모든 호출부**가 `null` 하나씩 더 붙여야 해요. 산탄총 수술이죠. 레벨 2에서 본 그거요.
- 잘못된 조합(LIMIT 없이 OFFSET만, GROUP BY 없이 HAVING)을 막을 자리가 없어요. 다 만들어진 다음에야 터지고요.

## 방향만 던질게요

길이 떠오를 거예요. 답은 본인 몫이고, 방향만 대 줄게요.

1. 쿼리를 한 방에 다 받지 말고, 필요한 절만 하나씩 얹어 가며 짓게 하면 어떨까요. "테이블은 users, 컬럼은 id랑 name, 조건은 age > 18, 리밋은 10" 이렇게 이름을 붙여 가면서요. 안 쓰는
   절은 아예 안 부르면 그만이고, 호출부에서 `null` 행렬이 사라지죠.
2. 그렇게 단계별로 얹는 방식이면, 새 절(DISTINCT, GROUP BY)이 와도 얹는 단계 하나만 새로 만들면 돼요. 기존 호출부는 한 줄도 안 바뀌고요. 그 절을 안 쓰던 사람은 생긴 줄도 몰라요.
3. 다 얹고 마지막에 "이제 지어 줘" 하는 순간이 하나 있으면, 거기서 잘못된 조합을 한 번에 검사할 수 있어요. OFFSET만 있고 LIMIT이 없으면 거기서 막는 거죠. 다 만들어진 다음이 아니라, 짓는 그
   순간에요.

도착점은 또렷해요. **선택 절을 추가해도 기존 호출부는 한 줄도 안 바뀐다. 호출부엔 쓰는 절만 이름 붙어 보이고, `null` 행렬이 없다. 잘못된 조합은 짓는 순간 걸러진다.** 거기까지 가면 after예요.

검증은 짧아요. 커밋 전에 자문해요. 새 선택 절(예: WINDOW 함수)을 추가하는데 기존 호출부를 여나요? 그리고 WHERE랑 LIMIT만 쓰는 쿼리를 지을 때, 안 쓰는 절이 호출부에 `null`로라도
보이나요? 호출부를 안 열고 안 쓰는 절이 아예 안 보이면 after예요.

## 회고: 이게 과한 건 아닌지도 적어요

세 번째 커밋. 회고는 [docs/README-template.md](../README-template.md) 틀로 적고요. 비용도 꼭이요. 비용은 이거예요. 쿼리 하나 지으려고 중간 조립 단계를 거쳐요.
`new Query(...)` 한 줄이면 되던 게 여러 줄이 됐고, 조립을 맡는 물건이 하나 더 생겼죠.

이건 농담 안 하고 말할게요. 조각이 두세 개뿐인 객체라면 이 단계는 과해요. 그냥 생성자가 나아요. 조각이 많고 대부분 선택일 때라야 본전을 뽑거든요. 그러니 "내 쿼리는 절이 몇 개였고, 그래서 이게 값을 했나"
를 꼭 적으세요. 안 쓰는 게 나았을 상황을 분간하는 눈이, 쓰는 법만큼 값나가요.
