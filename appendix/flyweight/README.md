# 부록: 플라이웨이트 (Flyweight)

여긴 부록이에요. 레벨하고는 규칙이 달라요. 레벨에선 패턴 이름을 끝까지 숨기고 본인이 다시 발견하게 했죠. 여긴 안 그래요. 이름 먼저 까고 시작해요. **플라이웨이트.** 왜냐면 이 패턴은 본인이 손으로 발견할 일이 거의 없거든요. 현대 환경에선 플랫폼이 이미 해 놨어요. 그래서 여기 목적은 "짜 보기"가 아니라 "이미 깔려 있는 걸 알아보기"예요.

본문 22개를 다 거쳐 왔으면, GoF에서 딱 하나 안 만난 게 이거예요. 도메인이 자연스럽게 안 불러서 억지로 안 끼웠어요. "도메인이 요구 안 하면 안 넣는다", 그 규칙대로요. 대신 부록으로 빼서, 코드까지 붙여 제대로 보여줄게요.

## 한 줄 요약

**같은 걸 여러 번 쓸 때, 그 "같은 부분"을 하나만 만들어 공유한다.** 끝이에요. 나머지는 디테일이고요.

핵심은 상태를 둘로 가르는 거예요.

- **intrinsic state (내재 상태):** 어디서 쓰이든 안 변하는, 공유 가능한 부분. 글자 'A'의 자형 비트맵 같은 거요. 'A'는 문서에 천 번 나와도 모양은 하나예요.
- **extrinsic state (외재 상태):** 쓰이는 자리마다 달라지는 부분. 그 'A'가 화면 어디에 어떤 색으로 찍히는지요.

intrinsic은 객체 안에 가두고 공유하고, extrinsic은 밖으로 빼서 쓸 때마다 인자로 건넨다. 이 분리가 플라이웨이트의 전부예요. 안 가르고 다 끌어안으면, 'A' 천 개가 자형 비트맵 천 벌을 들고 메모리를 잡아먹어요.

## 손으로 짜 보면 (사례)

고전 예제 그대로예요. 글자를 화면에 그리는데, 자형 비트맵은 무겁다고 칩시다(폰트 힌팅, 안티앨리어싱 다 거치면 진짜 무거워요). "MISSISSIPPI" 열한 자를 그리는데, 자형을 열한 번 만들 거예요? 서로 다른 글자는 M, I, S, P 네 개뿐인데요.

코드는 세 조각이에요. (아래 스니펫은 읽기용이고, 색칠된 실제 소스는 링크의 `.java` 파일에 있어요.)

- [`Glyph.java`](src/main/java/lab/appendix/flyweight/Glyph.java): 플라이웨이트 본체. intrinsic(글자, 비트맵)만 들고 있어요. `drawAt(x, y, color)`로 그리는데, x/y/color는 안 가지고 인자로 받아요. 그게 extrinsic이라서요.
- [`GlyphFactory.java`](src/main/java/lab/appendix/flyweight/GlyphFactory.java): 공유가 일어나는 길목. `get('S')`를 두 번 불러도 같은 인스턴스를 돌려줘요. 풀에 없을 때만 새로 만들고요. 생성을 여기 가두는 게 약속이에요. 밖에서 `new Glyph(...)` 하면 공유가 깨지니까요.
- [`FlyweightDemo.java`](src/main/java/lab/appendix/flyweight/FlyweightDemo.java): 돌려 보는 곳.

요지는 이 한 토막이에요.

    GlyphFactory factory = new GlyphFactory();
    int x = 0;
    for (char c : "MISSISSIPPI".toCharArray()) {
        Glyph glyph = factory.get(c);   // 같은 글자면 같은 인스턴스 (공유)
        glyph.drawAt(x, 0, 0x000000);   // 위치(extrinsic)만 바꿔 다시 그린다
        x += 10;
    }
    // 글자 11개를 그렸지만 자형은 4개만 만들어진다.

자형은 글자 종류 수(4)만큼만 만들어지고, 위치만 바꿔 열한 번 그려져요. 천 자, 백만 자로 가도 자형은 글자 종류 수에서 안 늘고요. 그게 이 패턴이 아끼는 거예요.

## 그런데, JDK가 이미 해 놨어요 (관찰이 진짜 핵심)

여기가 부록의 진짜 알맹이예요. 위에 걸 손으로 짤 일이 왜 거의 없냐면, 자주 쓰이는 값들은 플랫폼이 알아서 공유하거든요. 두 개만 봐요.

### 1) `Integer.valueOf`의 캐시

    Integer.valueOf(127) == Integer.valueOf(127)   // true  (캐시 범위 안, 같은 객체)
    Integer.valueOf(128) == Integer.valueOf(128)   // false (범위 밖, 매번 새 객체)

자바는 `-128 ~ 127` 범위의 `Integer`를 미리 만들어 캐시해 둬요. `valueOf`가 그 범위면 캐시본을 돌려주고요. 이게 바로 플라이웨이트예요. 값(intrinsic)이 같으면 같은 인스턴스를 공유하는 거죠. 오토박싱(`Integer i = 100;`)도 속으로 `valueOf`를 불러서 같은 혜택을 받아요. 그래서 `==`로 `Integer`를 비교하면 작은 수에선 우연히 맞고 큰 수에선 틀리는, 그 악명 높은 함정이 생기는 거고요. (그래서 박싱 타입은 `equals`로 비교해요. 이건 덤으로 알아 두면 좋아요.)

### 2) `String` 인터닝

    "kim" == "kim"                      // true  (컴파일 타임 상수 풀에서 공유)
    "kim" == new String("kim")          // false (new는 새 객체를 강제로 만든다)
    "kim" == new String("kim").intern() // true  (intern이 풀로 보내 공유본을 준다)

소스에 박은 문자열 리터럴은 상수 풀에 한 번만 올라가고 다 공유돼요. 이것도 플라이웨이트고요. `new String(...)`은 일부러 새 객체를 만드니까 공유가 깨지고, `intern()`을 부르면 풀의 공유본으로 합류해요.

직접 돌려서 눈으로 봐요.

```bash
./gradlew :appendix:flyweight:run     # FlyweightDemo 실행
./gradlew :appendix:flyweight:test    # 위 동일성들을 단언으로 확인
```

[`FlyweightTest.java`](src/test/java/lab/appendix/flyweight/FlyweightTest.java)가 이 동일성들(`assertSame` / `assertNotSame`)을 그대로 박아 뒀어요. `equals`가 아니라 `==`로요. "같은 인스턴스냐"가 이 패턴의 유일한 증거라서 그래요.

## 그럼 언제 손으로 짜요?

플랫폼 공유가 안 닿는, 메모리가 진짜로 빠듯한 자리예요.

- 임베디드처럼 메모리가 손바닥만 한 환경.
- 게임에서 파티클 수백만 개. 모양/텍스처(intrinsic)는 공유하고, 위치/속도(extrinsic)만 따로 들어요.
- 모바일에서 글리프 비트맵 수천 개. 같은 글자를 자형 한 벌로 공유해 메모리를 아껴요.

공통점이 보이죠. **같은 무거운 것이 엄청 많이 중복되고, 그게 메모리를 진짜로 누를 때.** 이 조건이 아니면 플라이웨이트는 과해요. 객체 몇 개 아끼자고 팩토리 끼우고 상태를 둘로 가르는 건, 복잡도만 늘리는 손해 보는 장사예요.

## 헷갈리기 쉬운 구분 하나

이 lab 바깥 얘기지만 짚어 둘게요. 예를 들어 한자 인식 앱을 만든다고 쳐요. 글자를 **알아보는 로직**(CNN 분류, 자형 정규화)은 플라이웨이트랑 아무 상관 없어요. 그건 인식이지 공유가 아니거든요. 그런데 같은 앱이라도 **그리기/메모리 관점**(모바일에서 글리프 비트맵을 자형 한 벌로 공유)에선 플라이웨이트가 정당해져요. 같은 도메인이라도 "무겁고 중복되는 걸 공유하느냐"라는 질문이 서는 자리에서만 이 패턴이에요. 그 질문이 없으면 아니고요.

## 비용

- 상태를 intrinsic / extrinsic으로 갈라야 해요. 이 가르기가 늘 깔끔하진 않아요. 어정쩡한 상태가 끼면 골치 아파요.
- 호출자가 extrinsic을 매번 들고 다녀야 해요. `drawAt(x, y, color)`처럼 인자가 늘죠.
- 공유본은 불변이어야 안전해요. 공유하는 객체를 누가 바꾸면 다 같이 바뀌니까요.
- 공유가 팩토리에 갇혀 있어서, 디버깅할 때 "이 인스턴스 어디서 왔지"가 한 겹 가려져요.

## 마무리

이걸로 GoF 23개를 다 봤어요. 22개는 본문에서 손으로 다시 발견했고, 마지막 하나는 여기서 "이미 깔려 있더라"로 확인했고요. 패턴을 외우는 게 목적이 아니었어요. 언제 필요하고 언제 과한지를 손으로 재 보는 게 목적이었죠. 플라이웨이트는 그중에서도 "대부분의 경우 안 짜는 게 정답"이라는, 좀 김빠지지만 제일 정직한 결론을 가진 패턴이에요. 그걸로 끝내는 게 어울려요.
